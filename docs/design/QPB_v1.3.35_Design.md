# Quality Playbook v1.3.35 — Design Document

**Version:** 1.3.35
**Status:** Shipped
**Date:** 2026-04-12
**Author:** Andrew Stellman
**Primary commit:** `ebfc9ea` — "v1.3.35: mandatory exploration, bug-finding patterns, phase gates"

---

## What This Version Introduced

Version 1.3.35 is the release where exploration stopped being an optional handoff convenience and became the foundation on which every downstream artifact is built. Three closely related changes landed together in the same commit, and each one reinforces the others. Read in isolation, each could look like a local improvement. Read together, they describe a different skill with a different center of gravity than the one that existed in v1.3.33.

The first change made `quality/EXPLORATION.md` mandatory in every mode — single-pass, multi-pass, and interactive. Prior to this release, single-pass runs were allowed to keep exploration findings in working memory and proceed directly to artifact generation, writing EXPLORATION.md only as an optional backup. The revised SKILL.md deletes that permission. The new instruction reads: "Write your Phase 1 exploration findings to `quality/EXPLORATION.md` before proceeding to Phase 2. This file is mandatory in all modes — single-pass, multi-pass, and interactive." What replaced the old permissive clause is a rationale paragraph explaining why. The discipline of writing exploration findings to disk is what forces thorough analysis; without it, the model keeps vague impressions in working memory and produces broad, abstract requirements that miss function-level defects. The same paragraph spells out what writing forces — file paths, line numbers, exact function names, concrete behavioral rules — and names that specificity as the quality that makes requirements precise enough to catch bugs during code review.

The second change introduced a new reference document, `references/exploration_patterns.md`, which formalizes three systematic bug-finding patterns: fallback path parity, dispatcher return-value correctness, and cross-implementation contract consistency. The file is 150 lines. It is not a collection of heuristics. Each pattern has a precise definition, a stated bug class it targets, five diverse examples drawn from different domains (authentication, database drivers, memory allocators, event loops, cache invalidation, file locking, and more), application instructions, and an explicit output format that slots into the EXPLORATION.md template. The EXPLORATION.md template in SKILL.md was updated with three new sections — one per pattern — so that exploration output is structurally aligned with the patterns that produced it. A reader of an EXPLORATION.md file after v1.3.35 can tell by section headings whether each pattern was applied.

The third change added phase completion gates at two critical boundaries: Phase 1 to Phase 2, and Phase 2 to Phase 2b. A phase gate is a short, mandatory verification checklist that must be satisfied before the next phase is allowed to begin. The gates turn informal transitions into explicit contracts. The model cannot simply decide it has "finished exploring" and move on. It must confirm, against objective criteria — file existence, minimum content length, presence of specific sections, presence of specific kinds of findings — that the prior phase produced the artifacts the next phase needs.

Taken together, these three changes shift the skill's center of gravity from artifact generation to exploration. In earlier versions the nine-file output was the product and exploration was the setup. In v1.3.35 exploration becomes the product — the artifacts are downstream transformations of what Phase 1 found and wrote down. EXPLORATION.md is no longer a diagnostic backup; it is the canonical substrate from which REQUIREMENTS.md, CONTRACTS.md, the functional tests, the code review protocol, and the spec audit are all derived.

The same commit also removed virtio-specific examples from SKILL.md and the requirements pipeline reference. Those examples had served as concrete anchors during development but were leaking domain specificity into what is meant to be a general-purpose skill. Specific edits in the diff show this cleanup at work. The original text that said "MSI-X vs INTx fallback" and "admin queue present vs absent" was replaced with "primary vs. degraded mode" and "negotiated vs. default configuration." The original text that mentioned `vring_transport_features()` whitelisting feature bits was replaced with "feature flag gates" and "capability whitelist helpers." The carry-forward rule that previously blamed "the v1.3.13 virtio regression where a previously learned requirement was lost" was rewritten to reference regressions generically, as "a direct cause of regressions where previously learned requirements are lost between runs." The virtio driver remains the benchmark repository the skill is tested against, but the skill no longer reads as if it were written for it.

The interaction among the three main changes is what makes v1.3.35 more than an incremental release. Mandatory exploration on its own would raise the floor of output quality but would not tell the model what to look for. Patterns on their own would tell the model what to look for but could be ignored if exploration were still optional. Phase gates on their own would enforce transitions but would not dictate what the phases must produce. Put together, the three changes form a reinforcing triad. Mandatory exploration establishes that exploration happens and produces an artifact. Patterns establish what exploration must look for and how its findings must be structured. Phase gates establish that the artifact must exist and contain pattern-specific findings before the pipeline moves on. Each change depends on the others to be effective; none of them is sufficient alone.

## Why It Was Needed

Empirical testing on the virtio driver gave the redesign its mandate. The commit message records the numbers. Version 1.3.33 found one bug on a clean run. Version 1.3.34 — which is the unreleased intermediate that made EXPLORATION.md mandatory and added phase gates but did not yet add the explicit patterns — found two bugs. That confirmed a hypothesis that had been accumulating across prior releases: the act of writing findings to disk changes the quality of the findings. But two known bugs remained undetected even with mandatory exploration, and they had a common shape. Both lived in code paths that a broad architectural summary would skim past. That is what motivated the third change: the exploration patterns themselves, designed specifically to force the explorer's attention onto the bug classes that subsystem-level descriptions miss. The commit message states this explicitly: "v1.3.35 adds the exploration patterns targeting the remaining 2 known bugs."

Three failure modes in prior versions made the redesign necessary. Each one had been observed across multiple runs, and each one had a common cause: the skill was too permissive about exploration depth.

The first failure was derivation without exploration depth. Requirements derivation works by turning specifications and code into REQ-NNN conditions of satisfaction. When exploration stayed at the subsystem level — "the reset subsystem handles errors" — the derived requirements inherited that vagueness. Abstract requirements do not catch bugs during code review because abstract requirements have no specific condition to check. A requirement that says "the driver must handle reset correctly" is satisfied by any reset code that runs without crashing. A requirement that says "`vm_reset()` at `virtio_mmio.c:256` must poll the status register until it reads zero after writing zero" either matches the code or does not, and that match-or-miss is what produces the bug finding. The revised SKILL.md uses exactly this example in its new "depth over breadth" paragraph, which is the most important piece of new exposition in the file. The paragraph frames the choice starkly: a narrow scope with function-level detail finds more bugs than a broad scope with subsystem-level summaries. The difference between a useful exploration and a useless one is specificity — file paths, function names, line numbers, exact behavioral rules.

The second failure was ad-hoc pattern application. When a skilled human reviews code for bugs, they apply mental patterns — checking fallback paths for parity, checking dispatchers for return-value correctness, checking cross-implementation consistency. These patterns are the difference between a shallow and a deep review. The prior versions of the skill assumed the model would apply these patterns on its own if the exploration instructions were phrased broadly enough. That assumption did not hold. The model was capable of applying the patterns when explicitly directed but did not consistently apply them unprompted. The formalization in exploration_patterns.md makes pattern application an explicit step rather than an emergent behavior. The revised SKILL.md adds a short but load-bearing paragraph titled "Bug-finding exploration patterns (mandatory)" that instructs the model to read the patterns reference and apply every pattern in it to each core module. The paragraph's final sentence is revealing: the patterns are "the most important part of exploration — they are what turns a subsystem summary into a list of requirements specific enough to catch bugs."

The third failure was phase transitions without entry conditions. In prior versions the pipeline described phases as sequential steps, but nothing enforced the sequence. The model could theoretically proceed to Phase 2 without having written a complete EXPLORATION.md, or proceed to Phase 2b without having generated all nine core artifacts. In practice this happened often enough in testing that the gates were needed. A phase gate is, in effect, a contract precondition: the next phase's preconditions are documented as its entry conditions, and the prior phase cannot declare itself complete until those conditions are satisfied. The language the revised SKILL.md uses for the gates is imperative. "Do not begin Phase 2 until all five checks pass." Nothing about this is phrased as a suggestion.

A fourth underlying motivation runs through all three changes: the skill's reliability was becoming uneven. The same skill applied to the same codebase by the same model could produce a thorough analysis on one run and a superficial one on the next. The thorough runs tended to have long EXPLORATION.md files with specific function-level detail; the superficial runs had short summaries or skipped exploration entirely. Making exploration mandatory, formalizing the patterns, and adding phase gates is an attempt to squeeze that variance out — to make the floor of the skill's behavior close to its ceiling.

There is also a more general lesson the release encodes, one that applies beyond this skill. When an AI agent is given discretion about how much context to gather, it will sometimes gather enough and sometimes gather too little, and the difference will correlate with the complexity of the task and the agent's implicit confidence. Shallow context feels sufficient until a bug is missed. The corrective is not to exhort the agent to "explore more" but to change the structural contract: require the context-gathering output as an explicit artifact, define what must be in the artifact, and check that the artifact was produced before the next step is allowed. v1.3.35 is an application of this pattern to requirements elicitation in particular, but the pattern generalizes. Any phase of an agentic pipeline whose quality depends on depth benefits from the same treatment.

## Mandatory Exploration

The pivotal sentence in the revised SKILL.md is this: "Write your Phase 1 exploration findings to `quality/EXPLORATION.md` before proceeding to Phase 2. This file is mandatory in all modes — single-pass, multi-pass, and interactive." The old sentence that followed the "mode" description — "In single-pass mode (the default when a human runs the skill interactively), EXPLORATION.md is optional — you can keep exploration context in working memory and proceed directly to Phase 2" — has been deleted. What replaces it is a rationale paragraph that names the behavioral mechanism. The discipline of writing exploration findings to disk is what forces thorough analysis. Without it, the model keeps vague impressions in working memory and produces broad, abstract requirements that miss function-level defects. Writing forces specificity. Specificity makes requirements precise enough to catch bugs.

The file itself is a structured Markdown document organized into sections: domain identification, architecture map, existing tests, specification summary, quality risks, skeleton and dispatch analysis, derived requirements (REQ-NNN), and derived use cases (UC-NN). The v1.3.35 template adds three new sections, one for each of the new patterns.

The fallback path analysis section template reads: "For each cascade/degradation strategy found: list the primary path and each fallback, note which operations each path performs, flag any behavioral differences between them. Each difference is a candidate requirement."

The dispatcher return-value analysis section template reads: "For each dispatcher function: enumerate input combinations, document the return value for each combination, flag any combination where the return value may be incorrect. Each flag is a candidate requirement."

The cross-implementation consistency section template reads: "For each operation implemented by multiple functions (e.g., reset in MMIO vs. PCI): list the implementations side by side, note which mandatory steps each performs, flag any step present in one but missing in another. Each gap is a candidate requirement."

The parenthetical "reset in MMIO vs. PCI" is the one place in the revised SKILL.md where a virtio-specific detail survives, because it provides a concrete enough example to make the instruction unambiguous without naming the driver directly.

The write-then-read cycle is the disciplinary core of mandatory exploration. After writing EXPLORATION.md the model must explicitly re-read the file from disk before proceeding. The re-read serves two purposes. It verifies that the file was written correctly — that what the model believes it wrote is actually on disk. More importantly, it forces the structured findings back into working memory as a single coherent document rather than as scattered impressions accumulated over the course of exploration. This "drop everything, re-read the summary" pattern is the same context-management discipline used at multi-pass boundaries, now applied even in single-pass mode where no boundary technically exists. The revised SKILL.md explicitly rewrites the surrounding paragraph to reflect this. The old text said that EXPLORATION.md was "a useful backup" in single-pass mode and "essential" in multi-pass mode. The new text says it is "essential in all modes" and explains why in a single sentence: "In single-pass mode it forces the model to articulate specific findings (file paths, function names, line numbers) before generating artifacts. In multi-pass mode it is also the handoff artifact between passes. Either way, the write-then-read cycle is the quality gate for exploration depth."

The rationale is blunt. Without the write-then-read cycle, the model keeps vague impressions in working memory and produces broad, abstract requirements that miss function-level defects. The cost is not zero. Writing a thorough EXPLORATION.md takes time and tokens. The cost pays for itself because every requirement that is insufficiently specific becomes a bug that is not found.

SKILL.md also reinforces that exploration is where requirements elicitation happens. Everything downstream is transformation. The functional tests transform requirements into executable checks, the code review transforms requirements into bug findings, the spec audit transforms requirements into contradictions. If a requirement is never derived during Phase 1, no downstream transformation will recover it. Phase 1 is the only place where new requirements can enter the system, and the only place where the depth of future bug-finding is set. The gate language reinforces this: "Phase 1 is your only chance to understand the codebase deeply. Every requirement you miss here is a bug you will not find in Phase 2b. Invest the time."

The mechanism by which writing improves depth deserves its own consideration. Working memory in a language model is not a neutral store; it is a summarization medium. As the model accumulates exploration findings across many source files, the internal representation necessarily compresses. What started as "this file has a dispatcher at line 256 that returns `IRQ_NONE` when only the config-change event fires" becomes, after enough further exploration, simply "the interrupt subsystem handles events." The compression is inevitable and usually undesirable. Writing to disk is an escape valve — it takes the detailed finding out of working memory and commits it to an external record before compression erodes it. The re-read at phase boundary is the inverse: it pulls the finding back into working memory in its uncompressed form. The net effect is that specific findings survive across the context window in a way they would not if they stayed in working memory only.

This mechanism is why the exploration discipline specifically requires writing, not merely thinking in more detail. A model instructed to "think carefully about the dispatcher" will do so, but the careful thought will be subject to the same compression pressure as everything else. A model instructed to "write the dispatcher's return-value analysis to disk in a specific format" produces a durable artifact that resists compression. The skill's design treats the file system as working memory's backing store.

## Bug-Finding Patterns

The three patterns in references/exploration_patterns.md were not chosen arbitrarily. The reference file is explicit about the derivation. Each pattern was derived from analyzing cases where an AI code review found bugs when seeded with specific requirements but failed to find the same bugs through independent exploration. In other words, each pattern represents a class of requirements that the model can recognize and use if given, but does not reliably derive on its own unless directly prompted. The patterns exist to close that gap.

The reference file opens with a framing sentence worth quoting in full: "Requirements problems are the most expensive to fix because they are not caught until after implementation. The exploration phase is requirements elicitation — it determines what the code review and spec audit will look for. A requirement that is never derived is a bug that is never found. These patterns exist to systematically surface requirements that broad exploration misses."

### Pattern 1: Fallback and Degradation Path Parity

When code provides multiple strategies for the same goal — a primary path and one or more fallback paths — each fallback must preserve the same behavioral invariants as the primary. The fallback may use a different mechanism, but the observable contract must be equivalent.

The bug class this targets is ubiquitous across domains. Fallback paths are written later, tested less, and reviewed with less scrutiny than primary paths. They often omit steps the primary path performs (validation, cleanup, index assignment, resource release) because the developer copied the primary path and simplified it for the "degraded" case. The result is a function that works correctly in the common case but violates its contract when the fallback activates.

The reference provides five diverse examples. An authentication service tries OAuth token validation, falls back to API key lookup, falls back to session cookie — and the API key fallback skips scope validation and grants full access. A database client tries the primary connection pool, falls back to a secondary pool, falls back to creating a one-off connection — and the one-off connection uses the driver default isolation level instead of the configured one. A memory allocator tries a fast slab path, falls back to a slow page-level path — and the slow path returns uninitialized memory because zero-fill was only in the slab fast path. A driver tries per-resource interrupt vectors, falls back to shared vectors, falls back to a single legacy interrupt line — and the legacy fallback uses a loop counter for resource indexing instead of querying the device for the correct index. A message broker tries binary serialization, falls back to JSON, falls back to string encoding — and the JSON fallback silently drops null fields that binary serialization preserves. The examples span different domains precisely to make the pattern recognizable across stacks the model has never seen.

The application instructions are specific. For each core module, look for conditional chains that try one approach then fall through to another, strategy or adapter patterns where multiple implementations are selected at runtime, retry logic with different strategies per attempt, and feature-negotiation cascades where capabilities determine which code path runs. For each cascade found, list the primary path and every fallback, check whether each fallback performs the same critical operations as the primary (validation, resource setup, index assignment, cleanup, error reporting), and record any operation present in the primary but missing in a fallback as a candidate requirement.

The output format requires the explorer to name the cascade, identify the primary and each fallback by function name and file:line, enumerate parity gaps, and list candidate REQ-NNN entries.

### Pattern 2: Dispatcher Return-Value Correctness

When a function dispatches on input type or condition and returns a status value, the return value must be correct for every combination of inputs — not just the primary case. Dispatchers that handle multiple event types, request types, or state transitions are particularly prone to return-value bugs in edge combinations.

The bug class this targets is subtler than the first. Dispatchers are typically written and tested for the common case. The return value is correct when the primary event fires. But when an unusual combination occurs — only a secondary event, no events at all, multiple concurrent events — the return-value logic may be wrong. It may return "not handled" for a handled event, return success for a partial failure, or return a stale value from a previous iteration.

The reference provides five diverse examples here too. HTTP middleware returns the wrong status code when rate-limiting triggers after authentication: rate-limited requests get 401 instead of 429. A poll/select loop handles read-ready, write-ready, and error conditions, but when only an error condition fires on a socket with no pending reads, the loop returns "no events" because the read-ready check was false and connection errors are silently ignored. A state machine dispatch function handles valid transitions, invalid transitions, and no-op transitions, but when a no-op transition occurs (current state equals target state), the function returns an error code intended for invalid transitions and idempotent operations fail when they should succeed. An interrupt handler checks for multiple event types (data-ready, configuration-change, error), but when only a secondary event fires, the handler returns "not mine" because the primary event check failed and the secondary path does not set the handled flag — legitimate secondary events are reported as spurious. A CLI dispatcher matches subcommands and flags, but when an unknown flag is combined with a valid subcommand, the dispatcher returns the subcommand's success code instead of a flag-error code and invalid flags are silently ignored.

The application instructions emphasize enumeration. For each dispatcher found, enumerate all input combinations — not just the ones with explicit case labels, but also the implicit "else" and "default" paths. For each combination, trace the return value. Any combination where the return value does not match the expected semantics is a candidate requirement. The point of the enumeration is to force the explorer to consider combinations the dispatcher does not explicitly label, because those are where the bugs live.

The output format requires the explorer to name the function and its file:line, list the input types the function dispatches on, enumerate the combinations checked (including "Condition A only," "Condition B only," "Both A and B," "Neither A nor B"), record the return value and whether it is correct for each combination, and list candidate REQ-NNN entries.

### Pattern 3: Cross-Implementation Contract Consistency

When multiple functions implement the same logical operation for different contexts — different transports, different backends, different protocol versions — they should all satisfy the same specification requirement. A step that is mandatory in the specification must appear in every implementation. A missing step in one implementation that is present in another is a strong bug signal.

The bug class this targets has a specific social origin. When the same operation is implemented in multiple places, each implementation is typically written by a different developer or at a different time. The specification says "reset must wait for completion," and the developer of implementation A writes the wait loop, but the developer of implementation B writes only the reset trigger and forgets the wait. The bug is invisible when testing implementation B in isolation because it "works" on fast hardware — the race condition only manifests under load or on slow devices.

The five examples in the reference span different stacks. A device reset specification says the driver must write zero and poll until the status register reads back zero; the PCI implementation includes the poll loop, the MMIO implementation writes zero but does not poll, and MMIO reset can race with reinitialization. A database connection-close specification says the driver must send a termination message, wait for acknowledgment, then release the socket; the PostgreSQL driver does all three, the MySQL driver sends the termination message and releases the socket without waiting, and the server may process the termination after the socket is reused. A serialization specification says all strings must be UTF-8 normalized to NFC before comparison; the JSON serializer normalizes, the protobuf serializer does not, and the same logical string compares as different depending on which serializer wrote it. A cache invalidation specification says invalidation must remove the entry and notify all subscribers; the in-memory cache does both, the distributed cache removes but does not broadcast, and other nodes serve stale data. A file locking specification says lock acquisition must set a timeout and clean up on failure; the local filesystem implementation sets the timeout, the NFS implementation uses blocking lock with no timeout, and NFS lock contention can hang the process indefinitely.

The application instructions are structural. For each core module, look for the same operation name implemented in multiple files or classes, interface or trait implementations across different backends, protocol-version-specific implementations of the same message, and transport-specific implementations of the same lifecycle operation. For each pair or set of implementations, identify the specification requirement they share, list the mandatory steps from the specification, check each implementation for each step, and record any step present in one but missing in another as a candidate requirement.

The output format requires the explorer to name the operation and its specification reference, list each implementation with its function name and file:line, enumerate the steps each implementation performs, name the gap (which step is missing from which implementation), and list candidate REQ-NNN entries.

### Admission Criteria for New Patterns

The reference file closes with an extensibility section that defines how the library grows over time. A new pattern is admitted when four conditions are met. First, it must be derived from a confirmed bug that was missed by exploration but would have been found with a specific analysis technique. Second, the technique must be generalizable — stated as a question the explorer should have asked about the code. Third, at least five diverse examples from different domains must be provided, not all drawn from the same project. Fourth, the expected output format for EXPLORATION.md must be defined, and the corresponding section must be added to the template in SKILL.md. The final admission is a dual-file update: a new entry in exploration_patterns.md and a new section in the SKILL.md template.

The stated goal is a library of systematic exploration techniques that accumulates over time as new bug classes are discovered. The three patterns shipping in v1.3.35 are the first entries, not the final set.

A pattern worth naming about the patterns themselves: each of the three targets a bug class that is common across codebases but invisible at the architectural level. Fallback path parity bugs hide in code that "works" on the happy path. Dispatcher return-value bugs hide in code that "works" when the primary input fires. Cross-implementation contract bugs hide in code that "works" when each implementation is tested in isolation. In every case, the shallow view — the architecture diagram, the subsystem summary, the single-path test — fails to expose the bug. The patterns exist precisely because shallow views are what a model produces by default, and exposing these bug classes requires the deeper view the patterns prescribe. The three patterns, taken together, amount to a portable technique for catching bugs that unaided architectural summaries miss.

The five-examples-per-pattern requirement is also significant. By spanning authentication, databases, memory allocation, interrupts, serialization, HTTP, event loops, state machines, CLIs, caching, and file locking, the reference makes clear that the patterns are not tied to any one domain. A model that has internalized fallback path parity from five different domains is more likely to recognize a new instance in a sixth domain than one that has seen the pattern only in interrupt handling. The diversity requirement is doing work: it forces abstraction at the point where the pattern is taught, so that the pattern transfers when it is applied.

## Phase Gates

A phase gate is a short, mandatory verification checklist that the model must satisfy before beginning the next phase. The gates turn phase transitions from informal decisions into explicit contracts.

### Phase 1 Completion Gate

The Phase 1 gate guards the transition from exploration to artifact generation. Before Phase 2 is allowed to begin, the model must verify five conditions.

First, `quality/EXPLORATION.md` exists on disk and contains at least sixty lines of substantive content. The sixty-line threshold is a minimum volume requirement that prevents the model from declaring exploration complete after writing a brief summary. It is arbitrary but calibrated: sixty lines is long enough to force structured findings across multiple sections but short enough not to reward padding.

Second, `quality/PROGRESS.md` exists and marks Phase 1 complete. PROGRESS.md is the pipeline's state file. Marking Phase 1 complete there provides an independent record of phase state that does not depend on the model's self-report.

Third, the model has re-read EXPLORATION.md from disk, not from memory. This is the write-then-read cycle enforced as a gate check. The model cannot proceed on the assumption that what it wrote matches what is on disk; it must verify.

Fourth, the Derived Requirements section contains at least one REQ-NNN with specific file paths and function names, not abstract subsystem descriptions. This is the specificity check — the gate's most philosophically loaded precondition. It operationalizes the depth-over-breadth principle. A model that has derived requirements like "the module should handle errors" will fail this check; a model that has derived requirements naming specific functions at specific lines will pass.

Fifth, the three new pattern sections — fallback path analysis, dispatcher return-value analysis, and cross-implementation consistency — each contain at least one finding, or an explicit "none found" with rationale. Empty sections are not acceptable. The gate specifically notes that if the codebase genuinely has no cascades, dispatchers, or cross-implementation pairs, the explorer must state that explicitly. The rationale requirement prevents the model from skipping the patterns by leaving their sections blank; it must either apply the pattern or justify why it does not apply.

The blunt instructions closing the Phase 1 gate are worth quoting directly: "Do not begin Phase 2 until all five checks pass. Phase 1 is your only chance to understand the codebase deeply. Every requirement you miss here is a bug you will not find in Phase 2b. Invest the time."

### Phase 2 Completion Gate

The Phase 2 gate guards the transition from artifact generation to code review. Before Phase 2b is allowed to begin, the model must verify four conditions.

First, all nine core artifacts exist on disk: QUALITY.md, CONTRACTS.md, REQUIREMENTS.md, COVERAGE_MATRIX.md, the functional test file, RUN_CODE_REVIEW.md, RUN_INTEGRATION_TESTS.md, RUN_SPEC_AUDIT.md, and RUN_TDD_TESTS.md. The gate enumerates the full list. Any missing artifact fails the gate.

Second, REQUIREMENTS.md contains requirements with specific conditions of satisfaction that reference actual code — file paths, function names, line numbers — not abstract behavioral descriptions. This is the Phase 2 analog of the Phase 1 specificity check. The specificity discipline is not just for exploration; it must carry forward through requirements generation. A requirement that was specific in EXPLORATION.md but became vague in REQUIREMENTS.md has lost the information that makes it useful. The gate catches that loss.

Third, if dispatch or enumeration contracts exist, the mechanical verification script `quality/mechanical/verify.sh` exists and has been executed. Mechanical verification is an earlier skill feature that checks contract consistency programmatically. The gate makes it a precondition to Phase 2b when contracts are present. This prevents the model from proceeding to code review before mechanical checks have been run and their results considered.

Fourth, PROGRESS.md marks Phase 2 complete with a timestamp. The timestamp is a small but meaningful detail. It makes completion objectively checkable and creates an audit trail across runs.

The gate also introduces an instructional reinforcement immediately after the four checks: before starting Phase 2b, re-read PROGRESS.md and REQUIREMENTS.md. The requirements are the target list for the code review — every requirement is a potential bug if the code does not satisfy its conditions. The re-read ensures the code review starts from the requirement list rather than from the model's summary impression of what was generated.

### How the Gates Enforce the Pipeline

The gates are phrased as imperatives, not suggestions. Each numbered check is a precondition the model must satisfy, and the phase cannot begin until the check passes. The gates also anchor the phase structure in disk state. Because PROGRESS.md tracks phase completion and the artifacts themselves are on disk, the model can verify its own progress by reading files rather than relying on memory. The gate checks are, in practice, file-system queries: does this file exist, does this section have content, does this line count exceed the threshold. That is by design. Disk is authoritative; memory is not.

Phase 2 itself is updated to explicitly forbid re-exploration. The revised Phase 2 instructions begin: "Use `quality/EXPLORATION.md` as your primary source for this phase — do not re-explore the codebase from scratch. The exploration findings contain the architecture map, derived requirements, use cases, and risk analysis that drive every artifact below. If you find yourself reading source files to figure out what the project does, go back to EXPLORATION.md instead. Re-exploration wastes context and produces inconsistencies between what Phase 1 found and what Phase 2 generates." The prohibition is strong and explicit. Re-exploration is not merely discouraged; it is framed as a pipeline failure mode.

The EXPLORATION.md-as-primary-source rule is a natural consequence of mandatory exploration. Once the artifact exists, it becomes the canonical input for everything downstream. The gate and the instruction together create a closed loop: Phase 1 must write EXPLORATION.md, Phase 2 must read it as its primary source, and Phase 2 cannot begin without evidence that EXPLORATION.md exists and is substantive.

The gate design also exhibits a property worth naming — the preconditions are objectively verifiable from disk state rather than from the model's self-report. Every Phase 1 check can be expressed as a shell command. Does the file exist? `test -f quality/EXPLORATION.md`. Does it exceed sixty lines? `wc -l`. Is a section header present? A grep. Is PROGRESS.md marked complete? Another grep. The check for the presence of a specific-to-abstract ratio in REQ-NNN entries is harder to automate but still externally observable. This design choice — to phrase gate conditions as disk-observable facts — is what makes the gates trustworthy. A check that depends on the model's judgment can be rationalized away; a check that depends on a file's existence cannot.

Phase 2's prohibition on re-exploration is enforceable for the same reason. If the model re-reads source files during Phase 2, that activity can in principle be observed in the agent's tool-call trace. In practice the enforcement is policy rather than instrumentation, but the policy is phrased clearly enough that deviations are recognizable in post-run review. The combination of disk-verifiable entry conditions and policy-verifiable in-phase behavior is what makes the pipeline auditable.

## Philosophical Shift — Explore Before Derive

The deepest change in v1.3.35 is not any individual feature but the reversal of an assumption that earlier versions made implicitly.

Earlier versions of the skill assumed that with enough upfront context — specifications, documentation, high-level code summaries — the model could derive requirements directly. Exploration was a setup activity: gather context, then derive. In this model, the quality of the derived requirements depends on the quality of the upfront context, and the exploration phase exists to make that context available. The underlying premise was that specifications and documentation describe the code faithfully enough that exploration can be light, and the model can focus its effort on the derivation step.

v1.3.35 reverses this assumption. Upfront context is never sufficient. Specifications are always incomplete, documentation always lags, and high-level summaries systematically elide the function-level detail where bugs live. Deriving requirements from summaries produces summary-level requirements, and summary-level requirements do not catch bugs. The correct sequence is: explore the codebase first, write function-level findings to disk, and then derive requirements from what was found rather than from what was summarized. Exploration is not the setup for derivation — exploration is the substrate from which derivation draws.

This reversal has several downstream consequences that the release makes explicit.

Exploration becomes a first-class phase rather than a preamble. Prior versions framed Phase 1 as "understand the project"; v1.3.35 frames it as the phase where requirements are elicited. The framing is slightly different and materially important. Understanding is a precondition; eliciting is the actual output. A phase that produces output has entry conditions, exit conditions, and artifacts — and Phase 1 gets all three in v1.3.35.

The EXPLORATION.md artifact becomes the primary input to every subsequent phase. This is the operational expression of the philosophical shift. If derivation is downstream of exploration, then derivation must read exploration's output. The forbidden-re-exploration rule in Phase 2 is the corresponding prohibition: if EXPLORATION.md is canonical, then reading the source again is both redundant and a threat to consistency.

Phase 2 is forbidden from re-exploring because the exploration output is already the canonical source. This is not an optimization; it is a consistency invariant. Re-exploration would produce a second set of findings that might disagree with the first. The skill cannot have two conflicting sources of truth about what the code does. Forbidding re-exploration makes EXPLORATION.md the single source.

The patterns reference exists because exploration is now systematic enough to have patterns in the first place. A preamble does not need patterns; it needs context-gathering heuristics. A first-class phase with a defined output needs a repeatable methodology, and repeatable methodologies have patterns. The decision to introduce patterns and the decision to make exploration mandatory are the same decision viewed from two angles.

The shift also changes what counts as a good requirement. Under the derivation-from-summaries model, a good requirement was one that captured a general behavioral expectation. Under the exploration-first model, a good requirement is one that specifies a concrete condition at an identifiable point in the code — a file, a function, a line, a rule. The new Phase 1 gate's fourth check enforces exactly this distinction. At least one REQ-NNN must have specific file paths and function names, not abstract subsystem descriptions. The gate is a mechanism, but the principle it encodes is the philosophical shift itself.

The operational name for this shift, as the revised SKILL.md frames it, is "depth over breadth (critical)." The new paragraph by that title is worth reading closely. A narrow scope with function-level detail finds more bugs than a broad scope with subsystem-level summaries. For each core module explored, the explorer must identify the specific functions that implement critical behavior and document them by name, file path, and line number. Requirements derived from "the reset subsystem should handle errors" will not catch bugs. Requirements derived from "`vm_reset()` at `virtio_mmio.c:256` must poll the status register after writing zero" will. The difference between a useful exploration and a useless one is specificity — file paths, function names, line numbers, exact behavioral rules.

The skill explicitly chooses depth. Explore fewer modules, but explore each one all the way down to the functions and the lines. This is the philosophical shift in one sentence.

It is worth noting what this shift does not change. The skill's ambition remains broad: it still generates nine core artifacts, still produces a multi-model spec audit, still runs the Council of Three, and still aspires to cover a full quality system rather than a narrow slice. What changes is where the depth lives. Breadth is achieved in the artifact set; depth is achieved in the exploration that grounds the artifacts. A prior-version user moving to v1.3.35 does not lose coverage; they gain specificity within the coverage. Every artifact the skill still produces is now grounded in exploration that named functions and lines rather than subsystems, and every requirement is now specific enough to either match the code or reveal a discrepancy.

The deeper intuition behind the shift is that requirements are where bug-finding begins. If requirements are abstract, code review against them is abstract; if requirements are specific, code review against them is specific. The whole downstream pipeline inherits the specificity or vagueness of the requirements. By forcing exploration to produce specific findings, and by forcing requirements to carry that specificity through to the code-level condition of satisfaction, v1.3.35 raises the precision floor of the entire pipeline. Nothing downstream changes except that it now operates on more specific inputs, and that specificity is the mechanism by which bug-finding rates improve.

## How It Fits Today

The changes in v1.3.35 became permanent architecture. Every subsequent version inherits the exploration-first model and builds on it rather than revising it.

Mandatory EXPLORATION.md is still the Phase 1 output format in v1.4.5 and v1.5.0. The file name, the section structure, and the write-then-read discipline all persist. The gate checks have been refined — later versions added more granular entry conditions and additional mandatory sections — but the underlying gate model introduced here remains the mechanism by which phases enforce their entry conditions. A reader familiar only with later versions would recognize v1.3.35 as the direct ancestor of what they are using.

The three patterns introduced in this release were the first entries in what has since become a larger library of systematic exploration techniques. `references/exploration_patterns.md` has accumulated additional patterns over time, each added according to the admission criteria specified in the original file: a confirmed bug missed by unaided exploration, a generalizable technique, diverse examples across domains, an output format for EXPLORATION.md. The three founding patterns — fallback path parity, dispatcher return-value correctness, cross-implementation contract consistency — remain the load-bearing patterns, because the bug classes they target are among the most common and most missed. Later patterns tend to address narrower bug classes or to target domain-specific anti-patterns, but they build on the framework v1.3.35 established rather than replacing it.

The phase gate pattern, once introduced, generalized. Later releases added gates at additional transitions — between Phase 2b and Phase 3, between spec audit passes, at the entry to the final review — and refined existing gates with more preconditions. The two-gate structure of v1.3.35 became the template for the multi-gate structure of subsequent versions. The shape of the gate — a short, numbered checklist of disk-verifiable conditions, followed by an imperative instruction not to proceed until all conditions are met — is the template.

The depth-over-breadth principle has also stayed. Later releases introduce scaling heuristics for very large codebases, smarter module selection for exploration, and more aggressive use of PROGRESS.md to track partial progress; but none of these relaxes the specificity requirement. Abstract requirements remain unacceptable; function-level requirements remain mandatory. The Phase 1 gate's fourth check — "at least one REQ-NNN with specific file paths and function names, not abstract subsystem descriptions" — survives in spirit and often in wording through every subsequent release.

The forbidden-re-exploration rule also survives. Every version since v1.3.35 has instructed Phase 2 to use EXPLORATION.md as its primary source and to treat re-exploration as a pipeline failure. The rule is one of the clearest invariants in the skill's evolution.

One subtle legacy of v1.3.35 is the treatment of EXPLORATION.md as a single-source-of-truth artifact rather than a backup. Before v1.3.35, the file was described in documentation as a "handoff artifact" in multi-pass mode and a "backup" in single-pass. After v1.3.35, it is described as the canonical exploration record in all modes, and downstream artifacts are described as transformations of it. This vocabulary shift has propagated through the skill's documentation and through user-facing explanations of what the skill does. Articles, specs, and patent drafts that describe the skill now tend to talk about EXPLORATION.md as the foundation rather than as a convenience.

Another consequence of v1.3.35 is that later design conversations about the skill use EXPLORATION.md as a frame. When discussing whether a new feature belongs in Phase 1 or Phase 2, the question is now "does it go into EXPLORATION.md or does it consume EXPLORATION.md." When discussing whether a new pattern is worth adding, the question is now "does it fit the admission criteria for exploration_patterns.md." When discussing failures in bug-finding, the diagnosis typically points back to an EXPLORATION.md section that was thin or missing. The artifact and its reference document have become fixtures in how the skill is reasoned about.

The empirical testing methodology introduced with v1.3.35 — running the skill against a known benchmark (the virtio driver) and counting confirmed bugs found — also became a durable practice. The release note's comparison of v1.3.33, v1.3.34, and v1.3.35 by bug count is the template for how later releases report their own effectiveness. This is a small but meaningful artifact of the release: it normalized the expectation that skill changes would be justified by measurable changes in bug-finding performance, not just by architectural tidiness.

If a single sentence captures the historical role of v1.3.35, it is this: v1.3.35 is the version where the skill stopped trusting upfront context and started demanding that the model earn its requirements by exploring the code first. Every version since has inherited that demand.
