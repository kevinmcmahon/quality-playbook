**Version:** 1.3.44
**Status:** Shipped
**Date:** 2026-04-13
**Author:** Andrew Stellman
**Primary commit:** `20bc895` — "v1.3.44: iteration mode + three regression fixes (gate hardening, pattern flexibility, triage evidence)"

---

## What This Version Introduced

Version 1.3.44 is the release that added a second way of running the skill. Until this version every Quality Playbook invocation was a fresh from-scratch analysis. v1.3.44 introduces iteration mode, an invocation that reads a prior run's `quality/EXPLORATION.md`, identifies what the previous run missed, and deliberately explores the gaps. The iteration does not replace the prior run; it supplements it. A new `quality/EXPLORATION_ITER2.md` captures the gap-targeted findings, and a merged `quality/EXPLORATION_MERGED.md` becomes the canonical input for Phase 2 through Phase 3. This is the seed concept for what v1.3.45 and later would generalize into a full iteration strategy library, but in v1.3.44 it ships as a single approach — gap-targeted re-exploration with explicit context budget discipline — documented inline in SKILL.md rather than in a standalone reference file.

Alongside the iteration mode introduction, the same commit lands three regression fixes derived from v1.3.43 benchmark failures. Each fix targets a specific failure mode observed when v1.3.43 was run against a particular benchmark repository, and each is cited by repo name in the SKILL.md prose so a future reader can trace the motivation. The first fix hardens the Phase 1 and Phase 2 gates to close a bypass that allowed chi and zod to skip exploration entirely and produce zero bugs. The second fix relaxes the pattern selection rule from "exactly 3 FULL patterns" to "3 to 4 FULL patterns" so that a fourth applicable pattern, when it covers distinct code areas, is no longer required to be demoted. The third fix lowers the triage evidentiary bar by stating explicitly that code-path traces are sufficient to confirm a bug, addressing a javalin run where four legitimate candidates were demoted because the triage demanded executed runtime evidence before confirmation.

Together the iteration mode and the three fixes represent a single release theme: raising the floor of what the skill reliably produces on runs where it otherwise underperforms. v1.3.43 was a structural redesign — three-stage exploration with gated Quality Risks — and the benchmarks exposed three distinct failure modes in three distinct repositories. The regression fixes close those failure modes directly. Iteration mode attacks a different floor-raising problem: even a well-executed first run covers a fraction of any nontrivial codebase, and a second run from scratch tends to re-find the same bugs. The total diff is 100 lines of SKILL.md changes — 86 insertions, 14 deletions, one file touched — but the conceptual surface is larger than the line count suggests. The rest of the skill is unchanged: the nine-artifact contract, the three-stage Phase 1 from v1.3.43, the Council of Three, the TDD protocol, and the mechanical verification scripts all behave as they did.

## Why It Was Needed

The four changes in v1.3.44 have a common origin: the v1.3.43 benchmark run. v1.3.43 had been released earlier the same day — the commit timestamps place v1.3.43 at 09:56 and v1.3.44 at 10:23, about twenty-seven minutes apart — and the benchmark sweep immediately after release revealed three patterns of failure that the structural redesign had not eliminated. Each shared a common feature: the skill's defenses against the failure mode existed on paper but were not tight enough in execution. Iteration mode addresses a separate problem — single-run coverage ceilings — that had become acute once per-run exploration quality was high enough that further yield gains had to come from covering more ground rather than exploring the same ground more deeply.

The chi/zod failure was the sharpest. Both repositories produced `EXPLORATION.md` files that looked plausibly structured but used completely wrong section titles — "Architecture summary," "Behavioral contracts," "Repository and architecture map" instead of the required `## Open Exploration Findings`, `## Quality Risks`, `## Pattern Applicability Matrix`, `## Pattern Deep Dive — ...`, and `## Candidate Bugs for Phase 2`. The Phase 1 gate, with its twelve numbered checks, was written as a re-read-and-verify instruction that relied on the model to run the checks and notice its own exploration was non-conformant. In both runs the model never ran the checks; it proceeded directly to Phase 2 on the assumption that what it had produced was structurally equivalent. Phase 2's entry gate was softer — a prose instruction without a hard section-title checklist. Neither gate was enforced mechanically enough to catch a model that had confidently misnamed its sections. Zero bugs resulted in both repos, a catastrophic outcome for a skill whose entire purpose is bug-finding.

The virtio pattern rigidity failure was quieter. The v1.3.43 pattern selection rule said "select exactly 3 patterns for deep-dive treatment," and the EXPLORATION.md template echoed the constraint. On virtio, four patterns were clearly applicable to distinct code areas — distinct fallback paths, distinct dispatchers, distinct cross-transport consistency questions, distinct whitelist/negotiation logic — and forcing the model to pick three meant demoting a fourth pattern that would have targeted a different class of bugs. The fix is not to loosen the constraint unboundedly; deep coverage of three patterns is still better than shallow coverage of six. The correction is to permit four when and only when the fourth pattern covers code the other three do not reach, with the default kept at three.

The javalin triage conservatism failure was the most philosophically interesting. The code review surfaced four candidate bugs with full code-path traces, and the spec audit confirmed each one with requirement violations. The triage step then demoted all four, with the rationale that "the highest-confidence items still require executed request-level evidence." This is a misreading of the playbook's evidence pipeline. The playbook is a confirm-then-prove-with-TDD system, not a prove-with-runtime-then-confirm system. The TDD protocol in Phase 2d exists to produce runtime evidence after a bug is confirmed; it is not a gate through which a code-reviewed finding must pass before it can be promoted. By demanding runtime reproduction as a prerequisite for confirmation, the triage had inverted the pipeline. Zero confirmed bugs from a repository where earlier versions had found five.

These three failures shared a common character: the skill's instructions were broadly correct but not tight enough to prevent a confident model from generating its own misinterpretation. In each case the failure was a silent pass-through rather than a crash, and the misinterpretation had surface plausibility — "my sections say the same thing even if the titles differ," "three is hard but four seems better," "we should have runtime proof before calling something a bug." The fixes in v1.3.44 are all of the same shape: name the failure mode explicitly, replace permissive instructions with imperative and verifiable ones, and cite the specific repo that exhibited the failure.

Iteration mode's motivation is separate. A first-run EXPLORATION.md typically covers three to five subsystems in depth; a library or framework repo typically has ten to twenty. The remaining subsystems contain their own bugs, and those bugs are invisible to any downstream phase because Phase 2 refuses to re-explore. Re-running the skill from scratch does not help, because the model's selection heuristics are roughly deterministic: the same subsystems surface as "most interesting" on a clean run, and the gap subsystems stay in the gap. Iteration mode lets a second run target exactly the subsystems the first run skipped, using the first run's EXPLORATION.md as its coverage map.

## Iteration Mode

The iteration mode section sits in SKILL.md immediately after the verification-phase invocation ("Run the spec audit protocol"), at the same heading level as the top-level invocation prompts. It is introduced by its own trigger prompt — "Run another iteration of the quality playbook to improve on the previous iteration." — and has its own dedicated completion gate. Structurally it is a sibling of the baseline invocation, not a mode flag on top of it.

The pivotal design idea is divide-and-conquer reading of the previous run. A first-run EXPLORATION.md in v1.3.43 is typically 200 to 400 lines, and in v1.3.44 iteration mode the prior EXPLORATION.md becomes the input rather than the starting point. Loading it all into context would burn most of the budget before the gap exploration begins, and gap exploration needs the budget more than the previous findings do — most of the previous findings will not be re-used, since the whole point is to explore elsewhere. The SKILL.md instructions state the budget explicitly: the coverage scan should consume about 20 to 30 lines of context, the targeted deep-reads should consume another 40 to 60 lines total, and the rest of the budget should be available for new exploration.

Step one is the coverage scan. The model reads just the section headers and first two to three lines of each section of the prior EXPLORATION.md, building a coverage map that records, for each section, the section name, the subsystems covered, the number of findings, and a depth classification — shallow if the section is a list of single-function mentions, deep if the section traces behavior across multiple functions. The coverage map is written to `quality/ITERATION_PLAN.md`, a new artifact introduced by this release and itself a gate check.

Step two is gap identification. From the coverage map, the model enumerates candidates for gap exploration: subsystems not explored at all in the first run, sections with shallow findings that could be deepened, Quality Risks scenarios that were listed but never traced to specific code, patterns from the applicability matrix that were marked SKIP but might apply, and domain-knowledge questions from Step 6 of the baseline Phase 1 that were raised but not addressed. The list is deliberately diverse in origin.

Step three is a targeted deep-read. For only two to three of the thinnest or most gap-rich sections, the model reads the full section content. This gives it enough specific context about the prior run to avoid duplicating findings, without spending context on sections it has classified as covered.

Step four is gap exploration proper. The model runs a focused Phase 1 against only the identified gaps, using the three-stage structure from v1.3.43 — open exploration, domain-risk analysis, selected pattern deep dives — but scoped to the uncovered areas. Findings go into `quality/EXPLORATION_ITER2.md`, using the same template as the baseline. The number (ITER2 for the first iteration, ITER3 for the next, and so on) reflects cumulative iteration count, not a strategy name.

Step five is the merge. The model creates `quality/EXPLORATION_MERGED.md`, concatenating findings section by section with attribution — `[Iteration 1]` for the baseline, `[Iteration 2]` for this run. The Candidate Bugs section is re-consolidated across both iterations. The merged file is what Phase 2 consumes. The baseline EXPLORATION.md is preserved intact, the ITER2 file is preserved intact, and the merged file becomes the single source of truth.

Step six is the continuation. Phases 2 through 3 run as they would in a baseline invocation, with `EXPLORATION_MERGED.md` as the primary input. The nine-artifact contract is unchanged, the Phase 2 entry gate still applies, and the Council of Three still runs. The only difference downstream is which exploration file the artifacts cite.

The iteration mode completion gate is a separate gate from the Phase 1 gate. It fires before Phase 2 begins and has five preconditions. `quality/ITERATION_PLAN.md` must exist and list at least three coverage gaps from the previous run. `quality/EXPLORATION_ITER2.md` must exist with at least 80 lines of substantive content — lower than the baseline's 120-line minimum because an iteration is scoped to gaps rather than the full codebase, but high enough to ensure real exploration rather than a nominal file. `quality/EXPLORATION_MERGED.md` must exist and contain findings from both iterations. The merged Candidate Bugs section must contain at least two new candidates not present in the first iteration. And at least one gap-exploration finding must cover a subsystem or module not explored in the first iteration. The last two preconditions are the ones that ensure the iteration actually produced net-new value; a run that finds no new candidates and explores no new subsystems fails the gate, and that failure is itself a useful signal that the codebase's bug yield may be saturated or that a different iteration strategy is needed.

Two properties of the v1.3.44 iteration design are worth naming. First, it does not yet call itself a strategy. The word "strategy" does not appear in the iteration-mode section. There is only one way to iterate — gap-targeted re-exploration — and it is described as the behavior of iteration mode, not as one of several interchangeable strategies. The strategy framing is what v1.3.45 adds when it extracts the iteration instructions into a standalone reference file and adds parity alongside gap. Second, the iteration's Phase 1 is scoped to gaps but still goes through the three-stage structure — open exploration, quality risks, selected patterns. Step four is explicit about this. The reasoning matches the three-stage design from v1.3.43: domain-driven exploration finds bug classes that structural analysis misses, and skipping the open stage in the name of focus would repeat the mistake the three-stage design was introduced to fix. The scope narrows; the method does not.

## Gate Hardening (Fix 1)

The gate hardening addresses the chi/zod failure mode and has three interlocking pieces: a new Gate Self-Check section in EXPLORATION.md, a hardened Phase 2 entry gate with an explicit section-title checklist, and a failure-mode narrative in SKILL.md that cites v1.3.43 chi/zod by name.

The Gate Self-Check section is the most mechanical of the three changes. Prior to v1.3.44, the Phase 1 completion gate instructed the model to re-read EXPLORATION.md and verify twelve numbered checks, with the implicit expectation that the model would track pass/fail internally. The v1.3.44 version still has the twelve checks, but it now requires the model to append a `## Gate Self-Check` section to EXPLORATION.md that lists each check number from 1 through 12 with a PASS or FAIL verdict and a one-line evidence note. The skill's instructions explicitly call out the logic: "This section proves the gate was executed. Do not write this section until you have actually verified each check against the file contents." The section must be on disk, not in memory; a model that executed the checks mentally but did not write the section has not passed the gate. The prior gate was trust-based — the model asserted "I checked" and the skill proceeded. The new gate is evidence-based: the model writes its check results to disk and the Phase 2 entry gate verifies the section exists before proceeding.

The Phase 2 entry gate rewrite is the second piece. The v1.3.43 entry gate was a single paragraph that instructed the model to read EXPLORATION.md and stop if the file was missing sections required by the Phase 1 completion gate. The v1.3.44 entry gate is an explicit numbered checklist of exact section titles: `## Open Exploration Findings`, `## Quality Risks`, `## Pattern Applicability Matrix`, at least three sections starting with `## Pattern Deep Dive — `, `## Candidate Bugs for Phase 2`, and `## Gate Self-Check`. Each is specified as required verbatim. The instruction explicitly forbids accepting equivalent sections under different names: "Do not attempt to proceed with 'equivalent' sections under different names — the exact titles above are required." The gate is now mechanically verifiable — a grep for each section title either finds it or does not — and it rejects semantic equivalence as a justification for non-conformance.

The third piece is the failure-mode narrative. The revised Phase 1 gate includes a paragraph titled "Common gate-bypass failure mode" that describes the chi/zod failure directly, naming the actual wrong section titles those runs produced and stating that "the model never ran the gate checks and proceeded directly to Phase 2, producing zero bugs." A similar callout appears at the end of the new Phase 2 entry gate: "In v1.3.43, two repos bypassed both gates and produced zero bugs." The narrative is instructional and auditable — the benchmarks are publicly identifiable and the zero-bugs claim is verifiable against benchmark logs.

The gate-closing imperative is also sharpened. The v1.3.43 gate ended with "Do not begin Phase 2 until all twelve checks pass." The v1.3.44 gate ends with "Do not begin Phase 2 until all twelve checks pass AND the `## Gate Self-Check` section is written to EXPLORATION.md on disk." A new paragraph follows: "If you find yourself about to start Phase 2 without having written a Gate Self-Check section, STOP. Go back and run the gate. This instruction exists because models reliably skip the gate when they feel confident about their exploration — and that confidence is precisely when bugs are missed." The diagnosis — confidence is the predictor of the gate being skipped — is itself the designer's empirical finding, encoded into the skill's prose so the model reading the skill is warned about its own failure mode. The headers also escalate: the v1.3.43 "Phase 1 completion gate (mandatory)" becomes "Phase 1 completion gate (mandatory — STOP HERE before Phase 2)," and the Phase 2 gate becomes "Phase 2 entry gate (mandatory — HARD STOP)."

A deeper design lesson is embedded in this fix. The prior gate was advisory — it told the model what to check and trusted the model to act on its own check results. The new gate is a write-through gate — check results must be committed to an externally visible medium before the next phase is allowed to begin. Advisory gates fail silently when the model's self-assessment is wrong; write-through gates cannot, because the write is the pass condition. The same principle governs the mandatory EXPLORATION.md discipline introduced in v1.3.35, and v1.3.44 extends it to the gate check itself. The gate on the gate is an artifact on disk.

## Pattern Flexibility (Fix 2)

The pattern flexibility fix is the smallest change in the commit by line count and the most straightforward in its logic. Three occurrences of "exactly 3" are replaced with "3 to 4," and a supporting sentence is added explaining when the fourth pattern is justified.

The change lands in three places in SKILL.md. Phase 1 Step 3 is rewritten from "select exactly 3 patterns for deep-dive treatment" to "select 3 to 4 patterns." The sentence immediately following becomes "depth on 3–4 beats shallow coverage of 6. Select 4 when a fourth pattern has clear applicability and would cover code areas not reached by the other three; default to 3 when in doubt." The EXPLORATION.md template's Pattern Applicability Matrix note changes from "Exactly 3 patterns must be marked FULL" to "3 to 4 patterns must be marked FULL," and the Pattern Deep Dive placeholder note changes from "Exactly 3 deep-dive sections total" to "3 to 4 deep-dive sections total."

Checks 8 and 9 of the Phase 1 completion gate enforce the matrix and deep-dive count. Check 8 becomes "Between 3 and 4 patterns (inclusive) are marked `FULL` in the applicability matrix." Check 9 becomes "There are between 3 and 4 sections (inclusive) whose titles begin with `## Pattern Deep Dive — `. The count must match the number of `FULL` patterns in the matrix." The added clause — deep-dive count must match FULL count — is an important consistency invariant. A model that marks four FULL but writes three deep dives fails the gate, and vice versa. Check 10, the pattern depth check, is rephrased to apply regardless of total: "At least 2 of the pattern deep-dive sections must trace a code path across 2 or more functions." The threshold of two depth findings does not scale with the total count; two is the minimum evidence that patterns were applied with genuine tracing.

The design restraint is worth dwelling on. The constraint could have been loosened to "3 to 6 patterns," but the fix caps at four. Two empirical beliefs underlie the cap. First, a model doing deep dives on more than four patterns in a single Phase 1 tends to produce shallow dives as attention divides. Second, the marginal yield of the fifth and sixth deep dives is low on typical codebases, and the same coverage can be achieved more reliably by running an iteration — which is now available in the same release. A user who wants all six patterns applied can run a baseline with three to four patterns and then an iteration that targets the remaining patterns as gap areas. Pattern flexibility and iteration mode are complementary: the first loosens the single-run constraint modestly, the second provides the mechanism for going beyond it.

The empirical motivation is the virtio benchmark. Four of the six patterns in `exploration_patterns.md` apply cleanly to distinct parts of virtio — fallback parity in the interrupt cascade, dispatcher return-value correctness in the IRQ handler, cross-implementation contract consistency between PCI and MMIO transports, and API surface consistency across feature negotiation. Forcing the model to pick three meant demoting a fourth that would have surfaced bugs in a code area the other three did not reach. The fix makes the four-pattern case legitimate without making it the norm.

## Triage Evidence (Fix 3)

The triage evidence fix lands in two places: a new subsection in the BUGS.md artifact specification titled "What counts as sufficient evidence to confirm a bug (critical)," and a new paragraph in the spec audit section's triage rules titled "Triage must not raise the evidentiary bar above code-path analysis." The two changes are coordinated — they establish the same evidentiary standard from two different angles — and a cross-reference between them lets a reader discover one from the other.

The BUGS.md subsection enumerates four specific types of evidence sufficient to confirm a bug without runtime reproduction. A code-path trace showing function A calls function B which does X but should do Y, with file:line references, is sufficient. A missing case or branch identified by enumeration comparison — the spec says X should be handled and the code has no handler — is sufficient. A requirement violation identified in Pass 2 of the spec audit, where the code demonstrably does not implement the specified behavior, is sufficient. A domain-knowledge finding where the input can be traced through specific code to the wrong output is sufficient. The enumeration is not exhaustive but is broad enough to cover the evidence types that the code review and spec audit phases produce.

The subsection then states the converse in prose: "Do NOT demand 'executed request-level evidence' or defer findings because 'they require runtime testing to distinguish implementation choice from spec gap.' If the spec or documentation says the behavior should be X, and the code demonstrably produces Y (traceable through the code path), that is a confirmed bug — not a candidate awaiting runtime validation. The regression test and TDD protocol exist to provide runtime evidence AFTER confirmation, not as a prerequisite FOR confirmation." The phrasing "after confirmation, not as a prerequisite for confirmation" names the order the pipeline is supposed to run in, and "implementation choice from spec gap" is the specific rationale the javalin triage used. Both phrases are quoted into the skill's prose so a future reader can tell which failure mode is being addressed.

The failure-mode narrative at the end of the subsection parallels the gate-hardening narrative: it quotes the exact "executed request-level evidence" rationale that the javalin triage used, reports the outcome ("zero confirmed bugs from a codebase where previous versions found 5"), and states the design principle in a single line — "the playbook's design is confirm-then-prove-with-TDD."

The spec audit section's triage rule is the second half of the fix. It lives at the end of the spec audit triage section, after the minority finding rule: "Triage must not raise the evidentiary bar above code-path analysis. The triage step confirms or rejects findings — it does not defer them pending runtime evidence." It closes with a cross-reference to the BUGS.md subsection. The doubling is intentional. The BUGS.md subsection is what the model reads when generating the bug report; the spec audit triage rule is what it reads when running the triage step. Placing the rule in both sections ensures the model encounters it at both decision points — either alone could be forgotten.

The philosophical content of this fix surfaces an ambiguity the earlier skill left implicit. The Quality Playbook has always been a code-review-first system: code review finds most bugs; the TDD protocol generates regression tests for bugs already found. But a code-review-first system depends on the reviewer trusting its own analysis enough to call a finding a bug when the code-path evidence is clear, rather than deferring to runtime as the arbiter of truth. The javalin failure was the model failing to trust its own review. The fix restores the code-review-first philosophy by stating explicitly what was previously implicit: code-path traces are primary evidence; runtime execution is secondary evidence produced by the regression test after the bug is confirmed.

A corollary, not explicitly spelled out in v1.3.44 but visible in the amended rules, is that the playbook is biased toward recall rather than precision. A code-path-traced finding that turns out to be a false positive will be exposed by the TDD red/green cycle downstream. A finding demoted prematurely is a bug that no downstream phase will recover. The asymmetry favors confirmation: a false positive is detectable later, a false negative is not.

## How the Four Changes Interact

The four changes in v1.3.44 are nominally independent, but they share structural features that make the release more coherent than its diff suggests. All three regression fixes share the same rhetorical shape: a behavioral rule, stated in imperative language, followed by a failure-mode narrative that cites a specific benchmark repository by name. The chi/zod citation grounds the gate hardening, the virtio reasoning grounds the pattern flexibility, and the javalin citation grounds the triage evidence rule. A rule without a rationale is easy to overlook; a rule attached to a specific empirical failure is harder to rationalize away. The failure-mode narrative serves as a teaching aid for the model reading the skill and as an audit trail for future releases.

The gate hardening and the iteration mode have the largest conceptual footprint, and they interact in a specific way. Iteration mode depends on a prior run's EXPLORATION.md being structurally conformant — the coverage scan reads section headers, and if the section headers are wrong the coverage map cannot be built. A first run that bypassed the Phase 1 gate and produced an EXPLORATION.md with "Architecture summary" instead of `## Open Exploration Findings` is a first run that cannot be iterated against. By enforcing that every first run produces a conformant EXPLORATION.md, the gate hardening makes iteration mode's divide-and-conquer reading strategy viable. The two changes are not directly coupled in the code, but they are coupled in the workflow.

The pattern flexibility fix and iteration mode are coupled in a different way. Before v1.3.44, a codebase where four patterns clearly applied forced the first-run model into a hard tradeoff — pick three, demote the fourth, and accept that the bugs the fourth pattern would have surfaced are lost. After v1.3.44, the model can either pick four patterns in the first run (if they cover distinct code areas) or pick three and recover the fourth's coverage in an iteration's gap exploration. Pattern flexibility handles the obvious case; iteration mode handles the uncertain one. Together they reduce the scenarios in which a legitimate pattern gets silently dropped.

All four changes are ultimately floor-raising mechanisms. The gate hardening raises the floor against silent Phase 2 entry without exploration. The pattern flexibility raises the floor against over-constrained pattern selection. The triage evidence fix raises the floor against over-conservative triage. Iteration mode raises the floor against single-run coverage ceilings. Each change targets a different failure mode, but each is about the same underlying problem: reliably getting the skill to find the bugs it is capable of finding, rather than losing them to procedural failures at the seams between phases.

## How This Sets Up What Comes Next

v1.3.45 is committed about six hours after v1.3.44 and makes three changes: extracting the iteration strategy details from SKILL.md into a standalone `references/ITERATION.md` reference file, adding a parity strategy alongside the gap strategy, and introducing a suggested-next-prompt UX at the end of each run. The SKILL.md iteration section shrinks from 115 lines to 27 lines in the process. The extraction is a direct consequence of the release pattern v1.3.44 initiated: the inline iteration-mode section grew large enough in v1.3.44 that a reference file became the natural home, and the monolithic gap-targeted approach grew a sibling once parity was added.

The seed concept in v1.3.44 is that there is one way to iterate — gap-targeted re-exploration — and the instructions live inline in SKILL.md. v1.3.45 replaces that with a multi-strategy framework: gap, unfiltered, parity, adversarial, each targeting a different class of bugs, with a recommended cycle order and a shared rules section. The adversarial strategy, which challenges dismissed or demoted findings from previous runs, has a direct ancestor in v1.3.44's triage evidence fix: if the baseline triage demotes a finding, the adversarial iteration is the mechanism by which that finding can be recovered with fresh evidence. Without the triage evidence rule of v1.3.44, the adversarial strategy would have a weaker foundation — it would be arguing against a triage rule the skill itself did not explicitly articulate.

v1.3.46 adds the demoted candidates manifest, parity sub-types, an adversarial bar, and TDD execution in iterations. The demoted candidates manifest tracks demoted findings so that the adversarial strategy can consult them, and the motivation for tracking demoted findings at all originates in the javalin failure that v1.3.44's triage fix addressed. The TDD execution rule for iterations extends the confirm-then-prove-with-TDD philosophy v1.3.44 articulated into iteration runs specifically. v1.3.47 through v1.3.49 continue to build on the iteration framework, but the primitive itself — a second invocation reads the first's EXPLORATION.md, identifies gaps, writes an ITER file, merges into a MERGED file, and runs Phase 2 against the merged input — remains stable. The file naming convention (`EXPLORATION_ITER{N}.md`, `EXPLORATION_MERGED.md`, `ITERATION_PLAN.md`), the divide-and-conquer reading discipline, the completion gate structure, and the 80-line minimum for an iteration's exploration all carry forward unchanged.

The gate hardening also persists through later releases. The Gate Self-Check section continues to be mandatory, and later revisions of the Phase 1 and Phase 2 gates refine the check list but retain the write-through gate design — check results are committed to disk, not asserted in memory. The pattern flexibility range of 3 to 4 becomes the stable setting. The triage evidence rule remains in both the BUGS.md subsection and the spec audit section, and the confirm-then-prove-with-TDD philosophy it encodes becomes a load-bearing principle in later design documents.

If a single sentence captures the historical role of v1.3.44, it is this: v1.3.44 is the release that introduced iteration mode as an inline feature and tightened three seams that v1.3.43 benchmarking had revealed as leaky. The iteration mode is the seed v1.3.45+ formalized into a strategy framework; the three fixes became permanent fixtures in how the skill enforces its own contracts.
