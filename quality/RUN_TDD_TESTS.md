# TDD Tests Protocol

Specifies the test-driven-development protocol that must execute for every confirmed bug in `quality/BUGS.md`. The protocol's invariant is: **every confirmed bug has a test that failed before the fix and passes after.** Sessions that skip the red or green phase produce sidecar verdicts that the gate rejects.

This protocol is exercised across Phase 3 (red phase — writing the failing test) and Phase 5 (green phase — applying the fix, flipping the verdict). Its output is `quality/results/tdd-results.json` (schema version `"1.1"`).

## Verdict enum

Per SKILL.md:154 and `quality_gate.py:497-498`:

- `TDD verified` — test exists, failed before fix, passes after fix. Full red-green cycle complete.
- `red failed` — test was expected to fail but passed, so the bug may be misstated or already fixed. Revisit the bug.
- `green failed` — test failed after the fix; the fix did not resolve the bug. Revisit the fix.
- `confirmed open` — bug is real, test fails, but no fix is attempted this cycle. Often used for deferred work or architectural issues.
- `deferred` — bug is acknowledged but deliberately not tested this cycle (e.g., requires design work, infrastructure the project lacks).

Each verdict is an exact string. Case and spacing are load-bearing. A parenthetical suffix is allowed only on `confirmed open` (e.g., `confirmed open (xfail)`).

## Red phase (Phase 3)

Red phase happens during the code review described in `RUN_CODE_REVIEW.md`. For each confirmed bug:

1. **Write the test first.** Before any fix discussion, write a `test_reg_cbN_<slug>` test in `quality/test_regression.py` that asserts the *desired* behavior. The test must fail against the current code.
2. **Mark `@unittest.expectedFailure`.** The `unittest` framework reports expected-failure tests as `x` (xfail). A test decorated with `@unittest.expectedFailure` that unexpectedly passes is an UNEXPECTED PASS (`X`) which fails the suite.
3. **Record the transcript.** Save the full output of `python3 -m unittest quality.test_regression -v` to `quality/results/tdd-red-phase.log`. Every test name must appear; every verdict must be `x` (xfail) or `F` (unexpected failure — this is a test writing error, not a bug).
4. **Write the sidecar entry (preliminary).** For each bug, add an entry to `quality/results/tdd-results.json` with verdict `confirmed open (xfail)` — the red phase is complete, green is not yet attempted.

**Red-phase test template.**

```python
@unittest.expectedFailure
def test_reg_cb3_line_count_threshold_drift(self):
    """CB-3 / REQ-004: Phase 2 gate must FAIL below 120 lines (currently WARN at 80).

    Source: run_playbook.py:455-457.
    """
    from tempfile import TemporaryDirectory
    from pathlib import Path
    with TemporaryDirectory() as tmp:
        (Path(tmp) / "quality").mkdir()
        (Path(tmp) / "quality" / "EXPLORATION.md").write_text("x\n" * 100, encoding="utf-8")
        result = run_playbook.check_phase_gate(Path(tmp), "2")
        # REQ-004: under 120 lines must FAIL. Current behavior is PASS with a
        # WARN. This assertion will fail today; once REQ-004 is implemented,
        # remove the @unittest.expectedFailure decorator.
        self.assertFalse(result.ok,
                         "Phase 2 gate must FAIL under 120 lines per REQ-004")
```

**Red-phase invariants.**
- Every test has a bug-ID and requirement-ID in its docstring.
- Every test cites the exact file:lines under test.
- No test is skipped with `@unittest.skip` or `self.skipTest` — deferred bugs use verdict `deferred` in the sidecar, not a suite-level skip. An xfail is how the sidecar's "confirmed open" status shows up in the test suite.
- The total test count in the transcript equals the number of confirmed bugs plus any deferred-with-xfail entries.

## Green phase (Phase 5)

Green phase happens during Phase 5 reconciliation. For each bug that had a red-phase test:

1. **Apply the patch.** The patch in `quality/patches/BUG-NNN-fix.patch` is applied by hand (the playbook runner does not apply it — `cleanup_repo` would revert direct source edits). A maintainer runs `git apply quality/patches/BUG-NNN-fix.patch`.
2. **Remove the `@unittest.expectedFailure` decorator** from the corresponding test.
3. **Re-run.** Execute `python3 -m unittest quality.test_regression -v` and save the full output to `quality/results/tdd-green-phase.log`. The previously-xfail test must now report `ok` (the `.` character in verbose output).
4. **Update the sidecar.** Change the verdict for this bug to `TDD verified`.

**Red→green transition invariants.**
- After removing the xfail decorator, the test passes. If the test fails, the fix is incomplete — verdict becomes `green failed` and the patch goes back to the fix author.
- The test file's imports, fixture, and assertion structure do NOT change between red and green. Only the `@unittest.expectedFailure` line is removed (and possibly one surrounding comment).
- If the test needs *structural* changes to pass (e.g., asserting a different value), the red-phase test was wrong. Write a new test; keep the old one documented in the writeup with verdict `red failed`.

## Sidecar schema

`quality/results/tdd-results.json` — schema version `"1.1"`. Full template:

```json
{
  "schema_version": "1.1",
  "skill_version": "1.4.5",
  "project": "quality-playbook",
  "model": "claude-opus-4-7",
  "model_provider": "Anthropic",
  "runner": "claude-code",
  "date": "2026-04-18",
  "bugs": [
    {
      "bug_id": "BUG-001",
      "requirement": "REQ-001",
      "candidate_origin": "CB-1",
      "title": "Version-parser divergence silently disables bare-name fallback",
      "file": "bin/benchmark_lib.py",
      "lines": "90-116",
      "severity": "HIGH",
      "verdict": "confirmed open (xfail)",
      "test": "quality/test_regression.py::test_reg_cb1_version_parser_divergence",
      "patch": "quality/patches/BUG-001-fix.patch",
      "notes": "Test written in Phase 3; patch written but not applied. Phase 5 applies and flips verdict to 'TDD verified'."
    }
  ]
}
```

**Required fields per bug.** `bug_id`, `requirement`, `title`, `file`, `lines`, `severity`, `verdict`, `test`. `patch` is required when verdict is `confirmed open (xfail)` or `TDD verified`; omitted for `deferred`.

**Schema validation.** The gate's `check_tdd_sidecar` function (quality_gate.py:445+) checks `schema_version`, the presence of `bugs[]`, and that each `verdict` is in the enum. Tests in `.github/skills/quality_gate/tests/test_quality_gate.py` cover these cases.

## Deferred verdict handling

A bug with verdict `deferred` still appears in the sidecar but:
- No entry in `test_regression.py` is required.
- `patch` field is omitted.
- `notes` must name the deferral reason (e.g., "requires architectural redesign; tracked in issue #NN").

The gate accepts `deferred`. This is the escape hatch for bugs that are genuinely out of scope for this cycle. Abuse — marking everything deferred — is caught by the QUALITY.md §Human Gate section: a human reviewer confirms the deferral is honest.

## Running the suite

From the QPB root:

```bash
# Red phase (Phase 3 end):
python3 -m unittest quality.test_regression -v 2>&1 | tee quality/results/tdd-red-phase.log

# Apply patches manually (Phase 5):
for p in quality/patches/BUG-*-fix.patch; do git apply "$p"; done

# Remove @unittest.expectedFailure from the fixed tests (Phase 5).
# Then re-run:
python3 -m unittest quality.test_regression -v 2>&1 | tee quality/results/tdd-green-phase.log
```

Every line in the transcript that starts with `test_reg_` must end in:
- `ok` → test passed (green phase)
- `expected failure` → test failed as expected (red phase, still xfail)
- `unexpected pass` → test passed while marked xfail — FAIL for the suite, indicates a broken bug claim
- `FAIL` → test failed (red phase, unexpected result) or failed while unmarked (green phase)

## Integration with Phase 6 verification

Phase 6 (verification) runs `quality_gate.py` end-to-end. The gate reads `tdd-results.json` and:
- Counts bugs by verdict.
- Ensures schema_version is "1.1".
- Confirms `date` is a valid ISO 8601 date (not a placeholder).
- Cross-checks `skill_version` against the installed SKILL.md.
- Verifies every referenced test name exists in `quality/test_regression.py` (via file scan).

A run that reaches Phase 6 with a correctly populated sidecar and a green-phase transcript should produce 0 FAIL.

## Bootstrap-specific notes

In the bootstrap self-audit, green phase is executed by a maintainer outside the playbook runner — the runner's `cleanup_repo` would revert direct source edits. The maintainer runs `git apply` on each patch, removes the xfail decorator, and re-runs the test suite. Only after that does the bug's verdict flip to `TDD verified`.

For this specific run (2026-04-18 bootstrap), Phase 3 will produce red-phase xfail tests but Phase 5 green-phase is intentionally deferred to the next release cycle — the patches will be reviewed before any source change. Verdicts will be `confirmed open (xfail)` across the board, with `deferred` reserved for bugs the maintainer explicitly wants to defer.
