# TDD Verification Protocol: Quality Playbook (Bootstrap Self-Audit)

> Generated by [Quality Playbook](https://github.com/andrewstellman/quality-playbook) v1.4.0 — Andrew Stellman
> Date: 2026-04-15 · Project: quality-playbook (bootstrap self-audit)

## Purpose

This protocol runs the red-green TDD cycle for each confirmed bug found during code review (Phase 3) and spec audit (Phase 4). For each bug:
- **Red:** Run the regression test against unpatched source. It must FAIL.
- **Green:** Apply the fix patch, run the same test. It must PASS.

## Framework

This project uses bash scripts for testing. The functional and regression tests are shell scripts that grep/awk the source files to verify spec-level properties. The test framework is the custom `test_functional.sh` runner.

## Spec-grounded test requirements

For each bug in `quality/BUGS.md`:
1. Read the bug's **spec basis** field to identify the documentation passage defining expected behavior.
2. Read the gathered doc at the cited section.
3. Write test assertions using language from the spec.
4. Include a comment block citing: requirement ID, bug ID, and spec passage.

## Red-green execution steps

For each confirmed bug with a regression test patch:

```bash
# ── Red phase: run test against unpatched source, expect FAIL ──
TEST_CMD="bash quality/test_functional.sh test_req001_summary_key_consistency"
OUTPUT=$($TEST_CMD 2>&1); EXIT=$?
printf 'RED\n--- Test output for BUG-NNN red phase ---\nCommand: %s\nExit code: %d\n%s\n' \
  "$TEST_CMD" "$EXIT" "$OUTPUT" > quality/results/BUG-NNN.red.log

# ── Green phase: apply fix, run test, expect PASS ──
git apply quality/patches/BUG-NNN-fix.patch
OUTPUT=$($TEST_CMD 2>&1); EXIT=$?
printf 'GREEN\n--- Test output for BUG-NNN green phase ---\nCommand: %s\nExit code: %d\n%s\n' \
  "$TEST_CMD" "$EXIT" "$OUTPUT" > quality/results/BUG-NNN.green.log
git apply -R quality/patches/BUG-NNN-fix.patch  # revert
```

Adapt the `TEST_CMD` for each bug's specific test function. Run this for every confirmed bug.

**Required top-level fields:** `schema_version`, `skill_version`, `date`, `project`, `bugs`, `summary`.
**Required per-bug fields:** `id`, `requirement`, `red_phase`, `green_phase`, `verdict`, `fix_patch_present`, `writeup_path`.
**Required summary sub-keys:** `total`, `verified`, `confirmed_open`, `red_failed`, `green_failed`.

Valid verdict values: `"TDD verified"`, `"red failed"`, `"green failed"`, `"confirmed open"`, `"deferred"`.

## Post-write validation

After writing tdd-results.json, reopen and verify:
1. Every required top-level field is present.
2. Every required per-bug field is present in each `bugs[]` entry.
3. All verdict, red_phase, and green_phase values use allowed enum values.
4. No extra undocumented root keys exist.

## Bug Writeup Generation

For each TDD-verified bug, generate `quality/writeups/BUG-NNN.md` with sections:
1. Summary
2. Spec reference
3. The code
4. Observable consequence
5. Depth judgment (if warranted)
6. The fix (inline diff required)
7. The test
8. Related issues (if any)
