### 1. Input Validation and Boundary Handling

#### quality_gate.sh

- **Line 76–78:** [BUG] `json_has_key()` uses `grep -q "\"${key}\""` to check for JSON key presence. This performs a flat text search, not structured JSON parsing. If a key name appears inside a string value (e.g., `"notes": "the schema_version was wrong"`), it will match as if the key `schema_version` exists at root level. This means a bug entry with `"notes": "missing verdict field"` would cause `json_has_key` to report that a `verdict` key exists even if it does not. Expected: structured JSON key checking. Actual: flat grep that matches key names anywhere in the file, including inside string values.

- **Line 82–84:** [BUG] `json_str_val()` uses `grep -o` to extract the first occurrence of a key-value pair. When the same key appears in multiple JSON objects (e.g., `"verdict"` appears once per bug entry), `head -1` returns only the first occurrence. This is correct for root-level keys like `"schema_version"` but incorrect for extracting per-entry values. The function is used safely for root keys (lines 204–206, 239, 334, 539) but would silently return wrong data if used for per-bug fields. Expected: documented limitation. Actual: no documentation of single-occurrence constraint.

- **Line 507:** [BUG] `for wf in ${q}/writeups/BUG-*.md` — the glob is unquoted and has no `nullglob` guard. When no writeup files exist, the loop iterates once with `wf` set to the literal string `${q}/writeups/BUG-*.md`. The `[ -f "$wf" ] || continue` guard on line 508 catches this (it tests whether the unexpanded glob literal is a file, which it is not, so it continues). However, this is fragile — it relies on the `-f` test failing on the literal glob string rather than explicitly handling the empty-match case. If a file named literally `BUG-*.md` existed (unlikely but possible), the guard would pass incorrectly. Expected: `shopt -s nullglob` before the loop or `compgen -G` check. Actual: relies on `-f` test failing against unexpanded glob literal.

- **Line 116:** [BUG] `ls ${q}/code_reviews/*.md 2>/dev/null` — unquoted glob passed to `ls`. On paths containing spaces, this will word-split and fail. The `$q` variable comes from `"${repo_dir}/quality"` (line 97) which IS quoted at assignment, but the `ls` invocation is unquoted. Expected: proper quoting or use of `find`. Actual: unquoted glob expansion vulnerable to spaces in paths.

- **Lines 125–126:** [BUG] Same unquoted glob issue: `ls ${q}/spec_audits/*triage*` and `ls ${q}/spec_audits/*auditor*`. Both are piped through `wc -l` and would fail if the path contains spaces. These produce correct counts but emit error messages to stderr (suppressed by `2>/dev/null`). Fundamentally the same word-splitting vulnerability as line 116.

- **Lines 396–405:** [QUESTION] `eval "find '${repo_dir}' ..."` — using `eval` with a variable interpolated into the command string. The `repo_dir` is single-quoted inside the eval string, but the `find_exclude` variable on line 395 is constructed with pre-expanded single quotes that become literal strings after eval. This works correctly for normal paths but introduces an injection risk if `repo_dir` contains a single quote. Expected: direct `find` invocation without `eval`. Actual: `eval` used for convenience of expanding the pre-built exclude pattern string.

### 4. Unit and Encoding Correctness

#### quality_gate.sh

- **Line 61–66:** [QUESTION] Version detection uses `grep -m1 'version:' "$loc"` followed by `sed 's/.*version: *//' | tr -d ' '`. This extracts the FIRST line containing "version:" from SKILL.md. In the frontmatter, this correctly captures `version: 1.4.0`. However, if any line before the frontmatter `version:` contains the word "version:" (e.g., in comments), it would extract the wrong value. Since SKILL.md's frontmatter is at the top (lines 1–9), and the `version:` field is on line 6, this is safe for the current file structure. Not a bug, but fragile.

- **Line 248–250:** [QUESTION] Date future-check uses string comparison: `if [[ "$tdd_date" > "$today" ]]`. This works for ISO 8601 dates (YYYY-MM-DD) because lexicographic ordering matches chronological ordering for this format. Correct for the intended use.

- **Lines 351–354:** [QUESTION] Use case counting uses `grep -cE 'UC-[0-9]+'` to count total occurrences and `grep -oE 'UC-[0-9]+' | sort -u | wc -l` for unique count. The regex `UC-[0-9]+` will match `UC-0`, `UC-1`, `UC-01`, `UC-123`, etc. It will also match embedded occurrences like `DOCUMENT-UC-01`. This is unlikely to cause false positives in practice since `UC-NN` is a controlled format in the requirements document.
