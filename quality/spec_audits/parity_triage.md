## Triage: CAND-P1 — json_has_key vs json_key_count inconsistency in summary checks

**Found by:** All three auditors (highest confidence)
**Code location:** `quality_gate.sh:259-265` (summary check) vs `quality_gate.sh:239-248` (per-bug check)

**Verification probe:** Does `json_has_key` actually produce a false positive for summary keys in practice?

The summary keys are: `total`, `verified`, `confirmed_open`, `red_failed`, `green_failed`.

Consider a `tdd-results.json` where the `notes` field of a bug reads: `"notes": "total bugs confirmed, all verified"`. The string `"total"` appears in the file without a following `:`. `json_has_key` uses `grep -q '"total"'` which matches `"total"` anywhere — including inside the notes string value. The gate would report PASS for "summary has 'total'" when no actual `total:` key exists in the summary object.

This is a lower-probability false positive than BUG-H1 (which can match in arbitrary string values), but the same root cause. The fix is the same: add colon anchor or use `json_key_count`.

**Categorization:** Real code bug — the function `json_has_key` has a confirmed false-positive pattern (BUG-H1), and using it in the summary check creates the same vulnerability. The inconsistency with the per-bug check (which correctly uses `json_key_count`) is an additional design defect.

**Verdict: CONFIRMED — BUG-L19** (net-new, LOW severity)

## Triage: CAND-P2 — Aggregate patch count vs per-bug log iteration

**Found by:** All three auditors (highest confidence)
**Code location:** `quality_gate.sh:562-588` (aggregate patch count) vs `quality_gate.sh:316-345` (per-bug log iteration)

**Verification probe:** Is the aggregate count approach actually incorrect, or just less precise?

Testing scenario: 2 confirmed bugs (BUG-H1, BUG-M3). Patches present: `BUG-H1-regression-test.patch` and `BUG-H1-regression-test-v2.patch` (duplicate due to agent error). `BUG-M3-regression-test.patch` absent.

- `reg_patch_count=$(ls ${q}/patches/BUG-*-regression*.patch 2>/dev/null | wc -l)` → returns 2 (counting both BUG-H1 patches)
- `[ "$reg_patch_count" -ge "$bug_count" ]` → `2 >= 2` → PASS

The gate reports "2 regression-test patch(es) for 2 bug(s)" — spurious PASS. BUG-M3 has no patch.

This IS a real failure mode, not just a theoretical concern. The per-bug iteration approach (like lines 316-345 for TDD logs) would iterate `bug_ids = "BUG-H1 BUG-M3"` and detect BUG-M3's missing patch.

**Categorization:** Real code bug — aggregate count can produce false PASS when patch set is wrong. Per-bug iteration is the correct approach (already used for TDD logs).

**Verdict: CONFIRMED — BUG-L20** (net-new, LOW severity)

## Triage: CAND-P4 — SEED_CHECKS.md absent from artifact contract table

**Found by:** All three auditors (highest confidence)
**Code location:** `SKILL.md:88-119` (artifact contract table) vs `SKILL.md:1641` (Phase 5 gate)

**Verification probe:** Does SKILL.md line 88 really say the table is "canonical"?

Reading SKILL.md line 85-88:
> "The quality gate (quality_gate.sh) validates these artifacts. If the gate checks for it, this skill must instruct its creation. This is the canonical list — any artifact not listed here should not be gate-enforced, and any gate check should trace to an artifact listed here."

This is a self-referential claim: the table is canonical AND gate checks must trace to it. But `SEED_CHECKS.md` is required by the Phase 5 gate (line 1641: "If Phase 0 or 0b ran: quality/SEED_CHECKS.md exists as a standalone file") — yet it's NOT in the canonical table.

The contradiction: the table is declared canonical, but a Phase 5 requirement exists outside the table. Either the table is not canonical (and the claim at line 88 is false), or the Phase 5 requirement is non-canonical (and agents need not follow it).

Additionally, quality_gate.sh does NOT check for SEED_CHECKS.md, violating the table's own statement that "the gate checks for it."

**Categorization:** Real spec bug — SKILL.md contains a self-contradiction. The artifact contract table claims to be canonical but omits an artifact that Phase 5 requires. This is a spec-primary fix (add SEED_CHECKS.md to the table).

**Verdict: CONFIRMED — BUG-L22** (net-new, LOW severity, spec-primary fix)
