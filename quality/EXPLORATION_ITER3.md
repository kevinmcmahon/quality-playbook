## Unfiltered findings

1. **Docs-warning contract is inverted in phase mode.**
   - Trace: `docs_present(repo_dir)` at `bin/run_playbook.py:560-562` feeds `run_one_phased()` at `bin/run_playbook.py:645-649`.
   - Current behavior: when `docs_gathered/` is missing or empty, phase mode prints `SKIP: ... docs_gathered/ is missing or empty` and returns `1`.
   - Spec contradiction: `SKILL.md` says Phase 1 must print a warning and proceed with code-only analysis, not skip the run.
   - This is a direct runner/spec mismatch, not a question of iteration preference.

2. **The same docs-warning inversion exists in single-pass mode.**
   - Trace: `docs_present()` at `bin/run_playbook.py:560-562` feeds `run_one_singlepass()` at `bin/run_playbook.py:671-675`.
   - Current behavior: full-run and single-pass invocation bail out before `archive_previous_run`, prompt generation, or any child runner execution.
   - Result: the documented code-only fallback path is unreachable in the normal single-pass path as well as the phased path.

3. **Prompt generation hardcodes one install layout instead of the documented fallback list.**
   - Trace: `phase1_prompt()` at `bin/run_playbook.py:258-282`, `phase2_prompt()` at `285-309`, `phase3_prompt()` at `312-334`, `phase4_prompt()` at `337-357`, `phase5_prompt()` at `360-382`, `phase6_prompt()` at `385-407`.
   - Every phase prompt instructs the child model to read `.github/skills/SKILL.md` and `.github/skills/references/...`.
   - The skill itself documents four legal locations (`SKILL.md`, `.claude/skills/quality-playbook/...`, `.github/skills/...`, `.github/skills/quality-playbook/...`).
   - This is a cross-file portability bug: source-checkout and Claude-layout runs are explicitly documented but not producible by the runner-generated prompts.

4. **The hardcoded-path drift also affects single-pass and iteration entrypoints.**
   - Trace: `single_pass_prompt()` at `bin/run_playbook.py:421-423` and `iteration_prompt()` at `426-427`.
   - Current text: `Read the quality playbook skill at .github/skills/SKILL.md ...`
   - Practical consequence: even if the runner itself could locate the skill elsewhere, the spawned child prompt instructs the model to use the wrong path.
   - This extends the portability bug beyond phase mode into the default runner flows.

5. **`run_prompt()` child exit codes are ignored in per-phase execution.**
   - Trace: `run_prompt()` returns an `int` at `bin/run_playbook.py:516-551`; `run_one_phase()` calls it at `609` and discards the result.
   - Current behavior: after a child exits non-zero, `run_one_phase()` still logs `Phase N complete` summaries and returns `True`.
   - This is a cross-function control-flow defect: the function with the failure signal exists, but its caller refuses to consume it.

6. **`run_prompt()` child exit codes are also ignored in single-pass execution.**
   - Trace: `run_prompt()` at `516-551` -> `run_one_singlepass()` at `696-705` -> `execute_run()` at `938-946`.
   - Current behavior: single-pass mode logs `Playbook complete` and returns `0` after `run_prompt()` even when the child failed.
   - This can falsely mark a failed full run as successful, and it contaminates any automation that keys off the process exit code.

7. **The false-success bug compounds with the existing suggestion logic.**
   - Trace: `run_one_singlepass()` returns `0` on child failure (`696-705`) -> `run_one()` forwards that `0` (`708-711`) -> `execute_run()` increments no failure counter for that repo (`938-946`).
   - Existing BUG-008 already covers the suggestion being failure-unaware *after* failures are counted.
   - This new defect is upstream of BUG-008: the runner can fail to count the failure at all.
   - Together they create a two-layer false-success path: the child fails, the per-repo run reports success, and the top-level command may suggest continuing.

8. **The language heuristic is contaminated by embedded fixture repos.**
   - Trace: `detect_project_language()` at `.github/skills/quality_gate/quality_gate.py:209-250` walks `repo_dir.rglob("*")`, excludes only `vendor`, `node_modules`, `.git`, and `quality`, then `check_test_file_extension()` at `795-839` trusts the result.
   - This repo contains benchmark fixtures under `repos/`, including Go code. Because `repos/` is not excluded, the gate sees `.go` files and classifies the project as Go before it reasons about the Python runner/gate code at repo root.
   - The current gate failure log already demonstrates the symptom in a live artifact.

9. **The fixture-contamination bug is a self-audit blind spot, not just a test artifact.**
   - Trace: `quality/results/quality-gate.log` reports the mismatch; the repository's existing gate tests and `quality/test_functional.py` do not cover the `repos/` nesting pattern.
   - This matters because the bootstrap repository is exactly the environment where the playbook claims to validate itself.
   - A validator that fails on the project's own supported layout is a functional defect, not a cosmetic warning.

10. **Demoted candidate: recheck-sidecar validation still appears thinner than the skill schema.**
    - Suspect surface: `.github/skills/quality_gate/quality_gate.py` recheck-sidecar checks compared with the Recheck schema in `SKILL.md`.
    - Why not promoted: the strongest symptom overlaps with BUG-010 (missing run-metadata enforcement) and BUG-015 (grammar looseness) rather than clearly establishing a new failure class in this run.
    - Keep it visible for parity/adversarial follow-up, but do not mint a new bug ID in this iteration.
