## Finding 5: quality_gate.sh does not validate date realism in tdd-results.json (month/day bounds) [unfiltered]

**File:line:** quality_gate.sh:257-273
**Bug hypothesis:** The date validation at lines 257-273 checks three things: (1) format matches YYYY-MM-DD regex, (2) not a placeholder like "YYYY-MM-DD" or "0000-00-00", (3) not in the future. But it does not check that the month is 01-12 or the day is 01-31. A date like "2026-13-45" passes the regex `^[0-9]{4}-[0-9]{2}-[0-9]{2}$`, passes the placeholder check, and may or may not pass the future-date check depending on the current date. On some systems, the `[[ "$tdd_date" > "$today" ]]` string comparison produces inconsistent results for invalid months/days. For example, "2025-99-99" is less than "2026-04-15" by string comparison and would pass all three checks despite being an impossible date.

**Code path trace:** quality_gate.sh:257 checks format. quality_gate.sh:259-260 checks for placeholder values. quality_gate.sh:264-265 uses bash string comparison (`[[ "$tdd_date" > "$today" ]]`) which does lexicographic comparison. "2025-99-99" < "2026-04-15" lexicographically, so it passes. The `pass "tdd-results.json date '2025-99-99' is valid"` message fires.

## Finding 8: quality_gate.sh bug_count stays at correct_headings on line 183 even when wrong-format headings exist [unfiltered]

**File:line:** quality_gate.sh:183, 200
**Bug hypothesis:** At line 183, `bug_count=$correct_headings`. Then the if-block at line 185 checks if ALL headings are correct. If some are correct and some are wrong, the else-branch fires (line 187). Lines 188-190 report the wrong headings as FAILs. But line 200 (`bug_count=$((correct_headings + wrong_headings + bold_headings + bullet_headings))`) only executes when the final else clause at line 199 fires -- which requires `correct_headings > 0 OR wrong_headings > 0` (i.e., at least one heading of any format). If there are correct_headings=5 and wrong_headings=2, the outer if at line 185 fails (wrong_headings != 0), and we enter the else branch. Lines 188-190 report the wrong headings. Then at line 191 the condition `correct_headings == 0 && wrong_headings == 0` is false (both are non-zero), so we fall to line 199-200 and set `bug_count=$((5+2+0+0))=7`. But wait -- line 183 already set `bug_count=5`. So bug_count gets OVERWRITTEN to 7 at line 200. This means downstream checks (patches, writeups, TDD logs) will expect 7 bugs when there are only 5 correctly-formatted ones. Actually, the intent seems to be to count ALL bugs regardless of heading format. This is arguably correct behavior since wrong-format headings are still bugs. But the logic is subtle and could silently inflate bug_count if, say, the same bug appears in both correct and wrong-format headings.

Wait -- re-reading more carefully, this logic seems intentionally designed to count all bugs regardless of heading format. The flow: if all headings are correct, bug_count = correct_headings (line 183 remains). If some are wrong, bug_count gets updated at line 200 to the total. ## Finding 10: quality_gate.sh date comparison uses string ordering, which breaks for some date edge cases [unfiltered]

**File:line:** quality_gate.sh:264-265
**Bug hypothesis:** The date comparison `[[ "$tdd_date" > "$today" ]]` uses bash string comparison (lexicographic order). For ISO 8601 dates, this normally works because YYYY-MM-DD sorts correctly when all fields are zero-padded. However, this comparison has a subtle edge case: if the system's `date +%Y-%m-%d` returns a non-standard format (some minimal Docker containers or BusyBox environments return dates without zero-padding), the comparison becomes unreliable. For example, if `today` is "2026-4-15" (no zero-padding on month), then "2026-04-16" > "2026-4-15" is true in string comparison (because '0' < '4' in ASCII), which would incorrectly flag a valid date as being in the future. This is an edge case, but the script runs on diverse systems.

This is LOW severity since `date +%Y-%m-%d` on POSIX systems always zero-pads. Demoting to observation.

## Domain-Knowledge Questions

**API surface inconsistencies between similar methods:**
- quality_gate.sh `fail()` vs `warn()` vs `info()` vs `pass()` have inconsistent semantic models: `fail()` and `warn()` increment counters, `pass()` and `info()` do not. But the exit code only checks FAIL count, making WARN semantically equivalent to INFO in terms of gate outcome. This design choice makes it easy to introduce BUG-001-class bugs where a spec-required check is accidentally downgraded.

**Ad-hoc string parsing of structured formats:**
- quality_gate.sh JSON helpers (lines 75-91) use grep to parse JSON. This is documented as a limitation (REQ-014) but the specific failure mode is concrete: a bug description containing a JSON field name (e.g., a description that says 'the "verdict" field was wrong') would inflate json_key_count().

**Inputs a domain expert would try:**
- Running quality_gate.sh on a directory containing a `quality/BUGS.md` with `#### BUG-001` headings (four hashes). The regex at line 173 matches `^### BUG-` but not `^#### BUG-`. The regex at line 175 for wrong_headings only checks `^## BUG-` (two hashes). A `#### BUG-001` heading would be silently invisible to all heading checks.

**Silently wrong metadata/configuration:**
- The `--general` strictness mode is only used in three places: the triage executable evidence check (line 157), the UC count check (line 417), and the integration-results.json check (line 383). All other checks are the same in both modes. But SKILL.md line 18 says "Strictness modes: --benchmark (default) and --general provide different enforcement levels, but spec-required artifacts should be FAIL in both." The actual behavioral difference is minimal -- only 3 of ~40 checks vary by strictness mode.
