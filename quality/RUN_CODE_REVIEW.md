# Code Review Protocol — Phase 3

This document specifies the three-pass code review Phase 3 must execute against the QPB codebase. The review's output is (a) `quality/BUGS.md` listing confirmed defects, (b) `quality/test_regression.py` with one failing test per confirmed bug (TDD red phase), and (c) one `quality/writeups/BUG-NNN.md` per confirmed bug. The Phase 3 agent runs in a fresh context — this file plus `REQUIREMENTS.md`, `CONTRACTS.md`, `QUALITY.md`, `EXPLORATION.md`, and `COVERAGE_MATRIX.md` is the entire hand-off.

## Review passes

The review consists of three passes. Each pass has a different analytical stance. A candidate bug (CB-NNN in EXPLORATION.md) is *confirmed* only when at least one pass produces a reproducible failure path (ideally a failing test in `test_regression.py`). Unconfirmed candidates graduate to **deferred** or **disputed** with a reason.

### Pass 1 — Structural review

**Goal.** Find defects from the code's shape alone, without any reference to documented intent.

**Technique.** For each module in scope:
- Read the module end to end.
- Note every branch that can be taken.
- Note every error path that is swallowed, rewritten, or ignored.
- Note every dead-code or unreachable branch.
- Note every place the code violates its own internal patterns (e.g., one function uses regex and a sibling uses substring for the same kind of parse).
- Note every untested or effectively-untested path.

**Expected output per module.** A list of observations, each tagged with a severity guess and a link to a location. Observations are candidates — they become bugs only after Pass 2 confirms them.

**Structural observation template.**
```
## Structural observation
- **Location:** <file>:<lines>
- **Observation:** <one-sentence description of the shape issue>
- **Suspected class:** <parser divergence | silently swallowed error | dead branch | non-atomic write | …>
- **Proof path:** <the minimal input or sequence that would expose it>
```

The 12 candidate bugs in EXPLORATION.md all have Pass-1 observations already. Pass 1 re-confirms them independently (does the code *still* look this way?) and looks for anything EXPLORATION.md missed.

**Stop condition.** Every module read, at least three structural observations per module with non-trivial source. No fewer than 20 total structural observations (matches EXPLORATION.md density). If fewer are produced, the pass is incomplete — iterate.

### Pass 2 — Requirement verification

**Goal.** Check every requirement (REQ-001 through REQ-017) against the code, using the file:line citations in REQUIREMENTS.md and CONTRACTS.md.

**Technique.** For each REQ-NNN:
- Open the files cited in the `References` field.
- Read the specific functions/lines.
- For each `Conditions of satisfaction` clause, ask: does the current code satisfy this? If not, is the gap already a candidate bug (cite CB-NNN)? If it is a new gap, add it as a new candidate.
- Where the requirement names a specific test (`test_regression.py::test_reg_*` in COVERAGE_MATRIX.md), write the test now — as a *failing* test. The test name must match the matrix.

**Expected output.** For each requirement: a verdict line (`satisfied`, `partially satisfied`, `not satisfied`) with specific citations. For `not satisfied`, a new or cross-referenced candidate bug.

**Requirement verification template.**
```
## REQ-NNN verification
- **Requirement summary:** <one-liner>
- **Code location checked:** <file>:<lines>
- **Conditions of satisfaction checked:** <list>
- **Verdict:** <satisfied | partially satisfied | not satisfied>
- **Evidence:** <the specific observation that grounds the verdict>
- **Related candidate bug:** <CB-N or "new: CB-N">
```

**Stop condition.** All 17 requirements have a verdict. At least one failing regression test exists in `test_regression.py` for every `not satisfied` verdict, per the names in COVERAGE_MATRIX.md. Tests use `@unittest.expectedFailure` or `self.skipTest` with an explicit reason; no test may silently skip.

### Pass 3 — Cross-requirement consistency

**Goal.** Find defects that only surface when two or more requirements interact.

**Technique.** The candidate bugs are not independent. Look for these interaction patterns:

- **Parser divergence cascading into benchmark invariance failure.** REQ-001 (single version helper) interacts with REQ-017 (shared closed sets) and REQ-005 (diagnostic on fallback miss). If the parser has three implementations and the diagnostic doesn't tell you which one failed, fixing one alone doesn't help.
- **Contract drift across modules.** REQ-002 (install locations) interacts with REQ-013 (gate parses EXPLORATION structure) and REQ-017. A single-source constant that the gate uses but the runner hard-codes separately is worse than two hard-codes (because one *looks* unified).
- **Gate incompleteness masquerading as test passing.** REQ-008 (run-metadata gate) interacts with REQ-013 (EXPLORATION structure gate). The gate produces "0 FAIL" reports on runs missing both — the green verdict lies.
- **Cleanup/archive protection cascade.** REQ-006 (AGENTS.md protection) interacts with REQ-009 (atomic archive). A half-archived run leaves `AGENTS.md` orphaned in an unsupported place.

**Expected output.** For each interaction pattern, a finding that names the requirements involved, the code surface where the interaction lives, and the empirical symptom. An interaction finding is a candidate bug only if the code, *as written*, exhibits the symptom — not merely "could" exhibit it.

**Cross-requirement finding template.**
```
## Cross-requirement finding
- **Requirements involved:** <REQ-A, REQ-B, …>
- **Code surface:** <files and functions>
- **Interaction pattern:** <one of the named patterns above, or "new: <name>">
- **Empirical symptom:** <the observable behavior that would result>
- **Candidate bug:** <CB-N, or "new: CB-N">
```

**Stop condition.** At least one cross-requirement finding exists for each of the four interaction patterns listed above, unless the agent can affirmatively show (with code citations) that the pattern does not apply here. "Affirmative show" means naming the specific line that isolates the two requirements from each other.

## Regression test conventions

Per `references/review_protocols.md` and the test framework notes in EXPLORATION.md.

**File.** `quality/test_regression.py`. Single file — do not split. Use stdlib `unittest`.

**Imports.** `from bin import run_playbook`, `from bin import benchmark_lib as lib`. For quality_gate, use `importlib.util.spec_from_file_location` as in `quality/test_functional.py::_load_quality_gate`.

**Test naming.** `test_reg_<bug-id>_<short-description>`, lowercase snake_case. Example: `test_reg_cb1_version_parser_divergence`. Names must match COVERAGE_MATRIX.md.

**xfail pattern.** Each test is decorated with `@unittest.expectedFailure` and must actually fail against current code. A test that passes under `@unittest.expectedFailure` is an UNEXPECTED PASS — which is a test failure. This is the red-phase expected-fail signal.

**Docstring.** First line cites the bug and the requirement. Second line names the file:line under test.

**Example skeleton.**
```python
import unittest
from bin import benchmark_lib as lib

class RegressionTests(unittest.TestCase):
    @unittest.expectedFailure
    def test_reg_cb1_version_parser_divergence(self):
        """CB-1 / REQ-001: skill_version must accept **Version:** bold form.

        Source: benchmark_lib.py:90-116.
        """
        import tempfile, pathlib
        from unittest import mock
        with tempfile.TemporaryDirectory() as tmp:
            path = pathlib.Path(tmp) / "SKILL.md"
            path.write_text("**Version:** 1.4.5\n", encoding="utf-8")
            with mock.patch.object(lib, "QPB_DIR", pathlib.Path(tmp)):
                self.assertEqual(lib.skill_version(), "1.4.5")

if __name__ == "__main__":
    unittest.main()
```

## Ordered review plan

Phase 3 must follow this order. It is designed so that each step produces output the next step consumes.

1. **Read context.** REQUIREMENTS.md, CONTRACTS.md, COVERAGE_MATRIX.md, QUALITY.md, relevant excerpts of EXPLORATION.md. Reset mental model to "blank slate, read what's on disk."
2. **Pass 1 — Structural.** Module-by-module: `bin/benchmark_lib.py`, `bin/run_playbook.py`, `.github/skills/quality_gate/quality_gate.py`, reference files, `agents/`. Accumulate observations.
3. **Pass 2 — Requirement verification.** Walk REQ-001 through REQ-017 in order. For each `not satisfied` verdict, write a failing regression test in `test_regression.py`. At the end of this pass, `test_regression.py` contains every red-phase test COVERAGE_MATRIX.md named.
4. **Pass 3 — Cross-requirement.** Check the four interaction patterns. Each finding either strengthens an existing candidate or becomes a new one.
5. **Confirm candidate bugs.** For every candidate bug CB-1 through CB-12 (plus any new ones), write a verdict: *confirmed* (goes in BUGS.md), *disputed* (goes in `BUGS.md` §Disputed with a citation), or *deferred* (goes in BUGS.md §Deferred with a reason).
6. **Write BUGS.md.** One row per confirmed bug with file:line, severity, verdict, test reference. Zero-bug case uses the exact sentinel: `## No confirmed bugs — detailed rationale below.`
7. **Write per-bug writeups.** One `quality/writeups/BUG-NNN.md` per confirmed bug.
8. **Run test_regression.py.** Every test must FAIL (be xfail). Run once:
   `python3 -m unittest quality.test_regression -v` — record every test name and its xfail status. Save the transcript to `quality/results/tdd-red-phase.log`.
9. **Prepare patches (do NOT apply).** For every confirmed bug where the fix is a patch: write the patch file to `quality/patches/BUG-NNN-fix.patch`. Do not edit source directly — `cleanup_repo` will revert. The patch is part of the bug writeup.
10. **Emit sidecar.** Write `quality/results/tdd-results.json` with `schema_version: "1.1"` and one entry per bug. Verdict is `red failed` (patch not yet applied) or `confirmed open` (no patch this cycle). The Phase 5 reconciliation updates these to `TDD verified` when patches are applied and re-tested.

## Council of Three cross-reference

This code review pairs with Phase 4's Council of Three (see `RUN_SPEC_AUDIT.md`). Where code review finds a contract violation, spec audit decides whether the violation is a code defect or a spec ambiguity. The handoff:

- **Code review → Spec audit.** Any requirement whose citation in REQUIREMENTS.md reads `[source]` (not `[SKILL.md §...]`) and whose code behavior is ambiguous deserves a spec-audit pass. Council of Three may recommend strengthening the spec, changing the code, or both.
- **Spec audit → Code review.** Any spec ambiguity the council flags that produces a candidate bug flows back as a new CB-NNN.

## Verification before Phase 4 entry

At the end of Phase 3, the following files must exist and pass the gate:

- `quality/BUGS.md` — with the exact zero-bug sentinel if there are no confirmed bugs, or one heading per confirmed bug.
- `quality/test_regression.py` — every expected-failure test accounted for in COVERAGE_MATRIX.md.
- `quality/results/tdd-results.json` — schema_version "1.1", one entry per bug, verdict from the enum.
- `quality/writeups/BUG-NNN.md` — one per confirmed bug.
- `quality/patches/BUG-NNN-fix.patch` — one per confirmed bug with a proposed patch (optional if the bug is deferred).

All seven of these are gate inputs. A Phase 4 entry without them must FAIL at `check_phase_gate("4")`.

## Bootstrap-specific caution

This is a bootstrap self-audit (UC-02). Special rules apply:

- **No direct source edits.** `cleanup_repo` will revert them. Patches only.
- **AGENTS.md is itself a test subject.** REQ-006 makes it a Phase 3 target. Phase 3 must not *regenerate* AGENTS.md — that's Phase 2's job — but may propose patches to it via `quality/patches/BUG-NNN-fix.patch`.
- **The agent reviewing is running against code that includes `bin/run_playbook.py`.** If the review finds a bug in the runner, writing the fix directly could break the currently-executing workflow. Fixes go in patches; the next run applies and verifies them.

## Quality signal

The Phase 3 run is "successful" when:

- At least 10 of the 12 candidate bugs receive a *confirmed* or *deferred* verdict with a regression test or explicit exemption.
- The structural pass surfaces at least 3 observations that were not in EXPLORATION.md. (New observations validate the pass wasn't just confirming EXPLORATION.md.)
- Cross-requirement findings name at least 2 concrete interaction cascades with file:line evidence.
- `test_regression.py` runs to completion and every test is an expected-failure xfail — no silent skips, no unexpected passes.
- Every row in COVERAGE_MATRIX.md's regression-locus column has a matching test name in `test_regression.py`.
