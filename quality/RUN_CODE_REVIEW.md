## Bootstrap (Read First)

Before reviewing, read these files for context:
1. `quality/QUALITY.md` — Quality constitution and fitness-to-purpose scenarios
2. `quality/REQUIREMENTS.md` — 14 testable requirements derived from specification analysis
3. `SKILL.md` — The master specification (2083 lines) — this is both the product and the primary review target
4. `quality_gate.sh` — The only executable code (676 lines of Bash) — the secondary review target
5. `quality/EXPLORATION.md` — Phase 1 findings, including 6 candidate bugs and 7 quality risks

## Pass 1: Structural Review

Read quality_gate.sh and SKILL.md. Report anything that looks wrong. No requirements, no focus areas — use your own knowledge of code correctness and specification quality.

### Guardrails

- **Line numbers are mandatory.** If you cannot cite a specific line, do not include the finding.
- **Read function bodies, not just signatures.** For quality_gate.sh, read the actual check logic, not just the section headers.
- **If unsure whether something is a bug or intentional**, flag it as a QUESTION rather than a BUG.
- **Grep before claiming missing.** If you think a check is absent, search quality_gate.sh. If found under a different section name, that's a location issue, not a missing feature.
- **Do NOT suggest style changes, refactors, or improvements.** Only flag things that are incorrect or could cause failures.

### Minimum Required Scrutiny Areas

1. **Input validation and boundary handling** — Check quality_gate.sh argument parsing (lines 42-57), version detection (lines 59-67), and all places where user-supplied paths are used in commands. Check SKILL.md's gate definitions for ambiguous conditions.

2. **Resource lifecycle** — Check quality_gate.sh for file handle leaks, temporary file cleanup, and the gate's own error handling if a check crashes mid-run.

3. **Concurrency and state management** — quality_gate.sh is single-threaded. Check for state corruption if the script is interrupted (e.g., does it leave partial output?). Check SKILL.md's phase state machine for deadlock conditions.

4. **Unit and encoding correctness** — Check quality_gate.sh's date validation regex (lines 256-276), version string comparison, and grep-based JSON parsing for encoding edge cases.

5. **Enumeration and whitelist completeness** — This is the most important area for this project. Compare quality_gate.sh's checked artifacts against SKILL.md's artifact contract table. For each artifact in the table, verify it has a corresponding check. For each check in quality_gate.sh, verify it traces to the table. Perform the two-list mechanical comparison.

## Pass 2: Requirement Verification

Read `quality/REQUIREMENTS.md`. For each requirement (REQ-001 through REQ-014), check whether the code satisfies it. This is a pure verification pass — your only job is "does the code satisfy this requirement?"

For each requirement, report one of:
- **SATISFIED**: The code implements this requirement. Quote the specific code.
- **VIOLATED**: The code does NOT satisfy this requirement. Explain what the code does vs. what the requirement says.
- **PARTIALLY SATISFIED**: Some conditions met, others not.
- **NOT ASSESSABLE**: Can't be checked from the files under review.

### Output

For each requirement:

#### REQ-NNN: [title]
**Status**: SATISFIED / VIOLATED / PARTIALLY SATISFIED / NOT ASSESSABLE
**Evidence**: [file:line] — [code quote or spec quote]
**Analysis**: [explanation]
[If VIOLATED] **Severity**: [impact description]

## Pass 3: Cross-Requirement Consistency

Compare pairs of requirements from `quality/REQUIREMENTS.md` that reference the same field, constant, or policy. For each pair, check whether their constraints are mutually consistent.

Key pairs to check:
- **REQ-001 + REQ-002:** Both reference the artifact contract table. Do they agree on which artifacts are required and what severity to enforce?
- **REQ-006 + REQ-002:** REQ-006 says exit code depends on FAIL count. REQ-002 says certain artifacts must use fail(). Are these consistent?
- **REQ-007 + REQ-001:** REQ-007 says test file detection must cover all patterns. REQ-001 says the three sources must agree. Does verification.md agree on test file patterns?
- **REQ-008 + REQ-001:** Both reference version stamps. Does quality_gate.sh's version stamp check match SKILL.md's stamp format definition?
