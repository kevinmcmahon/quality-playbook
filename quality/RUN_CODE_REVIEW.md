## Pass 1: Structural Review

Read the source files and report anything that looks wrong. No requirements, no focus areas — use your own knowledge of code correctness.

### Guardrails

- **Line numbers are mandatory.** If you cannot cite a specific line, do not include the finding.
- **Read function bodies, not just signatures.** Do not assume a function works correctly based on its name.
- **If unsure whether something is a bug or intentional**, flag it as a QUESTION rather than a BUG.
- **Grep before claiming missing.** If you think a feature is absent, search all files. If found in a different file, that's a location defect, not a missing feature.
- **Do NOT suggest style changes, refactors, or improvements.** Only flag things that are incorrect or could cause failures.

### Minimum Required Scrutiny Areas

1. **Input validation and boundary handling** — Check every trust boundary in quality_gate.sh: argument parsing, file path construction, grep pattern matching, glob expansion. Every unquoted variable, every unguarded glob, every `eval` call.

2. **Resource lifecycle** — Check file handle usage in quality_gate.sh: does every subshell close cleanly? Does the script handle interrupted execution (SIGINT/SIGTERM)?

3. **Concurrency and state management** — Not applicable for this project (single-threaded bash script, no concurrent execution). Document why rather than omitting.

4. **Unit and encoding correctness** — Check version number parsing in quality_gate.sh (sed/grep patterns). Check that date validation handles timezone edge cases. Check that UC-count parsing correctly handles all edge cases.

5. **Enumeration and whitelist completeness** — Perform the two-list check on:
   - Artifact contract table "Required: Yes" entries vs. quality_gate.sh file existence checks
   - Verdict enum values in SKILL.md vs. quality_gate.sh verdict validation regex
   - Summary keys in normative text vs. quality_gate.sh summary key checks

## Pass 2: Requirement Verification

Read `quality/REQUIREMENTS.md`. For each requirement (REQ-001 through REQ-012), check whether the code satisfies it. This is a pure verification pass — your only job is "does the code satisfy this requirement?"

Do NOT also do a general code review. Do NOT look for other bugs. Just check each requirement.

For each requirement, report one of:
- **SATISFIED**: The code implements this requirement. Quote the specific code.
- **VIOLATED**: The code does NOT satisfy this requirement. Explain what the code does vs. what the requirement says.
- **PARTIALLY SATISFIED**: Some aspects implemented, others missing.
- **NOT ASSESSABLE**: Cannot be checked from the files under review.

### Minimum evidence rule

Cite at least one code location (file:line or file:function) **per requirement**. Blanket satisfaction claims without per-requirement citations do not satisfy Pass 2.

### Output

For each requirement:

#### REQ-NNN: [requirement title]
**Status**: SATISFIED / VIOLATED / PARTIALLY SATISFIED / NOT ASSESSABLE
**Evidence**: [file:line] — [code quote or description]
**Analysis**: [explanation]
[If VIOLATED] **Severity**: [impact description]

## Pass 3: Cross-Requirement Consistency

Compare pairs of requirements from `quality/REQUIREMENTS.md` that reference the same field, constant, range, or policy.

### Key pairs to check:

1. **REQ-001 + REQ-004**: Both reference quality_gate.sh validation. Do they agree on what the gate should check?
2. **REQ-001 + REQ-012**: Both reference tdd-results.json per-bug fields. Do the "required" field lists agree?
3. **REQ-003 + REQ-007 + REQ-011**: All reference installation paths. Do they agree on what paths are supported?
4. **REQ-005 + REQ-008**: Both reference the artifact contract table. Do they agree on what "Required" means?
5. **REQ-002 + REQ-010**: Both reference phase numbering. Does the phase boundary for requirements derivation agree?
