## Open Exploration Findings

### Finding 1: [gap] quality_gate.sh crashes on macOS bash 3.2 when invoked with no arguments

**File:** quality_gate.sh:32, 37, 639
**Bug hypothesis:** Line 32 sets `set -uo pipefail`. Line 37 initializes `REPO_DIRS=()`. When the script is invoked with no arguments and no `--all` flag, `CHECK_ALL` is false, `REPO_DIRS` is empty (length 0). Line 634 checks `${#REPO_DIRS[@]} -eq 1` which is false (0 != 1), so execution falls to the `else` branch at line 637. Line 639 iterates `for name in "${REPO_DIRS[@]}"`. On bash 3.2 (macOS default), `"${arr[@]}"` on an empty array with `set -u` triggers "unbound variable" error. The script crashes before reaching the usage message at line 654.

**Trace:** `main` -> arg parsing loop (lines 44-57) adds nothing to REPO_DIRS -> line 634 condition fails -> line 639 `"${REPO_DIRS[@]}"` -> bash 3.2 "unbound variable" crash.

**Reproduction confirmed:** `bash -c 'set -uo pipefail; arr=(); for x in "${arr[@]}"; do echo "$x"; done'` produces "bash: arr[@]: unbound variable" on bash 3.2.57.

### Finding 2: [gap] TOOLKIT.md Copilot installation path not in SKILL.md reference resolution fallback

**File:** ai_context/TOOLKIT.md:24-26 vs SKILL.md:46-53
**Bug hypothesis:** TOOLKIT.md instructs Copilot users to install references at `.github/skills/references/`. But SKILL.md's reference resolution fallback chain (lines 48-51) tries:
1. `references/` (relative to SKILL.md)
2. `.claude/skills/quality-playbook/references/`
3. `references/` (repo root)
4. `.github/skills/quality-playbook/references/`

Path `.github/skills/references/` is NOT explicitly listed. Path 1 ("relative to SKILL.md") MIGHT resolve there if the agent uses SKILL.md's parent directory as the base, but this is implementation-dependent. Path 3 (`references/` at repo root) resolves to `<repo>/references/`, not `.github/skills/references/`. A user following TOOLKIT.md's Copilot installation instructions gets references at a location SKILL.md doesn't enumerate, risking silent failure where the agent proceeds without reference files.

**Trace:** TOOLKIT.md lines 22-26 (installation) -> SKILL.md line 53 (fallback walkthrough) -> no match for `.github/skills/references/` -> agent silently proceeds without references.

### Finding 3: [gap] Agent definition files disagree on SKILL.md search order

**File:** agents/quality-playbook.agent.md:17-19 vs agents/quality-playbook-claude.agent.md:21-23
**Bug hypothesis:** The generic orchestrator agent (quality-playbook.agent.md lines 17-19) searches for SKILL.md in the order: (1) `.claude/skills/quality-playbook/SKILL.md`, (2) `.github/skills/SKILL.md`, (3) `.github/skills/quality-playbook/SKILL.md`. The Claude orchestrator agent (quality-playbook-claude.agent.md lines 21-23) searches: (1) `.claude/skills/quality-playbook/SKILL.md`, (2) `.github/skills/quality-playbook/SKILL.md`, (3) `.github/skills/SKILL.md`. Paths 2 and 3 are swapped between the two agents. If both flat and nested Copilot installations exist (with potentially different versions), the two agents would find different SKILL.md files.

### Finding 4: [gap] Agent installation instructions use path not in SKILL.md reference resolution

**File:** agents/quality-playbook.agent.md:29-31 vs SKILL.md:46-53
**Bug hypothesis:** The generic agent's installation instructions (line 29-31) tell users to put Copilot references at `.github/skills/references/`. This is the same path as TOOLKIT.md Finding 2 above, confirming the inconsistency exists in two different user-facing documents. Both TOOLKIT.md and the agent definition instruct users toward a Copilot installation layout that SKILL.md's fallback chain doesn't explicitly enumerate.

### Finding 5: [gap] quality_gate.sh does not validate iteration-mode artifacts

**File:** quality_gate.sh (entire file) vs references/iteration.md:48-53
**Bug hypothesis:** references/iteration.md defines an "Iteration mode completion gate" (lines 48-53) requiring: `quality/ITERATION_PLAN.md`, `quality/EXPLORATION_ITER{N}.md`, and `quality/EXPLORATION_MERGED.md`. These artifacts are mandatory for iteration runs. However, quality_gate.sh does not check for any of these artifacts. A grep for "ITERATION_PLAN", "EXPLORATION_ITER", and "EXPLORATION_MERGED" in quality_gate.sh returns zero matches. Similarly, references/verification.md (45 benchmarks) does not mention any iteration artifacts. An iteration run that omits these required artifacts would still pass quality_gate.sh.

**Trace:** references/iteration.md line 48-53 (completion gate) -> quality_gate.sh (no corresponding check) -> gate exits 0 despite missing iteration artifacts.

### Finding 6: [gap] quality_gate.sh does not validate EXPLORATION.md existence despite it being the Phase 1 mandatory output

**File:** quality_gate.sh (entire file) vs SKILL.md:258-259, 893-902
**Bug hypothesis:** SKILL.md line 258 says "Write your Phase 1 exploration findings to `quality/EXPLORATION.md` before proceeding. This file is mandatory in all modes." The Phase 2 entry gate (lines 893-902) verifies its existence and structure. But quality_gate.sh never checks for EXPLORATION.md at all. A grep for "EXPLORATION" in quality_gate.sh returns zero matches. While EXPLORATION.md is not in the artifact contract table (lines 86-115), it IS described as "mandatory in all modes" in SKILL.md and is a critical handoff artifact. The baseline already noted this in the cross-implementation consistency analysis but did not trace its impact: without gate enforcement, a run could complete all phases and pass the gate despite having no EXPLORATION.md on disk.

### Finding 7: [gap] Duplicate tdd-results.json version check in quality_gate.sh

**File:** quality_gate.sh:594-598 vs 617-624
**Bug hypothesis:** The [Version Stamps] section (lines 594-598) checks `tdd-results.json` `skill_version` against `skill_version` (per-repo detected). The [Cross-Run Contamination] section (lines 617-624) checks the same `tdd-results.json` `skill_version` against the same `skill_version`. Both use the same variable and the same extraction. If the versions match, both pass. If they mismatch, both fail — producing two FAIL messages for the same problem. While not a functional bug (double-counting doesn't change the pass/fail outcome since FAIL > 0 triggers exit 1), it inflates the FAIL counter and could confuse users interpreting the gate output.

### Finding 8: [gap] quality_gate.sh --all with empty VERSION silently finds no repos

**File:** quality_gate.sh:60-67, 630-633
**Bug hypothesis:** When `--all` is passed without `--version` and no SKILL.md is found in any of the 6 search paths, VERSION remains empty (line 60). The glob at line 631 becomes `"${SCRIPT_DIR}/"*-""/` which matches directories ending in a bare dash (e.g., `foo-/`). In practice this matches nothing, so REPO_DIRS stays empty, and the script prints the usage message and exits 1. The failure is non-obvious: the user intended `--all` but the error message says "Usage: ..." with no indication that the problem was a missing VERSION. A warning like "WARNING: --all requires --version or a detectable SKILL.md" would be more helpful.

### Finding 9: [gap] SKILL.md "Phase 6: Results" collision exists in TWO reference contexts

**File:** references/review_protocols.md line 376 (already found), SKILL.md lines 1428 (potential context)
**Bug hypothesis:** The baseline found the Phase 6 numbering error in review_protocols.md (BUG-004). But exploring the integration test and TDD sections of SKILL.md reveals that line 1428 says the TDD protocol has the same three-phase UX pattern ("Phase 1: Plan", "Phase 2: Progress", results). SKILL.md line 1428 refers to "the same three-phase pattern as the integration tests" without specifying the numbering. If an agent reads review_protocols.md's "Phase 6" numbering and applies it to the TDD protocol (since SKILL.md says "same pattern"), the error propagates to a second context. This is a second-order effect of BUG-004, not a new bug, but it wasn't traced in the baseline.

### Finding 10: [gap] quality_gate.sh writeup check uses warn instead of fail for incomplete writeups

**File:** quality_gate.sh:558-562
**Bug hypothesis:** When writeup_count is between 0 and bug_count (i.e., some but not all bugs have writeups), line 561 uses `warn` instead of `fail`. SKILL.md line 1454 says "Writeup generation for all confirmed bugs (mandatory)... A run with confirmed bugs and no writeups directory is incomplete." verification.md benchmark 30 says "Every confirmed bug... must have a writeup." The spec treats incomplete writeups as a hard requirement, but the gate only warns. A run with 3 bugs and 2 writeups would pass the gate despite violating the spec.

## Quality Risks

### Risk 1 (HIGH): [gap] Bash 3.2 portability breaks quality_gate.sh on macOS
**Function:** main body, quality_gate.sh lines 639, 663
**Specific scenario:** macOS ships bash 3.2 as `/bin/bash`. `set -u` combined with iterating over an empty array using `"${arr[@]}"` crashes with "unbound variable". This affects the no-arguments case (line 639 in the resolution else-branch). A macOS user running `./quality_gate.sh` with no args gets a confusing bash error instead of the usage message.

### Risk 2 (MEDIUM): [gap] Installation instructions lead users to unreachable reference paths
**Files:** ai_context/TOOLKIT.md:24-26, agents/quality-playbook.agent.md:29-31, SKILL.md:46-53
**Specific scenario:** A Copilot user follows TOOLKIT.md's quick start and puts references at `.github/skills/references/`. The agent reads SKILL.md, tries the four fallback paths, none match. References are silently unavailable. The agent proceeds with degraded output (per REQ-004 in the baseline). Two user-facing documents instruct users toward a path that SKILL.md doesn't enumerate.

### Risk 3 (MEDIUM): [gap] Iteration artifacts escape gate enforcement entirely
**Files:** references/iteration.md:48-53, quality_gate.sh (entire), references/verification.md (entire)
**Specific scenario:** An iteration run produces EXPLORATION_ITER2.md and EXPLORATION_MERGED.md but omits ITERATION_PLAN.md. The quality gate passes because it doesn't check for any iteration artifacts. The iteration.md completion gate (lines 48-53) is a self-check that depends on the agent following instructions — there's no mechanical enforcement.

### Risk 4 (MEDIUM): [gap] Writeup completeness is WARN, allowing partial runs to pass
**File:** quality_gate.sh:558-562
**Specific scenario:** A run confirms 5 bugs but only generates 3 writeups. Line 561 reports `warn "3 writeup(s) for 5 bug(s) — incomplete"`. The gate exits 0 because only FAIL triggers exit 1. SKILL.md line 1454 says writeups are mandatory for all confirmed bugs.

### Risk 5 (LOW): [gap] Duplicate FAIL messages for single version mismatch
**File:** quality_gate.sh:594-598 + 617-624
**Specific scenario:** A stale tdd-results.json with skill_version "v1.3.0" when SKILL.md says "v1.4.0" produces two separate FAIL messages: one from [Version Stamps] and one from [Cross-Run Contamination]. FAIL counter increments by 2 for a single underlying issue. Not a false pass, but confusing for users.

## Skeletons and Dispatch

### Installation path resolution: three implementations

Three documents define installation paths:
1. **SKILL.md** (lines 46-53): defines 4 reference resolution fallback paths
2. **TOOLKIT.md** (lines 22-26): defines installation layout for Copilot and Claude Code
3. **agents/quality-playbook.agent.md** (lines 17-19, 29-31): defines SKILL.md search order and installation instructions

These are three implementations of the "find skill files" contract. SKILL.md is authoritative (it's the product), but the other two are user-facing and must agree.

### Iteration completion gate: specification without enforcement

references/iteration.md lines 48-53 define a completion gate for iteration mode with 5 checks:
- ITERATION_PLAN.md exists
- EXPLORATION_ITER{N}.md exists with 80+ lines
- EXPLORATION_MERGED.md exists
- Merged Candidate Bugs has 2+ new candidates
- At least 1 finding covers uncovered area

None of these are checked by quality_gate.sh or verification.md. The gate is purely prose-based.
