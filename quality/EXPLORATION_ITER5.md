### DC-002 RE-INVESTIGATION: Duplicate tdd-results.json version check

**Original dismissal:** "The duplicate check inflates the FAIL counter but does not cause a false pass."

**Re-promotion criteria:** "Demonstrate that the duplicate FAIL causes a user to misinterpret the gate output."

**Fresh evidence:**
- quality_gate.sh:594-598 — [Version Stamps] section checks `tdd-results.json skill_version` vs `skill_version` from SKILL.md
- quality_gate.sh:617-624 — [Cross-Run Contamination] section checks the exact same comparison: `tdd-results.json skill_version` vs `skill_version`

When tdd-results.json has a stale skill_version (e.g., "1.3.0" when SKILL.md says "1.4.0"), both checks fire. The output shows:
```
  FAIL: tdd-results.json skill_version '1.3.0' != '1.4.0'
  ...
  FAIL: tdd-results.json skill_version '1.3.0' != SKILL.md '1.4.0' — stale artifacts from prior run?
```

The FAIL count is incremented by 2 for a single underlying issue. A user sees "2 FAIL" and thinks there are two problems to fix. They fix the skill_version and re-run, and both FAILs disappear. The user now has reduced confidence in the FAIL counter's accuracy — a gate that double-counts is a gate that lies about severity.

**Adversarial analysis:** The counter inflation is observable in the gate output. The gate summary says "Total: 2 FAIL" when only 1 fix is needed. This violates the spirit of quality_gate.sh's purpose: to give accurate, actionable feedback. The duplicate check is at lines 617-624 — it performs the EXACT same comparison as lines 594-598. It is not testing a different condition; it is literally the same comparison in a different section.

However, the key question is: does this violate a documented contract? SKILL.md says nothing about FAIL counter accuracy. quality_gate.sh's own comments don't promise a 1:1 mapping between FAILs and unique issues. This is a code quality issue (DRY violation), not a behavioral divergence from a spec.

**Determination: DEMOTED — confirmed as code quality issue, not a spec-violating bug.** The duplicate check inflates the FAIL count, which reduces user trust in gate output, but no normative document contracts against this behavior. Filing this as a bug would require a weaker evidentiary standard than even the adversarial bar allows.

### DC-005 RE-INVESTIGATION: quality_gate.sh date validation accepts impossible dates

**Original dismissal:** "The date field is generated by the agent, not user-supplied. AI agents always generate valid ISO 8601 dates."

**Re-promotion criteria:** "Show a real scenario where an agent generates an impossible date that passes the gate."

**Fresh evidence:** quality_gate.sh lines 257-273 validate the date with:
```
^[0-9]{4}-[0-9]{2}-[0-9]{2}$
```
This accepts "2025-99-99", "2025-02-31", "2025-13-01" etc.

**Adversarial analysis with lower bar:** The dismissal's logic is "AI agents always generate valid dates." This is empirically false — AI agents regularly generate placeholder-like dates, and the gate already checks for two specific placeholders ("YYYY-MM-DD" and "0000-00-00"). If agents always generated valid dates, placeholder checks would be unnecessary. The existence of placeholder checks proves the premise wrong.

HOWEVER, the specific failure mode (impossible month/day like 2025-99-99) is different from placeholder dates. Looking at the date validation's purpose in the spec: SKILL.md line 145 says `date` must be "ISO 8601 (YYYY-MM-DD), not a placeholder, not in the future." ISO 8601 requires valid month (01-12) and valid day (01-28/29/30/31 depending on month). The gate accepts non-ISO-8601 dates like "2025-13-01" while claiming to validate ISO 8601 format. The regex matches the FORMAT but not the VALIDITY of the date.

The spec says "must be... ISO 8601 (YYYY-MM-DD)." A date of 2025-13-01 is NOT valid ISO 8601 even though it matches the digit format. The gate fails to enforce the spec.

However, the practical question remains: can this actually happen? Looking at the future-date check at line 265: `[[ "$tdd_date" > "$today" ]]`. Dates like "2025-99-99" would be greater than "2025-04-15" and would be caught by the future-date check. Dates like "2025-00-01" or "2025-01-00" pass both format and future-date checks. Are those reachable? An AI agent generating a date from current system time will always generate valid dates. But if the agent constructs the date string from components or copies it from a template with modifications, invalid dates are possible.

The re-promotion criteria demands a "real scenario where an agent generates an impossible date." I cannot provide one from experience. The gate validation has a DOCUMENTED gap (it claims to check ISO 8601 but doesn't), but the gap is not practically exploitable in the normal workflow.

**Determination: DEMOTED — the gap in ISO 8601 validation is real but the re-promotion criteria cannot be satisfied. The validation is incomplete per spec but the incompleteness has no practical impact in the agent-generated workflow.**

### SA-09 RE-INVESTIGATION: JSON helper grep-based parsing may false-match

**Original dismissal:** "Design choice. False matches unlikely for sidecar JSON."

**Fresh evidence:** json_has_key() at quality_gate.sh:77 does: `grep -q "\"${key}\"" "$file"`. This matches the key name inside any quoted string, not just as a JSON key.

Consider a tdd-results.json where a bug description contains: `"requirement": "UC-03: Validate that \"verdict\" field is canonical"`. The grep for key `verdict` would match BOTH the actual `"verdict": "TDD verified"` field AND the word `verdict` inside the requirement string value.

json_key_count() at line 90 does: `grep -c "\"${key}\""`. This counts LINES containing the pattern. If a bug's requirement field contains the word "verdict" in quotes, the count is inflated. The check at line 228 compares `fcount >= bug_count` — an inflated count could cause a false PASS (counting a string match as a field presence) but not a false FAIL.

**Adversarial analysis:** The dismissal says "unlikely for sidecar JSON." But consider: the tdd-results.json per-bug `requirement` field contains free-form text describing what requirement was violated. If a requirement mentions the word "verdict", "id", or any other JSON field name in its text, the grep-based counter is wrong.

Concrete example: A bug violates a requirement about "the verdict enum." Its tdd-results.json entry has:
```json
{"id": "BUG-003", "requirement": "REQ-014: \"verdict\" field must use canonical enum values", ...}
```
json_key_count for "verdict" would count this line (because it contains `"verdict"`) giving count = bug_count + 1. The gate would report PASS (count >= bug_count) — which happens to be the correct answer but for the wrong reason.

The more dangerous case: json_key_count for a per-bug field returns count LESS than bug_count because one entry is missing the field, BUT a string match in another entry's description inflates the count to exactly bug_count. The gate reports PASS when a field is actually missing.

This is a real behavioral bug: the gate can report a false PASS for per-bug field presence when a description contains a quoted field name AND a different entry is missing that field.

**BUT:** How likely is this compound failure? It requires: (1) a bug description containing a quoted JSON field name, AND (2) a different bug entry missing that exact field. Condition (1) is plausible (bugs about JSON validation might mention field names). Condition (2) is the exact scenario the check exists to catch. The combination means the check fails precisely when it matters most — when a field IS missing but a string match masks the gap.

**Determination: RE-PROMOTED as candidate.** The grep-based JSON parsing can produce a false PASS for per-bug field validation when a description contains a quoted field name and a different entry is missing that field. This is not a design choice — it is a behavioral bug where the validation mechanism fails in its primary purpose (catching missing fields).
