## Parallel groups enumerated

### Group 1 — Functional-test discovery helpers

1. `bin/benchmark_lib.py:19-26` defines `FUNCTIONAL_TEST_PATTERNS`.
2. `bin/benchmark_lib.py:164-165` applies that tuple in `find_functional_test()`.
3. `bin/benchmark_lib.py:291` uses `find_functional_test()` when printing the artifact summary.

**Observed contract:** the helper library treats the functional-test filename matrix as:

- `test_functional.*`
- `functional.test.*`
- `functional_test.*`
- `FunctionalTest.*`
- `FunctionalSpec.*`
- `test_functional_test.*`

### Group 2 — File-existence gate

1. `.github/skills/quality_gate/quality_gate.py:299-303` decides whether the functional test artifact exists.
2. That path hardcodes only:
   - `test_functional.*`
   - `FunctionalSpec.*`
   - `FunctionalTest.*`
   - `functional.test.*`
3. The gate message also hardcodes that narrower list.

### Group 3 — Extension-validation gate

1. `.github/skills/quality_gate/quality_gate.py:795-839` performs the language/extension parity check.
2. It locates the functional test file with `first_file_matching(q, ["test_functional.*"])`.
3. It locates the regression file with `first_file_matching(q, ["test_regression.*"])`.
4. If no `test_functional.*` file exists, it warns `No test_functional.* found` and returns before checking any alternative name.

### Group 5 — Regression-name follow-up

   - `test_regression.*`
   - `regression_test.*`
   - `test_regression_test.*`
   - `RegressionTest.*`

3. `.github/skills/quality_gate/quality_gate.py:894-901` only recognizes `test_regression.*` in the patch gate.

## Pairwise comparisons

### Comparison 1 — Helper library vs file-existence gate

- `benchmark_lib.FUNCTIONAL_TEST_PATTERNS` includes `functional_test.*` and `test_functional_test.*`.
- `quality_gate.check_file_existence()` omits both.
- This is a direct closed-set drift between two paths that both answer the same question: "does a functional test file exist?"

### Comparison 2 — Helper library vs spec naming matrix

- The helper library's `functional_test.*` entry matches `functional_tests.md:3` (`functional_test.go` for Go).
- The spec and helper agree here.
- Any gate path that omits `functional_test.*` is therefore contradicting both the helper and the docs.

### Comparison 3 — File-existence gate vs extension-validation gate

- The existence gate accepts `FunctionalSpec.*`, `FunctionalTest.*`, and `functional.test.*`.
- The extension gate refuses to even inspect those names because it only searches for `test_functional.*`.
- A Java/Scala/TypeScript repo can therefore PASS the existence gate and then get only a warning from the extension gate instead of a real validation result.

### Comparison 4 — Spec naming matrix vs extension-validation gate

- `SKILL.md:72` says Java uses `FunctionalTest.java`.
- `verification.md:49` says Java verification runs `mvn test -Dtest=FunctionalTest`.
- `check_test_file_extension()` still warns `No test_functional.* found` for that exact valid filename.

### Comparison 5 — TypeScript naming vs extension-validation gate

- `SKILL.md:72` names `functional.test.ts`.
- `verification.md:50` names `functional.test.ts`.
- `check_test_file_extension()` never looks for `functional.test.*`.

### Comparison 6 — Scala naming vs extension-validation gate

- `SKILL.md:72` names `FunctionalSpec.scala`.
- `verification.md:48` names `FunctionalSpec`.
- `check_test_file_extension()` never looks for `FunctionalSpec.*`.

### Comparison 7 — Summary path vs gate path

- `benchmark_lib.print_summary()` uses `find_functional_test()` (`benchmark_lib.py:291`) and would print `FUNC=Y` for a repo that only has `functional_test.go`.
- `quality_gate.check_file_existence()` would still FAIL that same repo as missing a functional test file.
- The runner summary and the gate are therefore capable of contradicting each other on the same artifacts.

### Comparison 8 — Regression-name follow-up

- `benchmark_lib` treats `RegressionTest.*` and `regression_test.*` as regression files.
- `check_test_file_extension()` and `check_patches()` do not.
- This is real parity drift, but the primary spec is clearer about functional-test naming than regression-test naming, so this candidate needs tighter spec grounding before promotion.

## Concrete discrepancy findings

### Finding P-1 — Functional-test existence matrix omits valid Go naming

- **Code paths:** `benchmark_lib.py:19-26` vs `.github/skills/quality_gate/quality_gate.py:299-303`
- **Why it matters:** a valid Go artifact named `quality/functional_test.go` is discoverable by the helper library and documented in `functional_tests.md`, but the gate still FAILs the repo as missing its functional test file.
- **Disposition:** promoted to BUG-024 candidate.

### Finding P-2 — Extension validation only works for Python-style prefixes

- **Code paths:** `.github/skills/quality_gate/quality_gate.py:795-839` vs `SKILL.md:72` and `functional_tests.md:3`
- **Why it matters:** valid Java/Scala/TypeScript functional-test filenames never reach extension validation; the gate warns and returns instead.
- **Disposition:** promoted to BUG-025 candidate.

### Finding P-3 — Regression-name parity is still thinner than helper behavior

- **Code paths:** `benchmark_lib.py:28-33` vs `.github/skills/quality_gate/quality_gate.py:798-799`, `894-901`
- **Why it matters:** the helper library and the gate disagree on which regression filenames count.
- **Disposition:** demoted pending stronger spec evidence; kept for the adversarial pass.
