# Quality Constitution — Quality Playbook

## Purpose

This constitution governs how quality is judged for the Quality Playbook (QPB) itself. The product is a specification — `SKILL.md` — plus a Python harness (`bin/run_playbook.py`, `bin/benchmark_lib.py`) and a post-run validator (`.github/skills/quality_gate/quality_gate.py`). The specification is the thing customers install; the harness is how the maintainer benchmarks it; the gate is how anyone can tell whether a run actually conformed. Quality here means: the words in `SKILL.md` match the closed sets and dispatch tables in the Python, and the gate mechanically detects when they do not. Anything less is coverage theater.

The principles below orient every quality artifact in `quality/`. They are named after the three thinkers who defined 20th-century quality engineering — Deming, Juran, Crosby — because QPB's novel contribution is the same contribution they made: make quality mechanical rather than aspirational.

**Deming — the system produces its outcomes.** QPB is a *system*: model plus prompt plus runner plus gate. When a run produces thin artifacts, the right question is which element of the system failed, not which session forgot. Every requirement in `REQUIREMENTS.md` identifies a system element — a parser, a gate, a closed set, a protected-prefix rule — whose behavior can be verified without rerunning an AI session. Fitness-to-purpose scenarios below are written as system failures, not user errors.

**Juran — the critical few.** The confirmed bugs and merged iteration findings are not a wishlist. They are the handful of system defects whose recurrence produces most of the observed quality loss across QPB runs: version-parser divergence (BUG-001), line-count drift (BUG-003), missing install path coverage (BUG-002, BUG-021), code-only-run blocking (BUG-020), false-success propagation (BUG-022), fixture-poisoned gate classification (BUG-023), gate-side filename-matrix drift for language-native functional tests (BUG-024, BUG-025), and helper-side permissiveness that certifies undocumented artifact names (BUG-026, BUG-027). Other defects exist; these are the ones a shippable release cannot carry.

**Crosby — conformance to requirements.** A pass/fail judgment on QPB is not an opinion; it is a comparison to a written contract. The contract has two halves: the written spec (SKILL.md prose, the sidecar JSON templates, the artifact contract) and the extracted closed sets (install paths, protected prefixes, verdict enum, iteration strategies, phase identifiers). Any mismatch between halves is a defect. `quality_gate.py` is the comparator; when the gate misses a mismatch, the gate is the defect.

## Coverage Targets

Regression tests target CB-3, CB-4, CB-6, CB-7, CB-8, CB-9, BUG-020, BUG-021, BUG-022.

## Coverage Theater Prevention

The QPB spec warns repeatedly (`SKILL.md:625-644`, `SKILL.md:638`, `exploration_patterns.md:253-261`) that a model can produce plausible-looking quality artifacts that collapse under any scrutiny. This project is the canonical case. The following counter-measures apply to every artifact:

**Closed-set mechanical extraction is mandatory.** QPB has at least five closed sets (`SKILL_INSTALL_LOCATIONS`, `PROTECTED_PREFIXES`, `ALL_STRATEGIES`, verdict enum, phase identifiers). `quality/mechanical/verify.sh` must extract every one from source and diff against a saved baseline. The v1.3.23 failure mode described at `SKILL.md:638` — model overwrites correct extraction with fabricated content — is exactly what this project would regenerate, so the gate must not accept an absent mechanical/ directory as "not applicable" (see QR-10). Phase 2 produced `quality/mechanical/verify.sh`; that shell file was executed during Phase 2 with receipts saved at `quality/results/mechanical-verify.log` and `quality/results/mechanical-verify.exit`.

**Line-count gates are necessary but never sufficient.** EXPLORATION.md is 631 lines, well above the 120-line minimum. The spec's concern (at `SKILL.md:914`) is that length is gameable — a model under context pressure can pad. The gate self-check in Phase 1 therefore asserted *structure* (exact section titles, ≥ 4 distinct modules, ≥ 4 FULL patterns, ≥ 3 TRACE findings) on top of length. REQUIREMENTS.md applies the same discipline: every REQ-NNN has a named file, a line range, a user story, and at least one condition of satisfaction that can be falsified by code.

**No placeholder tests.** `functional_tests.md:42-67` enumerates prohibited patterns: `assert True`, `pass`, "test passes if no exception", assertions that can never fail. Every test in `quality/test_functional.py` either asserts a specific value, asserts a schema-compliant mutation, or asserts a raised exception. Tests that exist only to raise the count are coverage theater.

**Verdicts are quoted exactly from the enum.** `SKILL.md:154` names five verdicts; `quality_gate.py:497-498` enforces them as a closed set. Phase 3 regression tests declare `TDD verified | red failed | green failed | confirmed open | deferred` verbatim; any paraphrase is a gate failure.

**Source-of-truth citations.** Every requirement and every test docstring names a file path and line range. A reviewer must be able to open the source and read the line being tested. Citations without line ranges are incomplete; citations without a visible file path are forbidden.

**Bootstrap-specific hazard.** In this self-audit, *any* agent-authored edit to `SKILL.md`, `references/`, or `bin/` is silently reverted by `cleanup_repo` (QR-3). Patches — not direct edits — are the only path for a proposed fix. BUGS.md writeups in Phase 3 must include the patch as `quality/patches/BUG-NNN-fix.patch`, never as a direct source modification. The regression test in `quality/test_regression.py` must still pass after the patch is applied by hand.

## Fitness-to-Purpose Scenarios

These are architectural vulnerability analyses. Each names a specific failure surface, traces the cascade, and anchors the risk to the code path that would have to change to eliminate it. Unlike a bug list, these describe the *structural* conditions that let a bug exist — so fixing an individual bug may not remove the fitness gap.

### Scenario 1 — An installed SKILL.md ships with the bold version banner

**Vulnerability.** `benchmark_lib.skill_version()` (lines 90–116) uses case-sensitive `startswith("version:")` on stripped lines. `benchmark_lib._read_version()` (lines 74–81) uses a regex that also accepts `**Version:**` (bold). When a future installer, release script, or hand-edit puts the bold form into an installed `SKILL.md`, `skill_version()` returns `None`.

**Cascade.** `run_playbook.py:217` uses the return value to decide whether to apply the version-append fallback for a bare-name target. On `None`, the fallback is silently skipped; the user sees only `ERROR: 'chi' is not a directory (resolved to /cwd/chi)` with no clue that the parser is what failed. Meanwhile the run header on the same run calls `lib.detect_repo_skill_version(repo_dirs[0])` which *does* accept bold — so the header prints `v1.4.5` while the fallback says "no version detected." Two tools in the same run, two answers.

**Why it matters.** Cross-model benchmark runs (UC-03) depend on reproducibility: if a new installer style disables the fallback for one model but not another, the benchmark numbers drift without explanation. The bug is a *parser* bug, but the symptom is a *benchmark* invariance failure.

**Structural fix surface.** REQ-001 and REQ-002. One canonical helper, all callers delegate, grammar accepts every form `_read_version` accepts.

### Scenario 2 — A thin EXPLORATION.md passes the runner's soft gate

**Vulnerability.** The line-count threshold is 120 in `SKILL.md:906,916`, 80 in `orchestrator_protocol.md:41`, and 80 in `run_playbook.py:455-457` — where it is a WARN, not a FAIL. A model under context pressure can write an 85-line file and satisfy the runner while failing the skill's self-gate.

**Cascade.** Phase 2 consumes the exploration. On thin input, derived requirements (REQ-NNN) lose depth; tests in `quality/test_functional.py` lose coverage; Phase 3 runs without the quality-risks context the regression tests need. The defect cascades through every subsequent artifact and is only caught, if at all, by the maintainer at PR review time.

**Why it matters.** The v1.3.41 and v1.3.43 failure modes called out at `SKILL.md:407,914` are exactly this. They were caught by human review, not mechanically.

**Structural fix surface.** REQ-004. Unify the threshold at 120, promote the runner gate from WARN to FAIL, and have `quality_gate.py` re-verify the line count (as well as section structure — REQ-013).

### Scenario 3 — A user installs under `.github/skills/quality-playbook/`

**Vulnerability.** `SKILL.md:48-55` documents four install locations. `benchmark_lib.SKILL_INSTALL_LOCATIONS` has three. The fourth — `.github/skills/quality-playbook/SKILL.md` — is supported by `quality_gate.py:972-976` but not by the runner's `find_installed_skill`.

**Cascade.** A user with this layout sees `WARN: No SKILL.md found for ...` on every run (`run_playbook.py:237`) even though the skill is installed where SKILL.md's own docs say it may live. The run proceeds, but the run header and any subsequent version-dependent logic use `QPB_DIR/SKILL.md` — the *maintainer's* SKILL.md — instead of the target's installed copy. Cross-version drift goes undetected.

**Why it matters.** UC-06 (non-maintainer runs the playbook on their own project). A first-time user's install works with Copilot's documented alternate path, and the playbook warns falsely. Support cost, not correctness cost — but the warning erodes trust in every other warning the runner emits.

**Structural fix surface.** REQ-002. Add the fourth path; keep order consistent with `quality_gate.py`.

### Scenario 4 — Parent process SIGKILL mid-archive

**Vulnerability.** `archive_previous_run` (lines 565–576) calls `copytree` and `rmtree` in sequence with no rollback. If the process is killed between the two, both `previous_runs/TIMESTAMP/quality/` and `quality/` exist.

**Cascade.** On the next run, `resolve_target_dirs` doesn't detect the partial archive. With `--with-seeds`, Phase 0 injects stale seeds from the half-archived prior run into the new run. The archived `control_prompts/` is *missing* entirely (the function `rmtree`s it without archiving), so even debugging why the crash happened is hampered.

**Why it matters.** UC-04 (full-cycle iteration chain) and UC-03 (cross-model comparison) both depend on clean per-run state. A stale archive corrupts iteration metrics silently.

**Structural fix surface.** REQ-009. Archive atomically (temp-archive → rename), include `control_prompts/` in the archive, and add a boot-time check for partial archives.

### Scenario 5 — Agent fixes a bug by editing `SKILL.md` directly in a bootstrap run

**Vulnerability.** `cleanup_repo` reverts tracked non-protected files. Source — `SKILL.md`, `references/`, `bin/` — is tracked and unprotected. An agent that interprets "apply the fix" as "edit the source" loses the fix silently at the end of the session, with only a log line (`"Tidied 1 tracked file(s) in QPB: SKILL.md"`) as evidence. `AGENTS.md` has the same problem: it is a *required* Phase 2 artifact (`SKILL.md:106,122`) but lives at project root, outside all four protected prefixes.

**Cascade.** In every bootstrap self-audit, the agent's good-faith attempt to apply a fix vanishes. The regression test the agent also wrote still passes — but against the unchanged source, not the fixed source. A reviewer looking at the diff sees no source change and concludes the fix was a no-op.

**Why it matters.** UC-02 is this exact scenario. The product's own self-audit depends on a workflow the product cannot enforce.

**Structural fix surface.** REQ-006. Either protect `AGENTS.md` (add to `PROTECTED_PREFIXES`), or add a mechanical check that a required artifact's content has not been silently reverted since the last phase.

### Scenario 6 — Parallel runs race on archive timestamp

**Vulnerability.** Two parent processes started within the same wall-clock second compute the same `previous_runs/TIMESTAMP/quality/` path. Per-parent PID files protect worker registration but not archive directory naming.

**Cascade.** The second parent's `copytree` sees `quality/` already deleted by the first; raises `FileNotFoundError`; worker exits; the run fails mid-startup. The first parent continues normally but its archive has been overwritten.

**Why it matters.** UC-03 (multi-model benchmark) often launches several runners against the same target from a shell loop. Sub-second launches race.

**Structural fix surface.** REQ-009 with the additional clause that timestamps must be unique per parent — either include the PID in the path, or use a monotonically unique counter.

### Scenario 7 — Copilot child workers orphaned after parent crash

**Vulnerability.** `_pkill_fallback` at `run_playbook.py:808-829` has patterns for `bin/run_playbook.py`, `claude -p`, `claude --model` — but no pattern for `gh copilot -p`. The Copilot dispatch path (`command_for_runner` line 486-494) invokes `gh copilot` as a subprocess whose argv does not match any of the three patterns.

**Cascade.** If the parent crashes (OOM, SIGKILL, terminal close) after spawning Copilot workers but before they finish, the PID file is orphaned. A later `--kill` invocation hits the pkill fallback because no PID file is found — and pkill misses every `gh` process. The workers keep burning API quota until the Copilot CLI's own timeout triggers.

**Why it matters.** UC-05 (kill a runaway run). The escape hatch is advertised and documented; for Copilot it silently doesn't escape.

**Structural fix surface.** REQ-012. Add `gh copilot` patterns to the fallback; narrow the existing `claude -p` pattern to the playbook's exact invocation string so the fallback doesn't nuke the user's unrelated Claude CLI sessions.

### Scenario 8 — Run-metadata JSON has no gate

**Vulnerability.** `SKILL.md:123,174-201` declares `quality/results/run-YYYY-MM-DDTHH-MM-SS.json` a **Required** artifact. `quality_gate.py:1027-1053` never calls a `check_run_metadata` function — none exists. The file can be absent, malformed, or have bogus schema without any FAIL.

**Cascade.** A benchmark operator comparing runs across models (UC-03) has no guarantee that the metadata fields they use for comparison (`model`, `model_provider`, `runner`, `start_time`, `end_time`) are present and correctly typed. Two runs with different models may appear indistinguishable if the field is missing.

**Why it matters.** The artifact contract promises mechanical verification of every Required artifact. This one breaks that promise.

**Structural fix surface.** REQ-008. Add `check_run_metadata` to `quality_gate.py`. Cross-references: REQ-007 on ISO 8601 datetime grammar.

### Scenario 9 — `docs_present` accepts `.DS_Store` as documentation

**Vulnerability.** `run_playbook.py:560-562` returns `True` if `any(docs_dir.iterdir())` — any entry at all, including `.DS_Store`, empty subdirs, or zero-byte files.

**Cascade.** The "docs_gathered/ is missing or empty" SKIP is bypassed. Phase 1 runs with no real documentation. The model has no cross-reference material for exploration, so findings are weaker and coverage-commitment claims in EXPLORATION.md are untethered.

**Why it matters.** The spec (`SKILL.md:1-505`) repeatedly stresses that doc-enriched runs find different, higher-confidence bugs. The guardrail exists but is trivially bypassed.

**Structural fix surface.** REQ-011. Require at least one non-hidden, non-empty `.md`/`.txt`/`.pdf` document before declaring docs present.

### Scenario 10 — Ship-worthy verdict from a broken run

**Vulnerability.** `execute_run` tracks failures and returns non-zero, but calls `print_suggested_next_command` *before* returning, regardless of failure status. The suggestion walks the iteration cycle (`--next-iteration --strategy gap ...`) unconditionally.

**Cascade.** A user whose Phase 1 crashed sees a terminal message advising them to iterate on a broken run. The correct next action — rerun Phase 1 — is never suggested. The same line of code runs whether the run succeeded or failed.

**Why it matters.** UX-level quality. The runner's own output contradicts the exit code. Automation (CI) that trusts either signal gets it wrong.

**Structural fix surface.** REQ-014. Condition the suggestion on `failures == 0`; on failure, tell the user to investigate the failed phase.

### Scenario 11 — Source-checkout bootstrap fails before Phase 1 starts

**Vulnerability.** `README.md:49-61` tells maintainers to launch the orchestrator prompt from the source checkout of this repository, but `agents/quality-playbook.agent.md:35-43` and `agents/quality-playbook-claude.agent.md:45-55` omit plain repo-root `SKILL.md` from their setup search lists.

**Cascade.** A maintainer following the README can get a "skill not found" failure before any exploration or artifact generation begins. The same repo advertises source-checkout usage and then refuses to look at the checkout-local skill file.

**Why it matters.** Bootstrap quality. This repo self-audits and dogfoods its own prompts; a broken source-checkout entrypoint undermines the primary maintainer workflow.

**Structural fix surface.** REQ-018. Add repo-root `SKILL.md` to both shipped orchestrator setup lists and keep the README launch contract aligned with them.

### Scenario 12 — General orchestrator collapses the clean-context model

**Vulnerability.** `agents/quality-playbook.agent.md:11-14` says the orchestrator never executes phase logic itself and that each phase gets its own clean context window, but `agents/quality-playbook.agent.md:77-81` tells Mode 1 to "Run Phase 1 in the current session."

**Cascade.** A model following Mode 1 can skip the phase boundary discipline that the playbook treats as essential for depth and traceability. The failure is subtle: the run may "work," but exploration quality degrades because the orchestrator and the phase logic are sharing one context.

**Why it matters.** The playbook’s whole architecture depends on clean handoffs between phases. A contradiction in the orchestrator prompt directly attacks that design.

**Structural fix surface.** REQ-019. Make the role text and the mode instructions describe one ownership model for phase execution.

### Scenario 13 — Local pytest shim turns discovery commands into execution or crashes

**Vulnerability.** `pytest/__main__.py:16-34` always executes `unittest.TextTestRunner.run(suite)` and forwards any non-directory / non-`.py` argument into `loader.discover(str(path))`. As a result, `--collect-only` still runs tests, and pytest node IDs (`path.py::Test::case`) can raise `ImportError`.

**Cascade.** Maintainers using documented commands from `SKILL.md` for collection-only checks or targeted TDD runs get misleading behavior: discovery commands execute the full suite, while targeted node-ID commands crash before the regression test runs. Both failure modes waste debugging time and weaken trust in the bootstrap tooling.

**Why it matters.** The repo explicitly ships a local `pytest` shim and documents `python -m pytest` usage. If the shim does not honor that contract, the self-audit workflow becomes unreliable exactly where maintainers expect a familiar interface.

**Structural fix surface.** REQ-020. Either implement the documented CLI forms in the shim or reject them explicitly before they reach `unittest` discovery.

### Scenario 14 — Missing docs make code-only audits impossible

**Vulnerability.** `bin/run_playbook.py:645-649` and `671-675` treat missing docs as a hard skip instead of the warning-only path required by the skill.

**Cascade.** The maintainer cannot produce a baseline playbook for a code-only repository, so the documented "warn and proceed" branch is dead code at the product level. This is especially damaging during first-contact audits, where documentation is often exactly what is missing.

**Why it matters.** The playbook advertises code-only analysis as a supported fallback. A runner that refuses to exercise that fallback blocks legitimate users before Phase 1 even starts.

**Structural fix surface.** REQ-021. Keep the warning, drop the skip.

### Scenario 15 — Runner-generated prompts strand valid install layouts

**Vulnerability.** `phase1_prompt()` through `phase6_prompt()`, plus `single_pass_prompt()` and `iteration_prompt()`, hardcode `.github/skills/SKILL.md` and `.github/skills/references/...` even though the spec documents repo-root and Claude layouts too.

**Cascade.** A child model launched from a source checkout or a `.claude/skills/quality-playbook/` install is pointed at the wrong path before any phase logic begins. The parent runner can be configured correctly while the child prompt still fails, which makes the bug hard to diagnose from the console output alone.

**Why it matters.** This is a portability failure inside the bootstrap path. The runner is generating instructions that contradict its own product documentation.

**Structural fix surface.** REQ-022. Make prompt text portable across every documented install layout.

### Scenario 16 — Child process failures are logged as completed runs

**Vulnerability.** `run_prompt()` returns a non-zero exit code on child failure, but `run_one_phase()` and `run_one_singlepass()` ignore it and still log `Phase complete` / `Playbook complete`.

**Cascade.** Automation that trusts either log output or process exit status can believe a run succeeded when the Claude/Copilot child already failed. In the single-pass path the false success can also suppress the top-level failure counter, compounding BUG-008’s suggestion problem.

**Why it matters.** A quality system that cannot report its own failure state accurately undermines every downstream artifact.

**Structural fix surface.** REQ-023. Propagate non-zero child exit codes and stop logging success after failure.

### Scenario 17 — Fixture repos redefine the bootstrap repo’s language

**Vulnerability.** `.github/skills/quality_gate/quality_gate.py:209-250` counts source files inside nested benchmark fixtures under `repos/`, then `check_test_file_extension()` trusts that result and expects the wrong functional-test extension for the audited repo.

**Cascade.** The bootstrap self-audit fails mechanically even when the Python artifacts are correct, because the gate decides the project is Go purely from embedded fixtures. That false failure obscures real gate regressions and makes the validator look untrustworthy on its own home repo.

**Why it matters.** Self-audit credibility depends on the validator understanding the repository it is validating.

**Structural fix surface.** REQ-024. Ignore nested fixture repos when classifying the audited project language.

### Scenario 18 — Valid language-native functional test files are rejected or skipped

**Vulnerability.** `.github/skills/quality_gate/quality_gate.py:299-303` and `795-839` implement two parallel filename contracts for the same artifact. The existence gate omits `functional_test.go`, while the extension checker only searches for `test_functional.*` and returns early when that one prefix is absent.

**Cascade.** A Go repo with `quality/functional_test.go` can FAIL as though it has no functional tests at all. A Java, Scala, or TypeScript repo with `FunctionalTest.java`, `FunctionalSpec.scala`, or `functional.test.ts` can PASS existence but still skip extension validation with only a warning. The gate therefore becomes unreliable exactly on the multi-language surface the skill advertises.

**Why it matters.** This is not a minor naming nit; it is a parity failure inside the validator. The helper library, the docs, the file-existence gate, and the extension gate are all supposed to enforce one artifact contract. Today they enforce different ones.

**Structural fix surface.** REQ-025 and REQ-026. Share the naming matrix across discovery, existence, and extension-validation paths.

### Scenario 19 — Undocumented functional-test aliases can hide a missing canonical artifact

**Vulnerability.** `bin/benchmark_lib.py:19-26` includes `test_functional_test.*` in `FUNCTIONAL_TEST_PATTERNS` even though `SKILL.md:72` and `functional_tests.md:3` never document that alias. `find_functional_test()` exposes the permissive set, and `run_playbook.final_artifact_gaps()` trusts it when deciding whether to report a missing functional test.

**Cascade.** A repo can carry only `quality/test_functional_test.py`, and the runner will omit `"functional test"` from its missing-artifact list. The maintainer sees a cleaner end-of-run gap report than the published contract actually allows, while the quality gate (correctly) still treats the repo as non-conformant.

**Why it matters.** This is the helper-side mirror image of BUG-024. Even if the gate is fixed, the runner can still certify an undocumented artifact name and mislead the maintainer before Phase 6 ever runs.

**Structural fix surface.** REQ-027. Keep helper discovery aligned to the published functional-test naming matrix and reject undocumented aliases.

### Scenario 20 — Benchmark summaries overstate regression readiness

**Vulnerability.** `bin/benchmark_lib.py:28-33` lets `find_regression_test()` match `RegressionTest.*`, `regression_test.*`, and `test_regression_test.*`. `build_summary_rows()` turns any such match into `REGR=Y`, even though the published artifact contract in `SKILL.md:97`, `README.md:334`, and `quality_gate.py:894-901` is canonical `quality/test_regression.*`.

**Cascade.** A benchmark operator can compare repos and conclude that regression coverage exists where only a helper-only alias is present. That false `REGR=Y` masks a real closure gap: the repo would still fail the documented artifact contract if someone ran the gate.

**Why it matters.** The benchmark summary is the maintainer's at-a-glance dashboard. If it overstates regression readiness, the playbook's own progress reporting becomes coverage theater.

**Structural fix surface.** REQ-028. Restrict helper regression discovery to canonical `test_regression.*` and keep summaries aligned with the published contract.

## AI Session Quality Discipline

QPB is produced by AI agents running QPB's own playbook. Session-level discipline matters because every artifact is a model output that must pass mechanical scrutiny. The rules below apply whenever an agent touches `quality/` in any future run.

**Prompt context is load-bearing.** Each phase runs in its own AI context window (UC-06 describes this explicitly). The agent cannot rely on "I remember from an earlier phase" — only what is on disk. PROGRESS.md and the artifact contract together are the hand-off. Phase 2 that writes `RUN_CODE_REVIEW.md` must be runnable in Phase 3 with no additional human context.

**No speculative requirements.** REQ-NNN entries must cite code or spec. A requirement that says "the user probably wants X" is not a requirement; it is a design suggestion. Phase 2's requirements pipeline rejects entries without a `[Tier N] [source]` or `[Tier N] [SKILL.md §...]` citation header. This rule is stricter than the generic spec (which allows architectural-guidance ratios up to 3 per 15 requirements).

**Refuse to generate artifacts that can't be falsified.** If a test cannot fail, it is not a test. If a requirement has no condition of satisfaction, it is not a requirement. If a bug has no regression test or explicit exemption, it is not closed. Each artifact in `quality/` carries its falsification mechanism as part of its contract.

**Patches over direct edits.** Because `cleanup_repo` silently reverts source edits (QR-3), any proposed fix is committed as `quality/patches/BUG-NNN-fix.patch`. A reviewer applies the patch by hand, confirms the regression test goes from FAIL to PASS, then commits the source change outside the agent's workflow. This is a project-specific rule; QPB's general doctrine is patch-then-apply, but bootstrap makes the rule load-bearing.

**Long artifacts require internal navigation.** EXPLORATION.md and REQUIREMENTS.md in this project cross 600 lines. Every section carries a stable heading that downstream artifacts can cite. Phase 3 will link to REQ-NNN by name, not by line number. If headings rename, the link-rot is visible in the code review, not silent.

**Sidecar JSON is the machine contract.** `tdd-results.json` (schema 1.1), `integration-results.json` (schema 1.1), and `recheck-results.json` (schema 1.0) are the primary interfaces to the gate. Prose artifacts summarize; JSON carries the verdict. Any drift between prose verdict and JSON verdict is a bug that the gate must catch. Phase 2's REQ-010 expresses this as a contract.

## The Human Gate

QPB automates judgment to the edge of what machines can verify; beyond that edge, a human decides. The artifact contract, closed-set extraction, sidecar JSON, and `quality_gate.py` together cover mechanical conformance. They do not and cannot cover:

- **Whether a proposed fix is the *right* fix.** `quality/patches/BUG-NNN-fix.patch` may apply cleanly and cause the regression test to pass while introducing a worse problem (e.g., loosening the version regex to accept anything). A human reviews the patch diff.
- **Whether an architectural-guidance requirement (REQ-017) is satisfied in spirit.** The requirement "all closed sets share a single source" can be formally satisfied by a tuple that nobody uses. Only a human reading the code notices that nothing actually calls the unified source.
- **Whether the Council-of-Three spec audit turned up a real contradiction or a surface rephrasing.** The audit writes to `RUN_SPEC_AUDIT.md`; the maintainer decides which findings become CB-NNN and which are filed under "clarify SKILL.md wording."
- **Whether a deferred verdict is an honest deferral or an evasion.** The verdict enum includes `deferred`. The gate accepts it. A human decides whether the deferral is reasonable (e.g., requires a multi-week redesign) or whether it should be `confirmed open (xfail)` right now.
- **Whether a bootstrap self-audit's findings should feed the next SKILL.md release.** That decision is out of scope for any artifact; it is a product decision.

The human gate is not a weakness of the system — it is the designed seam. Quality engineering's job (Deming, Juran, Crosby all say this) is to make the mechanical part complete enough that human judgment is spent on what only humans can judge. Everything in `quality/` exists to protect that seam.
