# Integration Tests Protocol

Defines the Phase 3 (and later) integration test plan for QPB. Integration tests exercise whole-system flows — the runner spawning a worker, the worker writing artifacts, the gate validating them. They complement `quality/test_functional.py` (unit-level assertions on each function) with UC-level scenarios.

Output of this protocol: a test script, a run log in `quality/results/integration-run.log`, and `quality/results/integration-results.json` (schema version `"1.1"`).

## Scope

**In scope.** Pipeline scenarios that cross two or more modules:

- **UC-01** — Runner → worker → artifacts (single target).
- **UC-02** — Bootstrap self-audit end-to-end.
- **UC-05** — Kill path (parent → workers → PID files → kill).
- **UC-07** — Post-run gate walk (every `check_*` function).

**Out of scope.** Tests that depend on `claude -p` or `gh copilot -p` subprocesses succeeding. Those are live-AI integration tests that cost money and cannot run in CI. They are a separate skill-integration test track, not covered here. In this protocol, the worker subprocess is mocked where necessary.

## Fixture design

### Synthetic target repo

All integration tests use a throwaway `TemporaryDirectory` shaped as a valid QPB target:

```
<tmp>/
  SKILL.md                              # frontmatter with version: 1.4.5
  docs_gathered/README.md               # ≥ 512 bytes
  repos/                                # unused, present for shape
```

The fixture initializes a git repo and commits the SKILL.md and docs. Every test starts from this fixture and mutates only what its scenario requires.

### Fake runner subprocess

Because real workers invoke `claude -p` or `gh copilot -p`, tests that exercise the worker pipeline use a fake runner: a short shell or Python script that:

- Writes a known artifact (`quality/EXPLORATION.md`, `quality/REQUIREMENTS.md`, etc.) to the target.
- Returns a specified exit code.

The fake is injected by `monkeypatch`ing `command_for_runner` or by placing a `claude` / `gh` shim earlier in `$PATH`.

### Sidecar and metadata templates

The fixture ships with minimally valid sidecars the tests use as baselines:

- `quality/results/tdd-results.json` — schema "1.1", zero bugs.
- `quality/results/integration-results.json` — schema "1.1", recommendation "SHIP".
- `quality/results/run-2026-04-18T23-43-14.json` — run metadata with all required fields.

Mutations for negative tests (wrong schema version, wrong verdict, missing field) derive from these baselines.

## Execution UX

Integration tests run as a single Python script, `quality/test_integration.py`, using stdlib unittest. Organized by group (one `TestCase` per group). Run from QPB root:

```bash
python3 -m unittest quality.test_integration -v
```

Output is captured to `quality/results/integration-run.log`. The log contains, per test, the test name, PASS/FAIL/XFAIL, and a 1-line summary.

After the run, the script emits `quality/results/integration-results.json`:

```json
{
  "schema_version": "1.1",
  "skill_version": "1.4.5",
  "project": "quality-playbook",
  "model": "claude-opus-4-7",
  "runner": "claude-code",
  "date": "2026-04-18",
  "recommendation": "SHIP" | "FIX BEFORE MERGE" | "BLOCK",
  "groups": [
    {
      "name": "A",
      "title": "Runner invocation and worker lifecycle",
      "result": "PASS" | "FAIL" | "PARTIAL",
      "tests_passed": <int>,
      "tests_failed": <int>,
      "tests_xfailed": <int>,
      "tests_skipped": <int>,
      "uc_coverage": ["UC-01", "UC-05"]
    }
    /* one entry per group */
  ],
  "uc_coverage": {
    "UC-01": "covered",
    "UC-02": "covered",
    "UC-05": "covered",
    "UC-07": "covered",
    "UC-03": "partial (cross-model runs can't be scripted in CI)",
    "UC-04": "partial (iteration-chain tested via unit tests only)",
    "UC-06": "out-of-scope (requires live AI agent)"
  }
}
```

**Recommendation rules.**
- `SHIP` — every `result` is `PASS` and no regression test in the full suite is an unexpected pass.
- `FIX BEFORE MERGE` — at least one group is `FAIL` but the failures are regression-test-covered candidate bugs (i.e., expected until Phase 5 reconciliation).
- `BLOCK` — any group `FAIL` that is not a regression-test candidate (i.e., a new defect).

## Failure isolation

Each scenario runs in its own `TemporaryDirectory`. No test shares state with another. Worker subprocesses (when used) have a strict 30-second timeout — if a fake worker hangs, the test is marked FAIL with a timeout reason, never left to stall the suite.

## Phase-3 deliverables from this protocol

- `quality/test_integration.py` — the full Python script implementing every scenario above.
- `quality/results/integration-run.log` — raw transcript of the run.
- `quality/results/integration-results.json` — structured sidecar matching the template above.

The sidecar is a required artifact per SKILL.md:94-123 (`integration-results.json` — Yes).
