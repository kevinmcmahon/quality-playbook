# Exploration Findings — Iteration 2 (Gap Strategy)

## Domain and Stack

The target remains the `quality-playbook` repository: a specification-first quality skill with supporting Python tooling, Markdown orchestrator agents, benchmark helpers, and a local `pytest` shim. This gap iteration focuses on user-facing instruction surfaces and thinly explored bootstrap tooling instead of the runner / gate code that dominated the baseline run.

## Coverage Scan from the Previous Run

The baseline run covered `bin/run_playbook.py`, `bin/benchmark_lib.py`, `.github/skills/quality_gate/quality_gate.py`, and the main skill/reference files in depth. It did **not** do equivalent pressure-testing on:

- `agents/quality-playbook.agent.md`
- `agents/quality-playbook-claude.agent.md`
- `pytest/__main__.py`
- README launch instructions that describe how those files are supposed to be used

That makes these files good `gap` targets: they are user-facing, behavior-shaping, and only lightly constrained by existing tests.

## Open Exploration Findings

1. **General orchestrator search paths omit repo-root `SKILL.md`.** `agents/quality-playbook.agent.md:35-43` searches `quality/SKILL.md`, `.github/skills/SKILL.md`, and `.claude/skills/quality-playbook/SKILL.md`, but not plain `SKILL.md` in the repo root. The repo README tells users to run the orchestrator agent from this repository root (`README.md:49-61`), where `SKILL.md` exists. The bootstrap path described to users can therefore fail before Phase 1 starts.

2. **Claude orchestrator has the same omission.** `agents/quality-playbook-claude.agent.md:45-55` checks `.claude/...`, `.github/...`, and `quality/SKILL.md`, but also omits plain `SKILL.md`. This is a cross-agent bootstrap hole, not a one-file typo.

3. **The omission survives a cross-file trace from README to agent setup.** The user flow is: README instructs "copy/paste `agents/quality-playbook.agent.md`" from the repo root (`README.md:49-61`) → the agent reads its own setup section (`agents/quality-playbook.agent.md:35-43`) → setup searches multiple installed-skill locations but never the root file that ships in this repo. The contradiction is between documented launch entrypoint and actual setup search list.

4. **General orchestrator role text contradicts Mode 1 execution text.** `agents/quality-playbook.agent.md:11-14` says "You do NOT execute phase logic yourself" and "Each phase runs in its own clean context window." Later, `agents/quality-playbook.agent.md:77-81` says "Run Phase 1 in the current session." The same file simultaneously mandates sub-agent isolation and direct in-context execution.

5. **The contradiction crosses three locations, not one sentence.** The role block (`agents/quality-playbook.agent.md:11-14`), the mode definition (`agents/quality-playbook.agent.md:62-76`), and Mode 1 instruction (`agents/quality-playbook.agent.md:77-81`) jointly create the bug. A consumer following the role guidance cannot also follow the mode guidance without violating the file’s own architecture.

6. **The local pytest shim ignores `--collect-only` semantically.** `pytest/__main__.py:16-34` parses arguments with `parse_known_args()`, but always ends by calling `unittest.TextTestRunner(...).run(suite)`. A runtime probe against `.github/skills/quality_gate/tests/test_quality_gate.py` showed that `python3 -m pytest --collect-only -q ...` executed the suite instead of performing collection-only behavior.

7. **The local pytest shim passes pytest node IDs directly into unittest discovery.** In `pytest/__main__.py:26-31`, anything that is neither a directory nor a `.py` file path is forwarded to `loader.discover(str(path))`. A pytest node ID like `quality/test_regression.py::TestRegression::test_bug_001` is not a filesystem path; the shim forwards it to discovery and triggers `ImportError` instead of a controlled response.

8. **The node-ID problem is documented-reachable.** `SKILL.md:1216-1243` and `SKILL.md:1278-1295` show `python -m pytest ...::test_bug_nnn` forms in the TDD instructions, and the README notes that the repo ships a local `pytest` shim (`README.md:314-316`). That means the repo documents a CLI grammar its own shipped shim does not support.

9. **There is no direct regression coverage for these instruction surfaces.** `quality/test_functional.py` and `quality/test_regression.py` cover runner, gate, and artifact behavior, but they do not assert the orchestrator search paths or the shim’s documented CLI forms. That coverage hole explains why the contradictions survived the baseline.

10. **Gap tests nearby reinforce the missing coverage pattern.** `bin/tests/test_run_playbook.py` and `.github/skills/quality_gate/tests/test_quality_gate.py` exercise Python runtime code but do not pressure-test README-to-agent alignment, agent-to-agent parity, or shim compatibility claims. The missing bugs sit in the seams between docs and bootstrap tools.

## Quality Risks

1. **Priority 1 — root-checkout bootstrap dead end.** Because `agents/quality-playbook.agent.md:35-43` and `agents/quality-playbook-claude.agent.md:45-55` omit plain `SKILL.md`, a user following `README.md:49-61` from the repo root can get a "skill not found" setup failure instead of entering Phase 1. This is high-impact for the self-audit / bootstrap path because it blocks the primary advertised entrypoint.

2. **Priority 2 — orchestration-context collapse.** Because `agents/quality-playbook.agent.md:11-14` requires clean per-phase contexts but `agents/quality-playbook.agent.md:77-81` tells the same agent to run Phase 1 in the current session, a model can skip the context boundary that the playbook treats as mandatory. That risks shallow exploration, phase bleed, and lower bug yield rather than an obvious crash.

3. **Priority 3 — collection-only commands can mutate audit state.** Because `pytest/__main__.py:16-34` always runs the suite, a user or automation invoking `python3 -m pytest --collect-only -q ...` can accidentally execute tests, create output, or spend time debugging why "collection" had side effects. For a self-auditing repo, a discovery-only command that runs real tests is the wrong failure mode.

4. **Priority 4 — TDD commands can crash on documented selectors.** Because `pytest/__main__.py:26-31` forwards node IDs into `unittest` discovery, a documented TDD command of the form `python3 -m pytest quality/test_regression.py::...` can terminate with `ImportError` before it reaches the regression test. That breaks a core verification workflow.

5. **Priority 5 — multi-surface contract drift can compound.** README launch docs, both orchestrator agents, the local pytest shim, and `SKILL.md` CLI examples define one bootstrap system across different media. When those surfaces drift independently, users get non-reproducible setup failures that look like tool instability rather than spec drift, making them harder to triage quickly.

## Skeletons and Dispatch

- **Instruction dispatch / setup search lists:** both agent files implement a closed-set search order for locating the skill file. Root `SKILL.md` is absent from both sets even though it is the repo-local source of truth.
- **Mode/state contract in general orchestrator:** the general agent has a state-like split between "Mode 1" and "Mode 2", but the constraints on who executes phase logic are not handled consistently across modes.
- **CLI grammar dispatch in pytest shim:** the shim has a three-way branch: directory, `.py` path, or "everything else". Pytest node IDs land in the catch-all branch even though they are a first-class CLI grammar in the repo’s TDD instructions.

## Pattern Deep Dive — Fallback and Degradation Path Parity

**Target:** repo-root launch path for orchestrator agents.

**Trace:** `README.md:49-61` tells the user to paste an orchestrator agent prompt from the repo root. The first operational step inside `agents/quality-playbook.agent.md:35-43` is "Find the skill file", but its ordered search list never checks `SKILL.md` in the current directory. The Claude variant repeats the same omission at `agents/quality-playbook-claude.agent.md:45-55`.

**Why this is a fallback bug:** the files already encode a degraded search order across multiple install layouts. The bug is not "no fallback exists"; it is "one supported layout from the repo’s own docs is absent from the fallback set." A user who happens to copy the skill into `.github/skills/` succeeds, while a user who runs directly from the source checkout can fail despite following the README.

**Pressure-test result:** confirmed net-new candidate. The omission is source-level and reproducible by inspection across both orchestrator files.

## Pattern Deep Dive — Cross-Implementation Consistency

**Target:** general vs Claude orchestrator setup rules.

**Trace:** both files describe themselves as orchestration-first wrappers around the same playbook, but the general file checks `quality/SKILL.md` before installed-skill locations while the Claude file checks `.claude/...`, then `.github/...`, then `quality/SKILL.md`. More importantly, both omit plain `SKILL.md`, so the shared inconsistency is against the README launch contract rather than against each other alone.

**Cross-file consequence:** a maintainer can fix or validate only one agent and still leave the bootstrap path broken for the other. The repo needs a shared bootstrap contract, not two drifting search lists.

**Pressure-test result:** supports the same net-new bug as the fallback analysis and strengthens severity because the drift spans both supported orchestrators.

## Pattern Deep Dive — API Surface Consistency

**Target:** general orchestrator role contract.

**Trace:** the role section (`agents/quality-playbook.agent.md:11-14`) says the orchestrator never executes phase logic itself and that each phase gets its own clean context. The mode section frames Mode 1 as the no-sub-agent path for tools without child agents, but the concrete instruction at `agents/quality-playbook.agent.md:77-81` is "Run Phase 1 in the current session." That instruction turns the orchestrator into the phase executor and defeats the clean-context requirement the file just declared.

**Consistency failure:** the same public interface tells the operator two incompatible things about who owns Phase 1 execution. This is not just a wording issue; it changes whether the playbook’s core phase-boundary discipline is preserved.

**Pressure-test result:** strong standalone bug candidate.

## Pattern Deep Dive — Spec-Structured Parsing Fidelity

**Target:** local pytest shim command grammar.

**Trace across code + docs:** `SKILL.md:1216-1243` and `SKILL.md:1278-1295` document pytest selectors using node IDs and collection-only flows. The shipped shim at `pytest/__main__.py:16-34` parses with `argparse.parse_known_args()` but always executes the unittest suite, and it routes any non-directory / non-`.py` token into `loader.discover(str(path))`. Runtime probing showed:

- `python3 -m pytest --collect-only -q .github/skills/quality_gate/tests/test_quality_gate.py` ran the full test suite instead of collecting only.
- `python3 -m pytest .github/skills/quality_gate/tests/test_quality_gate.py::TestValidateIsoDate::test_valid_today` crashed with `ImportError`.

**Why this is a parsing-fidelity bug:** the repo documents pytest grammar but the shim implements a much smaller ad hoc grammar without explicit rejection. The mismatch is silent for `--collect-only` and noisy for node IDs, both of which are bad operator experiences.

**Pressure-test result:** strong standalone bug candidate with both source and runtime evidence.
