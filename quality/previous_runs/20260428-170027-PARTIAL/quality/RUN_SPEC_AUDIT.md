# Spec Audit Protocol — Council of Three

Phase 4 applies the Council-of-Three protocol to the QPB specification: three independent audits of the same material, each with a distinct stance, converging on a triaged list of specification issues. The output is `quality/spec_audits/` (one markdown file per council member) and a reconciliation pass that folds findings into `quality/BUGS.md` or `quality/RUN_SPEC_AUDIT_FINDINGS.md` for spec-only issues.

## Pre-audit document validation

Per `references/spec_audit.md`, the audit must only run after the following are present and readable:

- `quality/EXPLORATION.md` — the Phase 1 findings that seed the audit.
- `quality/REQUIREMENTS.md` — the 17 requirements under audit.
- `quality/CONTRACTS.md` — the 21 behavioral contracts.
- `quality/QUALITY.md` — the constitution (so the audit knows the standard).
- `quality/COVERAGE_MATRIX.md` — the requirement-to-test map.
- `quality/BUGS.md` — Phase 3 confirmed bugs (if any).
- `quality/test_functional.py` — baseline behavior under audit.
- `quality/test_regression.py` — Phase 3 red-phase tests.
- `.github/skills/SKILL.md` (or installed equivalent) — the specification itself.

If any of these are missing or empty, the audit does not start. Verification is mechanical: each file exists and has > 100 lines.

## Scope of the audit

**Primary target.** `SKILL.md` — the 2342-line specification. Every phase section, every artifact-contract row, every sidecar JSON template, every reference-file citation.

**Secondary targets.** The 14 reference files in `.github/skills/references/`. Each reference file is an extension of SKILL.md prose; each is audited for internal consistency *and* consistency with SKILL.md.

**Tertiary targets.** `agents/quality-playbook.agent.md` and `agents/quality-playbook-claude.agent.md`. These are orchestrator instructions; the audit checks that they agree with SKILL.md on phase order, artifact names, and verdict enums.

**Excluded.** Source code (`bin/`, `.github/skills/quality_gate/quality_gate.py`). That's Phase 3's scope. The spec audit assumes Phase 3 has already generated the code-review and regression evidence.

## The three council stances

Each council member is a distinct AI agent session — a fresh context window — invoked with the same inputs but a different framing prompt.

### Council Member 1 — The Literalist

**Stance.** Every word of SKILL.md is load-bearing. If the spec says a file must have "exactly the section title `## Quality Risks`", the Literalist flags any instance in any artifact where the heading is `## Quality Risk` (singular), `## QUALITY RISKS` (caps), or `### Quality Risks` (wrong level).

**Inputs.** SKILL.md, REQUIREMENTS.md, the actual `quality/` artifacts produced by Phase 1 and Phase 2 in this run.

**Technique.** For each contract line in SKILL.md, quote the exact spec text, then find the corresponding artifact text, then diff. Flag every character-level divergence.

**Expected output.** `quality/spec_audits/council-1-literalist.md`. A table of findings:

| # | Spec citation | Spec text | Artifact text | Status |

**Typical findings.** Verdict enum case drift. Artifact contract naming (e.g., `AGENTS.md` vs `agents.md`). Section title pluralization.

### Council Member 2 — The Semanticist

**Stance.** Specs can be internally inconsistent even when every instance is literally compliant. The Semanticist looks for contradictions: two spec passages that mean different things about the same contract.

**Inputs.** SKILL.md, reference files, this project's EXPLORATION.md.

**Technique.** Build a contract-by-contract concordance: for each behavioral contract (C-1 through C-21 in CONTRACTS.md plus any new ones), collect every SKILL.md and reference-file passage that mentions it. Compare. Flag any two passages that say different things about the same contract.

**Expected output.** `quality/spec_audits/council-2-semanticist.md`. A per-contract section listing every citation and the differences:

```
## Contract: EXPLORATION.md line-count threshold
- SKILL.md:906 — "at least 120 lines of substantive content."
- SKILL.md:916 — "at least 120 lines of substantive content."
- orchestrator_protocol.md:41 — "more than 80 lines of substantive content."
- FINDING: two passages disagree on the threshold (80 vs 120). The skill's
  self-gate is 120; the orchestrator reference is 80. Three-way drift with
  the runner (also 80). Recommend unifying on 120.
```

**Typical findings.** Line-count thresholds (F-2). Install-path lists (F-3). Parser grammar in different reference files. Run-metadata schema differences between the artifact contract row and the run-metadata section.

### Council Member 3 — The Operationalist

**Stance.** A spec is only as strong as its machine-verifiable checks. If the spec says a thing is Required but no gate checks it, the spec is aspirational not operational. The Operationalist asks: for every contract, does `quality_gate.py` actually enforce it?

**Inputs.** SKILL.md, CONTRACTS.md, `.github/skills/quality_gate/quality_gate.py`.

**Technique.** For each contract in CONTRACTS.md, find the `check_*` function in quality_gate.py that enforces it. If none exists, flag. If one exists but the check is loose (substring regex, missing case), flag.

**Expected output.** `quality/spec_audits/council-3-operationalist.md`. A gap table:

| Contract | Gate function | Strictness | Gap |
|----------|---------------|------------|-----|
| C-1 Version source | `check_version_stamps` (line 969) | Loose (substring match) | Yes — substring accepts `description: "version: X"` |
| C-10 Run metadata | _no function_ | — | Yes — REQ-008 |
| C-17 ISO 8601 | `validate_iso_date` | Date-only; rejects datetime | Yes — REQ-007 |

**Typical findings.** Missing `check_*` functions. Regex drift. Loose sentinel matches. Validator grammar mismatches.

## Verification probes

Per `references/spec_audit.md`, each council member must include at least three verification probes — specific inputs the auditor pushed through the spec (or the gate, for the Operationalist) to falsify a hypothesis. A finding without a probe is a guess; a finding with a probe is evidence.

**Probe template.**
```
### Probe
- Hypothesis: <what the auditor expected>
- Input: <the exact text, file contents, or JSON>
- Observed: <what actually happened — which sentence of SKILL.md applied, which gate function fired, etc.>
- Conclusion: <confirmed | refuted | inconclusive>
```

**Example probe (Literalist).** Hypothesis: the spec requires `## Open Exploration Findings` exactly. Input: an EXPLORATION.md with `## Open Findings`. Observed: SKILL.md:914 requires the full phrase. Conclusion: confirmed; any variant is a spec violation.

**Example probe (Semanticist).** Hypothesis: the line-count threshold is unified at 120 across SKILL.md and reference files. Input: grep `"120"` and `"80"` in SKILL.md and `references/`. Observed: `orchestrator_protocol.md:41` says 80; SKILL.md says 120 (twice). Conclusion: refuted; the threshold is inconsistent.

**Example probe (Operationalist).** Hypothesis: `quality_gate.py` enforces the run-metadata filename format. Input: grep `check_run_metadata` in `quality_gate.py`. Observed: no such function. Conclusion: confirmed gap; REQ-008.

## Reconciliation pass

After all three council members have written their audits, a reconciler reads them together and produces `quality/RUN_SPEC_AUDIT_FINDINGS.md`.

**Reconciliation rules.**
- A finding flagged by **2 or more council members** is *confirmed*. The file records all three citations.
- A finding flagged by **only 1 council member** is *proposed*. The reconciler either promotes it (with explicit reasoning) or records it as *deferred-single-source*.
- A finding flagged by **all three** is *high-confidence*. These should be the first to flow into BUGS.md or a spec-patch pull request.
- Any disagreement between council members (e.g., Literalist says "compliant" but Operationalist says "spec is operational fiction") is called out explicitly — the reconciler picks a side and justifies.

**Reconciler output template.**
```markdown
# Spec Audit Findings (Reconciled)

## Confirmed findings (≥2 council members)
- **F-N: <title>**
  - Literalist: <quote or N/A>
  - Semanticist: <quote or N/A>
  - Operationalist: <quote or N/A>
  - Resolution: <spec fix | code fix | close without action, with reasoning>
  - Related requirements: REQ-NNN, CB-N

## Proposed findings (1 council member)
- ...

## Disagreements
- ...
```

## Output to BUGS.md

Spec-only findings (wording inconsistency, missing check) that do not already map to a CB-NNN in BUGS.md become new bugs with source stage `spec-audit`. They get the same writeup treatment as code bugs (see RUN_CODE_REVIEW.md's BUG writeup template).

A spec bug whose fix is a prose change to SKILL.md still uses a patch (`quality/patches/BUG-NNN-spec.patch`) — again, cleanup_repo reverts direct edits.

## Deliverables checklist

Before Phase 5 entry, the audit must produce:

- `quality/spec_audits/council-1-literalist.md` — with at least 3 probes.
- `quality/spec_audits/council-2-semanticist.md` — with at least 3 probes.
- `quality/spec_audits/council-3-operationalist.md` — with at least 3 probes.
- `quality/RUN_SPEC_AUDIT_FINDINGS.md` — the reconciled output.
- Any new CB-NNN entries appended to `quality/BUGS.md` with source stage `spec-audit`.
- Spec-fix patches, if applicable, under `quality/patches/`.

`check_phase_gate("5")` requires `quality/RUN_SPEC_AUDIT_FINDINGS.md` to exist (current gate checks RUN_SPEC_AUDIT.md only — REQ-010 expands this).

## Quality signal

The spec audit is "successful" when:

- All three council-member outputs contain at least 3 concrete findings each with verification probes.
- The reconciliation identifies at least 1 "confirmed" finding (flagged by ≥ 2 members).
- At least one council finding crosses back to a Phase 3 CB-NNN — demonstrating that code review and spec audit are both checking the same contract.
- No council member concludes with "no findings" — if the spec were perfect, we wouldn't be bootstrapping. A "no findings" result is almost always incomplete; the reconciler should push back.
