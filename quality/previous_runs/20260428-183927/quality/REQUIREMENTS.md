# Behavioral Requirements — quality-playbook
> Generated by [Quality Playbook](https://github.com/andrewstellman/quality-playbook) v1.5.3 — Andrew Stellman
> Date: 2026-04-28 · Project: quality-playbook

Total requirements: 8

quality-playbook is a hybrid repository: part executable Python benchmark/orchestration tool, part product-spec bundle made of `SKILL.md`, `references/`, install surfaces, and bootstrap documentation. Its users are maintainers evolving the playbook, benchmark operators installing it into target repos, and downstream AI agents that inherit whatever persistent instructions and artifacts the repository leaves on disk. That means the repository's "runtime" is broader than the Python process. A Markdown heading in `SKILL.md`, a copied install file, a gate invariant, and a live `quality/INDEX.md` payload all participate in whether later phases see the right truth.

The dominant data flow in this run starts at documentation intake and ends at later-phase evidence. Phase 1 should read `reference_docs/` when present, fall back cleanly to Tier 3 source evidence when not, and record what it learned in `quality/EXPLORATION.md`. Phase 2 and later phases then consume those artifacts by name: the exploration headings, the requirement manifest, the use cases, and the live run index all become machine- and human-readable sources of truth. If the repo publishes mismatched surfaces—for example, a skill that names one gate contract while `bin/run_playbook.py` enforces a weaker one, or an install path that copies `quality_gate.py` without its verifier dependency—later phases inherit a false but plausible view of the system.

The main design philosophy visible in exploration is "executed evidence outranks prose," but the repo currently contains several places where prose/code/install paths drift apart. The requirements below therefore focus on preserving closed sets, parity surfaces, and compensating paths. The review lens for this project is not just "does the command run?" but "does every documented path teach and enforce the same quality bar?" A playbook that silently weakens itself during install or self-audit still produces artifacts, but those artifacts are no longer trustworthy.

For this Phase 2 baseline, all requirements are Tier 3. `reference_docs/` was effectively empty in this checkout, so no Tier 1 or Tier 2 formal-doc citations were available. The authoritative source of truth for these REQ records is the explored code, the shipped skill bundle, and the repository's published bootstrap artifacts.

## Derived use cases

### UC-01: Benchmark operator runs Phase 1 on a source-heavy target

- **Actor:** Benchmark operator
- **Preconditions:** The target repo may have sparse or missing `reference_docs/`.
- **Steps:**
  1. Start a phased run from the repo root or an installed target.
  2. Let the runner inspect available documentation inputs.
  3. Continue into exploration using the strongest evidence the install actually provides.
- **Postconditions:** The operator receives accurate messaging about documentation strength and the exploration artifact reflects the real evidence baseline.
- **Alternative paths:** A modern install with populated `reference_docs/` must not be mislabeled as "code-only" because `docs_gathered/` is absent.
- **Requirements:** REQ-002, REQ-003, REQ-008

### UC-02: Maintainer self-audits QPB at the repo root

- **Actor:** QPB maintainer
- **Preconditions:** The repo root is the target and curated bootstrap docs are present.
- **Steps:**
  1. Run the playbook directly against the repo root.
  2. Let Phase 1 read the live skill plus the curated bootstrap context.
  3. Compare the resulting evidence surface with the benchmark harness install model.
- **Postconditions:** The self-audit sees the same effective documentation surface the harness would expose.
- **Alternative paths:** Curated docs may be supplemental, but they must not contradict the live release or route the audit through an obsolete terminology path.
- **Requirements:** REQ-002, REQ-007, REQ-008

### UC-03: Later phases consume a generated exploration artifact

- **Actor:** Later-phase AI agent
- **Preconditions:** Phase 1 wrote `quality/EXPLORATION.md` and `quality/PROGRESS.md`.
- **Steps:**
  1. Read the exploration artifact from disk.
  2. Validate the required headings and content structure.
  3. Generate downstream artifacts from that stable baseline.
- **Postconditions:** Later phases operate on a structurally valid exploration artifact rather than a long but malformed placeholder.
- **Alternative paths:** A missing or underspecified exploration artifact blocks progression instead of quietly slipping through.
- **Requirements:** REQ-003, REQ-004

### UC-04: Maintainer classifies a target as Code, Skill, or Hybrid

- **Actor:** Maintainer or automation
- **Preconditions:** The target repo contains enough evidence for project classification.
- **Steps:**
  1. Run the classifier.
  2. Persist the resulting classification into live run metadata.
  3. Let later tooling reason from that persisted value.
- **Postconditions:** Downstream artifacts and reviewers see the actual project type rather than a hardcoded default.
- **Alternative paths:** Explicit overrides still need to survive into `quality/INDEX.md`.
- **Requirements:** REQ-004

### UC-05: Auditor relies on byte-verified citations on an installed target

- **Actor:** Auditor or quality gate
- **Preconditions:** A requirement or bug cites a `reference_docs/cite/` excerpt.
- **Steps:**
  1. Read the installed gate package from the target repo.
  2. Re-run deterministic excerpt extraction against the cited document bytes.
  3. Fail if the stored excerpt and fresh extraction differ.
- **Postconditions:** Installed runs retain the same Layer-1 citation rigor as repo-root runs.
- **Alternative paths:** Missing verifier dependencies are treated as install defects, not as a silent downgrade in trust.
- **Requirements:** REQ-006

### UC-06: Operator terminates a stuck run without collateral damage

- **Actor:** Benchmark operator
- **Preconditions:** A run crashed or left worker processes behind.
- **Steps:**
  1. Invoke cleanup.
  2. Read the recorded PID files for the relevant run.
  3. Terminate only the workers belonging to that run.
- **Postconditions:** Cleanup resolves the stuck run without killing unrelated local sessions.
- **Alternative paths:** Missing PID files should degrade to a run-scoped no-op or explicit manual intervention, not a generic pattern kill.
- **Requirements:** REQ-005

### UC-07: Installer or downstream agent reads either shipped skill entry point

- **Actor:** Installer or AI agent
- **Preconditions:** The repo root and installed skill path both exist.
- **Steps:**
  1. Resolve the skill through one supported path.
  2. Resolve it again through the alternate supported path.
  3. Compare the instructions, version, and reference resolution guidance.
- **Postconditions:** Both entry points teach the same playbook contract.
- **Alternative paths:** Fallback path selection must never change the operator brief.
- **Requirements:** REQ-001

## Cross-cutting concerns

Three concerns span the whole requirements set. First, **closed-set preservation**: REQ-003 and REQ-004 protect enumerated contracts where omission is the bug. Required headings and supported project types are both small, explicit sets, so they need mechanical preservation instead of best-effort prose alignment. Second, **parity across surfaces**: REQ-001, REQ-002, REQ-006, and REQ-008 all defend places where two legitimate entry points should describe or enforce the same behavior. When these paths drift, the repo still appears usable while different operators get different answers.

Third, **safe degradation**: REQ-005 and REQ-007 cover fallback paths that should stay truthful under weaker conditions. Missing PID files must not widen cleanup scope, and curated docs must not become stale enough to mislead self-audits. Reviewers should treat these "still works" paths as first-class quality surfaces, because this repository's biggest risks are not hard crashes but success-shaped drift.

## Packaging and Installed Surface Integrity

This section covers the files and install paths that determine what downstream agents actually read. The important review question is whether alternate supported entry points remain semantically equivalent after copying, mirroring, or fallback resolution.

### REQ-001: Installed skill entry points must stay synchronized with the repo-root skill

- **Tier:** 3
- **Functional section:** Packaging and Installed Surface Integrity
- **Pattern:** parity
- **Source type:** code-derived
- **Summary:** The repo-root `SKILL.md` and installed `.github/skills/SKILL.md` entry point must always resolve to the same instructions and version.
- **User story:** As an installer or downstream AI agent, I expect either documented skill path to yield the same playbook contract **so that** fallback resolution never changes what work gets performed.
- **Implementation note:** The repository currently ships both paths and exploration confirmed they are aligned today (`SKILL.md:1-20`, `.github/skills/SKILL.md:1-20`).
- **Conditions of satisfaction:**
  1. The installed skill entry point renders the same version stamp and top-level instructions as the repo-root skill.
  2. Reference-file resolution guidance is identical across both entry points.
  3. Packaging changes cannot introduce instruction skew between the two supported skill paths.
- **Alternative paths:**
  - Symlinked and copied installs must both preserve semantic equivalence.
  - Fallback readers may use either path without needing special-case logic.
- **Use cases:** UC-07
- **References:** `SKILL.md:1-20`; `.github/skills/SKILL.md:1-20`

### REQ-002: The runner must treat `reference_docs/` as the authoritative documentation input for Phase 1

- **Tier:** 3
- **Functional section:** Packaging and Installed Surface Integrity
- **Pattern:** parity
- **Source type:** code-derived
- **Summary:** Modern installs that provide `reference_docs/` must not be downgraded or warned as "code-only" because `docs_gathered/` is absent.
- **User story:** As a benchmark operator, I expect the runner's preflight messaging to match the modern documentation-intake contract **so that** I know when a run truly lacks documentation.
- **Implementation note:** `formal_docs_guard_banner()` already recognizes `reference_docs/`, while `docs_present()` still checks only `docs_gathered/` (`bin/run_playbook.py:1552-1622`).
- **Conditions of satisfaction:**
  1. Phase 1 documentation-availability messaging keys off `reference_docs/` on modern installs.
  2. A populated `reference_docs/` tree suppresses any "code-only" warning path that exists only because `docs_gathered/` is missing.
  3. Direct-root and installed runs describe the same documentation strength when they expose the same effective `reference_docs/` content.
- **Alternative paths:**
  - Empty or missing `reference_docs/` may still produce a non-blocking Tier 3 warning.
  - Supplemental `docs_gathered/` content may exist, but it must not outrank canonical `reference_docs/`.
- **Use cases:** UC-01, UC-02
- **References:** `README.md:25-50`; `bin/run_playbook.py:1552-1622`; `bin/reference_docs_ingest.py:270-300`

### REQ-003: Phase 2 entry gating must enforce the full `EXPLORATION.md` section contract, not just file length

- **Tier:** 3
- **Functional section:** Packaging and Installed Surface Integrity
- **Pattern:** whitelist
- **Source type:** skill-section
- **Skill section:** Phase 2: Generate the Quality Playbook
- **Summary:** A run must not advance beyond Phase 1 unless `quality/EXPLORATION.md` satisfies the exact heading and content checks named in the Phase 2 skill contract.
- **User story:** As a later-phase AI agent, I expect the exploration artifact I inherit to satisfy the named contract **so that** downstream artifacts are anchored to a stable, auditable baseline.
- **Implementation note:** The written contract lives in `SKILL.md:1026-1042`, but `check_phase_gate()` currently enforces only existence and length (`bin/run_playbook.py:1152-1165`).
- **Conditions of satisfaction:**
  1. The executable gate rejects any exploration artifact missing `## Open Exploration Findings`, `## Quality Risks`, `## Pattern Applicability Matrix`, `## Candidate Bugs for Phase 2`, or `## Gate Self-Check`.
  2. The executable gate rejects any exploration artifact missing the required minimum number of pattern deep dives and content-shape checks from the written contract.
  3. A 120-line placeholder without the required headings never unlocks Phase 2.
- **Alternative paths:**
  - A complete exploration artifact may still pass with more than the minimum findings or deep dives.
  - The gate may report multiple missing-heading diagnostics, but it must not silently pass on length alone.
- **Use cases:** UC-01, UC-03
- **References:** `SKILL.md:1026-1042`; `bin/run_playbook.py:1152-1165`; `bin/tests/test_run_playbook.py:116-130`

### REQ-004: Live run metadata must preserve the actual Code/Skill/Hybrid project classification

- **Tier:** 3
- **Functional section:** Packaging and Installed Surface Integrity
- **Pattern:** whitelist
- **Source type:** code-derived
- **Summary:** Once a target is classified, that exact `Code` / `Skill` / `Hybrid` value must survive into live `quality/INDEX.md` metadata.
- **User story:** As a maintainer or reviewer, I expect run metadata to reflect the actual target shape **so that** later tooling applies the right Hybrid/Skill reasoning instead of silently assuming `Code`.

  2. `write_live_index_final()` preserves the same project type when re-rendering the live index.
  3. All supported project-type values (`Code`, `Skill`, `Hybrid`) remain representable in live run metadata.
- **Alternative paths:**

  - Unknown project type is only acceptable when classification genuinely cannot be determined.
- **Use cases:** UC-03, UC-04

## Safety, Cleanup, and Documentation Fidelity

This section covers the repository's fallback and support surfaces. These behaviors often execute only when something is missing or unusual, which makes them easy to under-test and easy to rationalize away. That is precisely why they need explicit requirements here.

### REQ-005: Cleanup paths must only terminate playbook-owned processes

- **Tier:** 3
- **Functional section:** Safety, Cleanup, and Documentation Fidelity
- **Source type:** code-derived
- **Summary:** Operator cleanup must remain scoped to recorded run workers and must not widen into generic agent-command pattern kills.
- **User story:** As a benchmark operator on a shared machine, I expect recovery actions to stop the stuck run I own **so that** unrelated Claude or Copilot sessions are never collateral damage.
- **Implementation note:** `kill_recorded_processes()` is safe when PID files exist; the risk is the `_pkill_fallback()` branch (`bin/run_playbook.py:2807-2875`).
- **Conditions of satisfaction:**
  1. Cleanup reads recorded PID files and terminates only those PIDs when they exist.
  2. Missing PID files do not trigger a workstation-wide pattern kill across generic agent command lines.
  3. Any degraded cleanup path reports that manual intervention is required rather than guessing at ownership.
- **Alternative paths:**
  - Multiple per-parent PID files may coexist and still remain run-scoped.
  - Partial or malformed PID files may skip bad entries, but they must not expand scope to unrelated processes.
- **Use cases:** UC-06
- **References:** `bin/run_playbook.py:2807-2875`

### REQ-006: Every installed citable-doc workflow must retain byte-equality citation verification

- **Tier:** 3
- **Functional section:** Safety, Cleanup, and Documentation Fidelity
- **Pattern:** parity
- **Source type:** code-derived
- **Summary:** If the repository advertises byte-verified citation excerpts, installed targets must receive the verifier dependency needed to enforce that contract.
- **User story:** As an auditor, I expect repo-root and installed runs to apply the same Layer-1 citation checks **so that** citation-backed findings do not become weaker after installation.
- **Implementation note:** `quality_gate.py` soft-imports `bin.citation_verifier`, and the current install recipe copies the gate without that helper (`.github/skills/quality_gate/quality_gate.py:32-44`; `repos/setup_repos.sh:195-200`).
- **Conditions of satisfaction:**
  1. Installed targets include the verifier code needed for `quality_gate.py` to perform byte-equality citation checks.
  2. Repo-root and installed gate executions apply the same citation-verification behavior for the same cited artifact.
  3. Missing verifier support is treated as an installation defect rather than a silent weakening of the citation guarantee.
- **Alternative paths:**
  - Standalone installs may bundle helper modules in a different location, but the gate must still import them successfully.
  - Targets with zero citable docs may skip actual excerpt checks, but the install surface must remain capable of running them.
- **Use cases:** UC-05
- **References:** `README.md:39-42`; `bin/citation_verifier.py:1-14`; `.github/skills/quality_gate/quality_gate.py:32-44`; `repos/setup_repos.sh:195-200`

### REQ-007: Bootstrap self-audit docs must remain current with the live skill version and terminology

- **Tier:** 3
- **Functional section:** Safety, Cleanup, and Documentation Fidelity

- **Summary:** Curated bootstrap docs may compress context, but they must not present an older release or obsolete documentation path as the current product surface.
- **User story:** As a maintainer self-auditing QPB, I expect the curated bootstrap packet to describe the same current release I am auditing **so that** Phase 1 does not derive stale requirements from outdated guidance.

  1. Curated bootstrap docs that name the current release use the live root skill version.
  2. Curated bootstrap docs describing documentation intake use the modern `reference_docs/` terminology when speaking about the current workflow.
  3. Supplemental docs that intentionally describe historical behavior label that history explicitly instead of presenting it as current.
- **Alternative paths:**
  - Historical or archived material may preserve older terminology when clearly scoped as historical.
  - Supplemental docs may omit version claims entirely when they are intentionally timeless.
- **Use cases:** UC-02
- **References:** `docs_gathered/01_README_project.md:1-29`; `docs_gathered/29_improvement_axes_and_version_history.md:68-72`; `README.md:1-5`; `SKILL.md:1-13`

### REQ-008: Direct bootstrap and harness-installed bootstrap must expose the same documentation surface

- **Tier:** 3
- **Functional section:** Safety, Cleanup, and Documentation Fidelity
- **Pattern:** parity
- **Source type:** code-derived
- **Summary:** Running the playbook directly at the repo root and running it through the benchmark harness must expose the same effective documentation tree to Phase 1.
- **User story:** As a maintainer comparing direct-root and harness-installed self-audits, I expect both paths to review the same documentary evidence **so that** bug yield differences reflect model behavior, not different inputs.
- **Implementation note:** The harness mirrors `docs_gathered/` into `reference_docs/`, but the current repo-root layout leaves those trees different (`repos/setup_repos.sh:202-220`; `docs_gathered/30_benchmark_protocol_and_self_audit.md:87-99`).
- **Conditions of satisfaction:**
  1. Direct-root bootstrap and harness-installed bootstrap expose the same effective plaintext docs to `reference_docs_ingest`.
  2. Any curated docs intentionally used for self-audit are visible through the same path in both execution modes.
  3. Operators can compare repo-root and harness-installed runs without correcting for documentation-surface skew.
- **Alternative paths:**
  - The mirror may happen eagerly in the repo or lazily during setup, but the effective file set must match.
  - Empty-docs self-audit modes must declare themselves explicitly rather than accidentally differing by install path.
- **Use cases:** UC-01, UC-02
- **References:** `repos/setup_repos.sh:202-220`; `docs_gathered/30_benchmark_protocol_and_self_audit.md:87-99`; `docs_gathered/INDEX.md:1-37`
