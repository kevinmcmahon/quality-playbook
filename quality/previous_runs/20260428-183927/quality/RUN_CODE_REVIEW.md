## Pass 1: Structural Review

Read the code and report anything that looks wrong. No requirements, no focus areas beyond the guardrails.

### Guardrails

- **Line numbers are mandatory.**
- **Read function bodies, not just signatures.**
- **If unsure whether something is a bug or intentional, mark it QUESTION instead of BUG.**
- **Grep before claiming missing.**
- **Do not suggest style changes.**

### Required scrutiny areas

#### 1. Input validation and boundary handling

Inspect every trust boundary where the repo converts files, headings, versions, or paths into later-phase truth. Focus on:

- `citation_verifier.extract_excerpt()` locators and blank-anchor handling
- `check_phase_gate(..., "2")` heading enforcement
- `formal_docs_guard_banner()` vs `docs_present()` evidence detection

#### 2. Resource lifecycle

Inspect cleanup and install flows:

- `kill_recorded_processes()` and `_pkill_fallback()`
- `repos/setup_repos.sh` install/mirror steps
- any early-exit path that leaves stale live run metadata or half-installed gate dependencies

#### 3. Concurrency and state management

This repo has limited shared-state concurrency, but the run-state lifecycle still matters. Explicitly document:

- per-parent PID file behavior
- whether live `quality/INDEX.md` state can drift between stub/final writes
- whether parallel operators can interfere through cleanup or shared install artifacts

#### 4. Unit and encoding correctness

Check every place where the repo turns bytes or timestamps into published data:

- UTF-8 strict decode and newline normalization in `citation_verifier.py`
- timestamp / run-metadata formatting in `quality/INDEX.md`
- version-stamp handling across skill/docs/install surfaces

#### 5. Enumeration and whitelist completeness

Use the mechanical artifacts, not memory:

- `phase2_gate_required_headings.txt` vs `check_phase_gate()` enforcement
- `project_type_classifier_values.txt` vs `index_writer_project_type_literals.txt`
- `citation_install_copy_commands.txt` vs the install-time dependency set needed by `.github/skills/quality_gate/quality_gate.py`

For each authoritative item, mark it FOUND with evidence or NOT IN CODE.

### Project-specific scrutiny areas

2. Do the live index writers preserve the classifier's actual Code/Skill/Hybrid result?

### Evidence rule

Every REQ must end with one of:

- **SATISFIED** with at least one file:line citation
- **VIOLATED** with specific divergence evidence
- **PARTIALLY SATISFIED** with exact missing surface
- **NOT ASSESSABLE** only if the relevant artifact genuinely is not present

Do not group more than three REQs under one citation.
