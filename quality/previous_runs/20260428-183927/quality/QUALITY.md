## Purpose

Quality for the Quality Playbook is not "the Python tests still pass." It is preserving the repository's promised audit surface across prose, code, packaging, and bootstrap installs so that a maintainer, benchmark operator, or downstream AI agent sees the same contracts regardless of how the skill is invoked. The failure mode that matters most here is silent drift: the playbook says one thing, the runner or setup path does another, and later phases inherit a weaker or contradictory quality system without an obvious crash.

For this project, Deming means the bar has to live in persistent artifacts before anyone edits the runner, gate, or skill text. Juran's fitness-for-use test is whether the repository can still generate a trustworthy phase-by-phase playbook on itself and on installed targets: documentation intake must be accurate, gate contracts must be enforced by name, project-type metadata must survive to live artifacts, and citation guarantees must remain mechanically checkable after install. Crosby applies because every one of these drifts turns into a longer downstream bug hunt later: a "code-only" warning on a documented install, a malformed exploration file that still unlocks Phase 2, or an installed gate that quietly downgrades byte-equality checking to a warning.

## Coverage Targets

| Subsystem | Target | Why |
|-----------|--------|-----|
| `bin/run_playbook.py` phase gates, documentation guards, and live index writers | 95% | This file owns the highest-risk contract drift in the run: Phase 2 entry gating, documentation-source messaging, process cleanup, and run metadata all cross subsystem boundaries and can fail silently. |
| `.github/skills/quality_gate/quality_gate.py` and `bin/citation_verifier.py` | 90% | These modules are the anti-hallucination core. If citation verification or manifest validation softens unexpectedly, later phases can look authoritative while running on weakened evidence. |
| `repos/setup_repos.sh` install/mirroring logic | 90% | Install-time copying defines what downstream benchmark targets actually receive. Missing one support file here can degrade the shipped skill without changing any repo-root tests. |
| `SKILL.md`, `.github/skills/SKILL.md`, and `references/` resolution surface | 85% | The skill's own prose is product code in this repo. Entry-point drift or fallback-resolution skew changes what downstream agents are told to do. |
| `docs_gathered/` bootstrap/self-audit documents | 80% | These files are intentionally curated and therefore especially vulnerable to stale version/terminology drift that seeds bad requirements into future self-audits. |
| `bin/classify_project.py` project-type classifier | 80% | The classifier is only useful if later artifacts preserve its result. Misclassification suppresses Hybrid/Skill-specific reasoning downstream. |

## Coverage Theater Prevention

1. A test that only checks that `quality/INDEX.md` exists is theater for this repo. The important property is whether the JSON payload preserves the right project type, phases, and artifact inventory.
2. Treating `reference_docs/` and `docs_gathered/` as equivalent without asserting the exact preflight messaging is theater; this run found a specific drift where the modern ingest path and the legacy warning path disagree.
3. A citation test that only confirms `quality_gate.py` imports is theater if it never proves the installed package can still load `bin/citation_verifier` or byte-verify excerpts.
4. Comparing only version strings while ignoring terminology drift (`docs_gathered/` vs `reference_docs/`) is theater; stale bootstrap docs can still mis-seed Phase 1 even when a version number looks plausible.
5. Mocking away process cleanup without checking the no-PID fallback behavior is theater; the real risk is that a shared workstation cleanup path kills unrelated agent sessions.

## Fitness-to-Purpose Scenarios

### Scenario 1: Modern documentation installs are still warned as code-only

**Requirement tag:** [Req: inferred — REQ-002]

**What happened:** `formal_docs_guard_banner()` already treats `reference_docs/` as the canonical Phase 1 documentation surface, but `docs_present()` in `bin/run_playbook.py:1552-1559` still looks only at `docs_gathered/`, and that legacy result is used by later execution paths. A target can therefore ship real `reference_docs/` input and still be told it is doing "code-only" analysis. The run does not crash; it simply weakens operator trust in the evidence baseline.

**The requirement:** Phase 1 preflight and operator-facing warnings must use `reference_docs/` as the authoritative documentation signal on modern installs.

**How to verify:** Run `python3 -m unittest quality.test_functional.SpecRequirementTests.test_req_002_reference_docs_is_authoritative_for_phase1`.

### Scenario 2: A malformed exploration file still unlocks downstream artifact generation

**Requirement tag:** [Req: inferred — REQ-003]

**What happened:** The written Phase 2 contract in `SKILL.md:1026-1042` enumerates exact required headings and content checks, but `check_phase_gate()` currently reduces the gate to file existence plus a 120-line threshold (`bin/run_playbook.py:1158-1165`). A session can therefore hand later phases a long but structurally invalid exploration file and still advance. The downstream artifacts look official while missing the evidence shape they were supposed to inherit.

**The requirement:** The executable Phase 2 gate must reject any exploration artifact missing the exact required headings and minimum content shape.

**How to verify:** Run `python3 -m unittest quality.test_functional.SpecRequirementTests.test_req_003_phase2_gate_requires_exact_sections`.

### Scenario 3: Hybrid targets collapse back to `Code` in live run metadata

**Requirement tag:** [Req: inferred — REQ-004]

**What happened:** `bin/classify_project.py` can classify a target as `Code`, `Skill`, or `Hybrid`, but both live index writers in `bin/run_playbook.py:1819-1831` and `1888-1895` still emit `"Code"`. The run does not fail; it simply publishes incorrect metadata that later tooling and reviewers may trust. That is especially dangerous in self-audits where the repo is explicitly a Hybrid surface.

**The requirement:** Once project type is known, live run metadata must preserve that exact classification across stub and final `quality/INDEX.md` renders.

**How to verify:** Run `python3 -m unittest quality.test_functional.SpecRequirementTests.test_req_004_live_index_preserves_project_type`.

### Scenario 4: Cleanup after a crash kills someone else's agent session

**Requirement tag:** [Req: inferred — REQ-005]

**What happened:** `kill_recorded_processes()` correctly prefers per-parent PID files, but when none exist it falls back to `_pkill_fallback()` with broad patterns like `gh copilot -p` and `claude -p` (`bin/run_playbook.py:2836-2865`). On a shared workstation, a single recovery action can terminate unrelated sessions whose only "crime" is matching the same command-line substring. The operator sees successful cleanup; another user loses work.

**The requirement:** Cleanup paths must stay run-scoped even when PID files are missing; absence of recorded workers must not trigger workstation-wide pattern kills.

**How to verify:** Run `python3 -m unittest quality.test_functional.SpecRequirementTests.test_req_005_cleanup_without_pid_files_stays_run_scoped`.

### Scenario 5: Installed citation workflows silently lose byte-equality enforcement

**Requirement tag:** [Req: inferred — REQ-006]

**What happened:** The repo-root gate imports `bin.citation_verifier`, but the benchmark install path currently copies only `quality_gate.py` (`repos/setup_repos.sh:195-200`). In that installed shape, the gate warns and skips byte-equality verification instead of enforcing the README's advertised guarantee. The resulting run still emits citation-shaped artifacts, but the strongest Layer-1 anti-hallucination check has been downgraded off the operator's critical path.

**The requirement:** Every installed citable-doc workflow must include the verifier dependency needed to keep byte-equality citation enforcement live.

**How to verify:** Run `python3 -m unittest quality.test_functional.SpecRequirementTests.test_req_006_installed_gate_retains_citation_verifier`.

### Scenario 6: Bootstrap docs teach an older product than the repo actually ships

**Requirement tag:** [Req: inferred — REQ-007]

**What happened:** The curated bootstrap docs remain valuable context, but several files still describe older releases and older terminology while the repo-root `README.md` and `SKILL.md` are already at v1.5.3. A self-audit that trusts the curated pack can therefore derive requirements from stale claims about the current version or the canonical docs path. Nothing explodes; the audit simply starts from the wrong brief.

**The requirement:** Curated bootstrap/self-audit docs must stay current with the live skill version and the modern `reference_docs/` terminology whenever they describe the current release.

**How to verify:** Run `python3 -m unittest quality.test_functional.SpecRequirementTests.test_req_007_bootstrap_docs_match_live_version_and_terms`.

### Scenario 7: Direct-root bootstrap and harness installs audit different evidence trees

**Requirement tag:** [Req: inferred — REQ-008]

**What happened:** The harness mirrors `docs_gathered/` into `reference_docs/`, but the repo-root self-audit path currently leaves `reference_docs/` empty while `docs_gathered/` stays populated. The same repository therefore presents different documentation surfaces depending on whether it is audited directly or through the benchmark harness. The discrepancy is subtle because both paths still "have docs"—just not the same docs.

**The requirement:** Direct bootstrap and harness-installed bootstrap must expose the same effective documentation surface to Phase 1.

**How to verify:** Run `python3 -m unittest quality.test_functional.SpecRequirementTests.test_req_008_direct_and_harness_bootstrap_share_docs_surface`.

### Scenario 8: Installed skill entry points diverge and downstream agents read different instructions

**Requirement tag:** [Req: inferred — REQ-001]

**What happened:** The repo ships both `SKILL.md` and `.github/skills/SKILL.md`, and fallback resolution depends on either one being safe to read. If those entry points drift, two equally valid install paths can teach different instructions, version stamps, or reference-file expectations. That sort of packaging split is hard to notice because each file still looks complete in isolation.

**The requirement:** The installed skill entry point must stay synchronized with the repo-root skill so any documented fallback path yields the same instructions.

**How to verify:** Run `python3 -m unittest quality.test_functional.SpecRequirementTests.test_req_001_skill_entry_points_are_synchronized`.

## AI Session Quality Discipline

1. Read `quality/EXPLORATION.md`, this file, and `quality/REQUIREMENTS.md` before changing runner, gate, install, or bootstrap-doc surfaces.
2. Treat Markdown skill/docs files as product code whenever they define operator-visible contracts.
3. When a change touches a closed set (required headings, project types, install files), preserve the full set mechanically rather than by memory.
4. Prefer executable evidence over prose assertions whenever a claim depends on code presence, copied files, or exact line content.
5. Update this file whenever a new cross-surface drift mode is discovered.

## The Human Gate

- Deciding whether legacy `docs_gathered/` messaging should remain visible at all on modern installs.
- Choosing the long-term product stance on standalone `quality_gate.py` installs versus bundled helper modules.
- Any intentional backward-compatibility tradeoff for process cleanup semantics on shared hosts.
- Deciding which curated bootstrap docs deserve promotion into canonical `reference_docs/` material rather than remaining supplemental.
