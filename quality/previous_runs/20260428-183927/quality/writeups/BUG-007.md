## Summary
The direct-root bootstrap path leaves the canonical `reference_docs/` tree effectively empty while the harness install mirrors the curated docs packet into that tree, so Phase 1 sees different documentation surfaces depending on how QPB is invoked.

## Spec reference
- Requirement: REQ-008
- Spec basis: `quality/REQUIREMENTS.md:247-264`
- Behavioral contract quote: "Direct-root bootstrap and harness-installed bootstrap expose the same effective plaintext docs"

## The code
At `repos/setup_repos.sh:216-220`, harness installs copy `docs_gathered/` into `reference_docs/`. In contrast, `quality/PROGRESS.md:25-28` records that the repo-root `reference_docs/` tree contains only `.gitkeep`, while `docs_gathered/INDEX.md:1-37` shows the curated plaintext bootstrap packet that Phase 1 actually needs. The two execution modes therefore expose different effective doc trees to the same playbook.

## Observable consequence
Running QPB directly at the repo root yields a weaker canonical docs surface than running it through the harness installer. The immediate symptom is that direct-root self-audits fall back to an effectively empty `reference_docs/` tree while installed targets receive the curated bootstrap packet through the canonical path.

## The fix
Mirror curated bootstrap docs into `reference_docs/` for direct-root bootstrap runs before the Phase 1 docs-presence checks execute.
