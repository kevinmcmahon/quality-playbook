## Summary
`docs_present()` in `bin/run_playbook.py` still treats a repo with only populated `reference_docs/` input as docs-missing, so the Phase 1 runner emits the legacy code-only warning on a modern install.

## Spec reference
- Requirement: REQ-002
- Spec basis: `bin/run_playbook.py:621-630`; `bin/run_playbook.py:1582-1622`; `quality/REQUIREMENTS.md:130-147`
- Behavioral contract quote: "A populated `reference_docs/` tree suppresses any \"code-only\" warning path."

## The code
At `bin/run_playbook.py:1552-1559`, `docs_present()` looks only at `docs_gathered/` and returns `False` when that legacy directory is empty. The same file's Phase 1 prompt and reference-doc guard code already describe `reference_docs/` as the authoritative modern documentation surface, so this helper reports the wrong documentation-strength signal for the current contract.

## Observable consequence
When an operator runs the playbook against a repo that provides documentation only through `reference_docs/`, the runner still behaves as if the repo were code-only. The immediate symptom is the misleading "docs_gathered/ is missing or empty; proceeding with code-only analysis" warning even though usable modern docs are present.

## The fix
Teach `docs_present()` to recognize canonical plaintext under `reference_docs/` before falling back to legacy `docs_gathered/` detection.
