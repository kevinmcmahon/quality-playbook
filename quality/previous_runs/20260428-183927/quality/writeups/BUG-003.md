# Bug Writeup: BUG-003
> Generated by [Quality Playbook](https://github.com/andrewstellman/quality-playbook) v1.5.3 — Andrew Stellman
> Date: 2026-04-28 · Project: quality-playbook

## Summary
The live `quality/INDEX.md` write path in `write_live_index_stub()` / `write_live_index_final()` collapses valid `Skill` and `Hybrid` classifications back to `Code`, so the published run metadata lies about the target project type.

## Spec reference
- Requirement: REQ-004
- Spec basis: `bin/classify_project.py:261-286`; `quality/REQUIREMENTS.md:169-186`
- Behavioral contract quote: "that exact `Code` / `Skill` / `Hybrid` value must survive"

## The code
At `bin/run_playbook.py:1819-1831`, the stub INDEX payload hardcodes `"target_project_type": "Code"`. At `bin/run_playbook.py:1888-1895`, the final INDEX render passes `target_project_type="Code"` again. Those two sites overwrite whatever `bin/classify_project.py` actually recorded for the run.

## Observable consequence
When the target is explicitly classified as `Skill` or `Hybrid`, both live INDEX renders still publish `Code`. Reviewers and downstream tooling therefore see the wrong project-type metadata even though the classifier already derived the more specific value.

## The fix
Read the recorded classification and thread it into both live INDEX writers instead of hardcoding `Code`.
