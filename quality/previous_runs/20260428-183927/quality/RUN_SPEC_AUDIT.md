**Requirement confidence tiers:**

- **formal** — authoritative cited project spec
- **user-confirmed** — explicit user instruction
- **inferred** — derived from explored code behavior

All requirements in this run are inferred Tier 3 requirements. Findings against them must be reported as **NEEDS REVIEW** until triage verifies them.

**Rules:**

- Only list defects.
- Every defect must cite a file and line number.
- Before claiming something is missing, grep the code/docs surface.
- Before claiming something exists, read the actual function body or document lines.
- Classify each finding: MISSING / DIVERGENT / UNDOCUMENTED / PHANTOM.
- Because this run has no Tier 1/2 REQs, do not pretend a formal citation exists.

## Pre-audit docs validation

This section is required in the triage report.

### Baseline for this run

- `reference_docs/` is effectively empty in the current checkout, so there are no Tier 1/2 formal-doc citations in scope.
- `docs_gathered/` exists and is useful supplemental context, but exploration already flagged version/terminology drift relative to root `README.md` and `SKILL.md`.

### Required spot-checks

Before trusting `docs_gathered/` claims, spot-check at least these:

1. Any statement describing the current live release version
2. Any statement describing the canonical documentation intake path
3. Any statement describing installed citation-verification behavior

Format each spot-check as:

```text
Claim: "..."
Actual lines: ...
Result: CLAIM IS TRUE / CLAIM IS FALSE / NEEDS REVIEW
```

If a claim is false, downgrade any dependent finding to NEEDS REVIEW until a verification probe confirms it directly from code.

## Running the Audit

1. Give the identical prompt to three independent models.
2. Keep the reports isolated.
3. Save reports to `quality/spec_audits/YYYY-MM-DD-[model].md`.

## Triage Process

1. Record council status (`fresh report`, `timeout`, etc.).
2. Merge overlapping findings and label auditor agreement (`3/3`, `2/3`, `1/3`).
3. For any disagreement about factual code presence, deploy a verification probe.

### Verification probes must produce executable evidence

Use assertions, not prose.

Examples for this repo:

```python
assert "## Open Exploration Findings" in Path("quality/EXPLORATION.md").read_text(encoding="utf-8")
assert '"target_project_type": "Hybrid"' in Path("bin/run_playbook.py").read_text(encoding="utf-8")
assert "citation_verifier" in Path("repos/setup_repos.sh").read_text(encoding="utf-8")
```

If you cannot write a mechanical assertion proving a rejection, do not reject the finding.

## Categorize Each Confirmed Finding

- **Real code bug** — executable implementation violates the inferred REQ
- **Documentation gap** — code is fine, docs are stale or misleading
- **Design decision** — human judgment needed
- **Missing test** — requirement is real but not yet exercised
- **Inferred requirement wrong** — fix `quality/REQUIREMENTS.md` rather than the code

## Cross-artifact consistency check

After triage, compare confirmed findings against:

- `quality/RUN_CODE_REVIEW.md` outputs
- `quality/mechanical/*.txt`
- `quality/test_functional.py`

Any contradiction about what the code actually contains must be resolved with a verification probe before closing the triage.
