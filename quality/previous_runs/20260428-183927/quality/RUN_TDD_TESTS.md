> Generated by [Quality Playbook](https://github.com/andrewstellman/quality-playbook) v1.5.3 — Andrew Stellman

## Working Directory

Run all commands from the repository root.

## Purpose

Every confirmed BUG from Phase 3 or Phase 4 must go through an explicit red/green cycle. For this repo, the TDD protocol has to prove both code-path defects and product-surface defects (skill/docs/install drift), so executable evidence may come from Python tests, mechanical extraction diffs, or both.

## Pre-Flight Check

- [ ] `quality/test_regression.py` exists
- [ ] `quality/BUGS.md` exists
- [ ] `mkdir -p quality/results quality/writeups quality/patches`
- [ ] `python3 -m unittest discover bin/tests`
- [ ] `python3 -m unittest discover -s .github/skills/quality_gate/tests/ -p 'test_*.py'`
- [ ] `bash quality/mechanical/verify.sh`

## Per-BUG red/green cycle

For each confirmed bug:

1. Identify the regression test in `quality/test_regression.py`.
2. Remove or disable the skip/expected-failure guard.
3. Run the red phase against unpatched code and save output to `quality/results/BUG-NNN.red.log`.
4. Apply the fix patch (if one exists).
5. Re-run the test for the green phase and save output to `quality/results/BUG-NNN.green.log`.
6. Restore the skip guard if the fix is not being committed in the working tree.
7. Write `quality/writeups/BUG-NNN.md`.

## JUnit XML capture

For each red/green execution, also emit JUnit XML via pytest's native reporter:

```bash
pytest quality/test_regression.py -q --junitxml quality/results/BUG-NNN.red.junit.xml -k 'test_bug_nnn'
pytest quality/test_regression.py -q --junitxml quality/results/BUG-NNN.green.junit.xml -k 'test_bug_nnn'
```

Keep the human-readable `.red.log` / `.green.log` receipts as the primary narrative artifacts, but always produce the XML sidecars as well.

## Patch validation gate

Before applying any patch, validate it in a temporary worktree or another reversible scratch copy of the repo:

```bash
git worktree add /tmp/qpb-phase5-check HEAD
cd /tmp/qpb-phase5-check
git apply --check quality/patches/BUG-NNN-regression-test.patch
git apply --check quality/patches/BUG-NNN-fix.patch
python3 -m py_compile bin/*.py .github/skills/quality_gate/*.py
python3 -m unittest discover bin/tests
python3 -m unittest discover -s .github/skills/quality_gate/tests/ -p 'test_*.py'
```

If you cannot use a temporary worktree, use a stash-and-revert equivalent that preserves the source-boundary rule. Do not apply a patch blindly to the main working tree without first passing `git apply --check`.

## Command matrix

| Bug shape | Red phase | Green phase |
|-----------|-----------|-------------|
| Runner/gate Python defect | `pytest quality/test_regression.py -q --junitxml quality/results/BUG-NNN.red.junit.xml -k 'test_bug_nnn'` plus `python3 -m unittest quality.test_regression.CodeReviewRegressionTests.test_bug_nnn_*` for the saved `.red.log` receipt | same commands after applying fix, writing `BUG-NNN.green.junit.xml` / `.green.log` |
| Install-shape / packaging defect | `pytest quality/test_regression.py -q --junitxml quality/results/BUG-NNN.red.junit.xml -k 'test_bug_nnn'` plus `python3 -m unittest quality.test_regression.CodeReviewRegressionTests.test_bug_nnn_*` and `bash quality/mechanical/verify.sh` when the bug changes extraction receipts | same commands after applying fix |
| Bootstrap-doc drift defect | `pytest quality/test_regression.py -q --junitxml quality/results/BUG-NNN.red.junit.xml -k 'test_bug_nnn'` plus `python3 -m unittest quality.test_regression.CodeReviewRegressionTests.test_bug_nnn_*` | same command after doc fix |
| Cross-surface parity defect | red regression test plus any relevant baseline suite (`bin/tests` or gate suite) | same after patch |

## Result recording

After all bugs are processed, write `quality/results/tdd-results.json` with:

| Field | Required value / shape |
|-------|------------------------|
| `schema_version` | `"1.1"` |
| `skill_version` | `"1.5.3"` |
| `date` | `YYYY-MM-DD` |
| `project` | `"quality-playbook"` |
| `bugs` | array |
| `bugs[].id` | `BUG-NNN` |
| `bugs[].requirement` | `REQ-NNN` |
| `bugs[].red_phase` | `"fail"` or `"pass"` (red should normally be `fail`) |
| `bugs[].green_phase` | `"pass"`, `"fail"`, or `"skipped"` |
| `bugs[].verdict` | `"TDD verified"`, `"red failed"`, `"green failed"`, `"confirmed open"`, or `"deferred"` |
| `bugs[].fix_patch_present` | boolean |
| `bugs[].writeup_path` | existing `quality/writeups/BUG-NNN.md` path |
| `summary.total` | integer |
| `summary.verified` | integer |
| `summary.confirmed_open` | integer |
| `summary.red_failed` | integer |
| `summary.green_failed` | integer |

## Phase-specific notes for this repo

1. Bugs touching `quality_gate.py` should usually re-run `python3 -m unittest discover -s .github/skills/quality_gate/tests/ -p 'test_*.py'`.
2. Bugs touching `bin/run_playbook.py`, `bin/classify_project.py`, or `bin/citation_verifier.py` should usually re-run `python3 -m unittest discover bin/tests`.
3. Bugs about packaging/install parity should also re-run `bash quality/mechanical/verify.sh` if they change any saved extraction artifact.
4. Bugs about stale bootstrap docs should include before/after citations in the writeup, even when the executable test is a file-content assertion.

## Post-write validation

After writing `tdd-results.json`, reopen it and confirm:

1. required root keys are present,
2. every bug row references an existing writeup,
3. `verdict`, `red_phase`, and `green_phase` use allowed enum values,
4. the summary contains `total`, `verified`, `confirmed_open`, `red_failed`, and `green_failed`,
5. no undocumented root keys exist.
