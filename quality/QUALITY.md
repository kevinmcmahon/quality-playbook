# Quality Constitution: Quality Playbook (Bootstrap Self-Audit)

## Purpose

Quality in this project is defined by three principles:

**Deming ("quality is built in, not inspected in"):** The quality playbook's own quality is built into the specification text, the reference files, and the validation script — not inspected after the fact. Every instruction in SKILL.md must be internally consistent and mechanically verifiable. When SKILL.md says "quality_gate.sh will check for X," that check must actually exist. The gate is the inspection layer, but the quality is in the specification's self-consistency.

**Juran ("fitness for use"):** Fitness for this project means: an AI agent reading SKILL.md can execute the playbook end-to-end on an arbitrary codebase and produce conformant artifacts that pass quality_gate.sh without human intervention. Every instruction must be unambiguous — if two canonical examples disagree, the agent cannot produce correct output. Every path reference must resolve — if the skill says "read `.github/skills/references/X.md`" but the skill is installed at `.claude/skills/quality-playbook/`, the agent fails. Fitness is not "the spec looks right" but "an agent following the spec succeeds."

**Crosby ("quality is free"):** Maintaining internal consistency costs less than debugging agent failures caused by specification drift. A 10-minute review of cross-references after editing SKILL.md prevents hours of debugging when agents produce non-conformant output because they followed a contradictory example.

## Coverage Targets

| Subsystem | Target | Why |
|-----------|--------|-----|
| SKILL.md (specification) | 95% — all internal cross-references, JSON schemas, phase numbering, artifact lists | Because two contradictory JSON schema examples (lines 127 vs 1384) mean an agent following one will produce output that contradicts the other. Every cross-reference and schema template must be consistent. |
| quality_gate.sh (validation) | 90% — all validation checks, enum lists, threshold logic | Because the gate is the mechanical proof that artifacts conform. Under-validation (checking 6 of 27 required artifacts) means the gate passes non-conformant runs. Every claim SKILL.md makes about gate enforcement must be true. |
| references/ (phase references) | 85% — cross-references to SKILL.md, field lists, template consistency | Because agents read these files during execution. A reference file that lists different required fields than SKILL.md produces inconsistent artifacts. |

## Coverage Theater Prevention

The following patterns look like quality checks but do not catch real bugs in this project:

- **Checking that files exist without checking their content.** quality_gate.sh checks that BUGS.md exists but does not verify it contains actual bug descriptions. A zero-byte BUGS.md passes.
- **Checking that sections exist without checking their content.** The Phase 2 entry gate checks that EXPLORATION.md has specific section headings but does not verify the sections contain substantive findings.
- **Validating JSON field names without validating field values.** quality_gate.sh checks that tdd-results.json contains `"verdict"` keys but only checks whether the values match a known enum — it does not check whether the verdict is consistent with the red_phase/green_phase values (e.g., `verdict: "TDD verified"` with `red_phase: "pass"` should be impossible but is not rejected).
- **Counting artifacts without checking traceability.** Checking that 5 UC identifiers exist in REQUIREMENTS.md does not verify they appear in integration-results.json's uc_coverage — a gap the current gate does not detect.
- **Version stamp presence without version stamp correctness.** The gate checks that PROGRESS.md has a "Skill version:" line matching SKILL.md, but does not check that every generated Markdown file has the version banner.

## Fitness-to-Purpose Scenarios

### Scenario 1: Contradictory JSON Schema Templates Produce Non-Conformant Agent Output

**Requirement tag:** [Req: formal — SKILL.md lines 127, 1384, 1394]

**What happened:** SKILL.md contains two canonical examples of tdd-results.json. Example 1 (line 127) uses `"tdd_verified": 2` in the summary. Example 2 (line 1384) uses `"verified": 4`. The normative text (line 1394) says the canonical key is `verified`. An agent faithfully copying Example 1 produces a summary with `tdd_verified` — a key that the normative text does not recognize. quality_gate.sh (line 229) checks for neither `tdd_verified` nor `verified`, so the contradiction is invisible to the mechanical gate. The result: agents produce inconsistent JSON across runs, downstream tooling that reads `verified` gets no data from runs that used `tdd_verified`, and the gate provides false assurance of conformance.

**The requirement:** All canonical examples of tdd-results.json in SKILL.md must use identical summary field names matching the normative text. quality_gate.sh must validate the `total` and `verified` summary keys.

**How to verify:** Grep SKILL.md for all occurrences of `tdd_verified` and `verified` within JSON examples. Assert that no example uses `tdd_verified`. Grep quality_gate.sh for summary key validation and assert that `total` and `verified` are checked.

### Scenario 2: Hardcoded Reference Paths Fail at Claude Code Installation

**Requirement tag:** [Req: formal — SKILL.md lines 872–877, docs_gathered/readme.md lines 26–30]

**What happened:** SKILL.md Phase 2 instructions say "Read `.github/skills/references/requirements_pipeline.md`" (line 872). But when the skill is installed at `.claude/skills/quality-playbook/SKILL.md`, the reference files are at `.claude/skills/quality-playbook/references/requirements_pipeline.md`. The path `.github/skills/references/requirements_pipeline.md` does not exist. An agent running from a Claude Code installation will attempt to read a nonexistent file at the start of every phase that requires a reference. The orchestrator agents correctly check multiple SKILL.md locations (three paths each), but SKILL.md itself does not adapt its internal reference paths — it hardcodes a single path. This means every Claude Code user's first experience is a reference-not-found error.

**The requirement:** Every reference file path in SKILL.md must resolve correctly regardless of installation location. Use relative paths (`references/X.md`) or include path resolution logic.

**How to verify:** Grep SKILL.md for `.github/skills/references/`. Count occurrences. Assert count is 0 (all references use relative paths) or each occurrence includes a fallback.

### Scenario 3: Gate Under-Validation Passes Non-Conformant Runs

**Requirement tag:** [Req: formal — SKILL.md line 72, quality_gate.sh lines 107–108]

**What happened:** The artifact contract table (SKILL.md lines 72–101) lists 27 artifacts. Of these, 7 are marked "Required: Yes" but are NOT checked by quality_gate.sh's file existence validation: CONTRACTS.md (line 78), test_functional.* (line 79, only warned not failed), RUN_CODE_REVIEW.md (line 81), RUN_INTEGRATION_TESTS.md (line 82), RUN_SPEC_AUDIT.md (line 83), RUN_TDD_TESTS.md (line 84), and AGENTS.md (line 89). Additionally, EXPLORATION.md is required by the Phase 1 completion gate but is not in the artifact contract table and not checked by the mechanical gate. An agent could skip Phase 1 entirely, produce no EXPLORATION.md, omit CONTRACTS.md, skip all four RUN_*.md protocols, and still pass quality_gate.sh. The gate principle (line 72) says "If the gate checks for it, this skill must instruct its creation" — but the converse is also necessary: if the skill marks it as Required, the gate must check for it.

**The requirement:** quality_gate.sh file existence checks must cover all artifacts marked "Required: Yes" in the artifact contract table.

**How to verify:** Extract the list of "Required: Yes" artifacts from SKILL.md lines 76–101. For each, grep quality_gate.sh for a file existence check. Assert all are present.

### Scenario 4: Phase 7 Unreachable from Standard Flow

**Requirement tag:** [Req: formal — SKILL.md lines 12–31, 1797, 1890]

**What happened:** The plan overview (lines 12–31) describes Phases 0–6. Phase 7 ("Present, Explore, Improve") is defined at line 1890 but does not appear in the plan overview, the Phase completion checklist (line 688), the end-of-Phase-6 message (line 1838), or either orchestrator agent. The convergence check at line 1797 says "skip to Phase 7" — referencing a phase that the execution roadmap does not include. An agent following the plan overview will execute Phases 0–6, see the STOP message at end of Phase 6, and never discover Phase 7 exists. If Phase 7 is intentional, it is unreachable. If it is legacy, the convergence check reference creates confusion.

**The requirement:** Every phase referenced in cross-references must appear in the plan overview. Phase 7 must either be added to the plan overview and orchestrators, or its cross-references must be removed.

**How to verify:** Grep SKILL.md for "Phase 7" and "phase 7". For each occurrence, check whether it appears in lines 12–31 (plan overview). Assert consistency.

### Scenario 5: AGENTS.md Omits Claude Code Installation Path

**Requirement tag:** [Req: formal — AGENTS.md lines 27–30, docs_gathered/readme.md lines 26–30]

**What happened:** AGENTS.md — the "read this first" file for AI agents working on the repository — contains installation instructions that only show the `.github/skills/` path (Copilot). The README documents both `.github/skills/` and `.claude/skills/quality-playbook/` paths. A Claude Code user whose agent reads AGENTS.md first (as designed) will install the skill at `.github/skills/` — which works for Copilot but uses a non-standard path for Claude Code. The agent will then find SKILL.md but the orchestrator's path-finding logic will not match the expected `.claude/skills/quality-playbook/` layout.

**The requirement:** AGENTS.md installation instructions must cover all supported platforms, matching the README.

**How to verify:** Grep AGENTS.md for `.claude/skills`. Assert at least one occurrence showing the Claude Code installation path.

### Scenario 6: Artifact Contract Table Phase Assignment Contradicts Instructions

**Requirement tag:** [Req: formal — SKILL.md lines 97–98, 531, 555]

**What happened:** The artifact contract table (line 97) says `quality/mechanical/verify.sh` is created in "Phase 5". But SKILL.md line 531 instructs creation during the requirements derivation (which occurs in Phase 1/2's step 7). Line 555 has the "Immediate integrity gate (mandatory, Phase 2a)" that requires verify.sh to run before writing any contract. An agent consulting the artifact table will expect to create verify.sh in Phase 5. But the Phase 2a instructions demand it already exists. The agent will either fail the Phase 2a gate (because verify.sh does not exist yet) or skip the gate (because the table says verify.sh is a Phase 5 artifact).

**The requirement:** The "Created In" column must match the earliest phase where the instructions direct creation.

**How to verify:** Read SKILL.md lines 97–98, 531, and 555. Assert that the "Created In" value for verify.sh matches the phase of line 531/555.

## AI Session Quality Discipline

1. Read QUALITY.md before starting work on the skill.
2. After editing SKILL.md, grep for all cross-references to the edited section. Verify consistency.
3. After editing a JSON schema example, verify ALL examples use the same field names as the normative text.
4. After adding a gate check to quality_gate.sh, verify the artifact contract table lists the artifact and the "Required" column is correct.
5. After editing reference file paths, verify the path works at all four supported installation locations.
6. Never remove a fitness-to-purpose scenario. Only add new ones.

## The Human Gate

- **Phase 7 disposition:** Whether Phase 7 should be retained, removed, or made optional requires design judgment about the interactive improvement workflow.
- **Strictness mode thresholds:** Whether quality_gate.sh's `--general` mode should fail or warn on missing triage evidence requires judgment about the target user audience.
- **Path resolution strategy:** Whether to use relative paths (simpler but fragile with symlinks) or multi-location fallback (more robust but more complex) requires architectural judgment.
- **Artifact table accuracy:** Whether the "Created In" column should reflect first creation or authoritative creation requires design intent clarification.
