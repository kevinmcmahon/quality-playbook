## Purpose

Quality for the Quality Playbook skill means that every AI agent reading SKILL.md and its reference files produces consistent, complete, and correct quality artifacts for any target codebase. This is Juran's "fitness for use" applied to a meta-level tool: the skill is fit for use when it reliably finds real bugs in real projects, and its mechanical validation (quality_gate.sh) accurately distinguishes passing runs from failing ones.

Following Deming, quality is built into the specification itself. SKILL.md encodes the quality bar directly in its phase gates, artifact contracts, and behavioral rules. Every AI session inherits this bar by reading the file, not by having a human explain it. The playbook's internal consistency IS its quality -- if the specification contradicts itself, agents produce contradictory artifacts.

Following Crosby, quality is free. The cost of maintaining consistent cross-references between SKILL.md, quality_gate.sh, and verification.md is far lower than the cost of debugging false-pass gates that let broken artifacts through. Each bug found in the specification prevents hundreds of downstream failures across every project the skill is applied to.

## Coverage Targets

| Subsystem | Target | Why |
|-----------|--------|-----|
| quality_gate.sh (validation logic) | 95% | This is the only executable code. A false-pass in the gate means broken artifacts ship to users. Every conditional branch must be exercised because each represents a different artifact conformance check. |
| SKILL.md (phase gates and artifact contracts) | 90% cross-reference consistency | Three independent documents define what's required (SKILL.md, quality_gate.sh, verification.md). Inconsistency between them means the spec promises something the gate doesn't enforce. Finding 6 and Risk 2 documented 10+ such inconsistencies. |
| SKILL.md (behavioral rules) | 85% testable-rule coverage | Rules like "every confirmed bug must have a regression-test patch" need mechanical enforcement. Rules that exist only as prose are aspirational, not enforced. |
| references/*.md (reference documents) | 80% internal consistency | Reference documents must be internally numbered correctly and consistent with SKILL.md. The Phase 6/Phase 3 numbering collision in review_protocols.md demonstrates the failure mode. |

## Coverage Theater Prevention

For this specification-primary repository, coverage theater takes specific forms:

- **Checking file existence without checking content.** quality_gate.sh checks that REQUIREMENTS.md exists but does not check that it contains requirements with REQ-NNN headings. A zero-byte REQUIREMENTS.md passes the existence check.
- **Checking section headings without checking section content.** A test that verifies EXPLORATION.md has the right section titles but doesn't verify the sections contain actual findings with file:line references.
- **Asserting gate passes without introducing failures.** A test that runs quality_gate.sh on a valid artifact set and checks exit 0 proves nothing -- it only confirms the happy path. The gate's value is in catching invalid artifacts. Tests must also run the gate on deliberately broken artifacts and verify it catches them.
- **Testing grep helpers on synthetic JSON without edge cases.** quality_gate.sh's json_has_key/json_str_val/json_key_count use grep-based parsing. Tests using clean JSON miss the failure modes: keys inside string values, multi-line values, escaped quotes.
- **Counting tests without checking assertions.** A test file with 20 functions that all run quality_gate.sh and check exit 0 is one test repeated 20 times, not 20 tests.

## Fitness-to-Purpose Scenarios

### Scenario 1: The Silent Gate Pass

**Requirement tag:** [Req: formal -- SKILL.md lines 83-85]

**What happened:** quality_gate.sh classifies CONTRACTS.md, RUN_CODE_REVIEW.md, RUN_SPEC_AUDIT.md, RUN_INTEGRATION_TESTS.md, RUN_TDD_TESTS.md, and test_functional.* as WARN-if-missing (lines 117-129), despite SKILL.md's artifact contract table marking all six as "Required: Yes." A playbook run that produces only the 6 FAIL-checked artifacts (BUGS.md, REQUIREMENTS.md, QUALITY.md, PROGRESS.md, COVERAGE_MATRIX.md, COMPLETENESS_REPORT.md) but none of the 6 WARN-checked artifacts passes the gate with exit 0. The user sees "GATE PASSED" and trusts the output, but half the required quality infrastructure is missing. This affects every project the skill is applied to -- the gate's false pass is systematic, not a one-time error.

**The requirement:** quality_gate.sh must use fail() (not warn()) for all artifacts marked "Required: Yes" in SKILL.md's artifact contract table (lines 86-115). The WARN severity should be reserved for optional or conditional artifacts.

**How to verify:** Create a test directory with only the 6 FAIL-checked artifacts. Run quality_gate.sh. It must exit 1, not 0.

### Scenario 2: The Confused Agent

**Requirement tag:** [Req: formal -- references/review_protocols.md line 378]

**What happened:** references/review_protocols.md defines three communication phases for the integration test execution UX: "Phase 1: The Plan" (line 344), "Phase 2: Progress" (line 364), and "Phase 6: Results" (line 376). The jump from Phase 2 to Phase 6 means an AI agent reading both SKILL.md and review_protocols.md simultaneously encounters "Phase 6" in two entirely different contexts -- the playbook's Phase 6 (Verification Benchmarks) and the integration test UX's Phase 6 (show results to user). Because AI agents follow instructions literally and SKILL.md has extensive Phase 6 content (self-check benchmarks, verification gate, quality_gate.sh execution), an agent could attempt to run verification benchmarks when the integration test protocol means "show the results table."

**The requirement:** Section numbering within reference documents must be internally sequential. The integration test execution UX phases must be numbered Phase 1, Phase 2, Phase 3.

**How to verify:** Grep review_protocols.md for "### Phase" headings in the Execution UX section. Verify they are sequential (1, 2, 3), not (1, 2, 6).

### Scenario 3: The Invisible Test File

**Requirement tag:** [Req: formal -- SKILL.md line 69]

**What happened:** SKILL.md (line 69) tells agents to name functional test files using the project's language conventions: `test_functional.py` for Python, `FunctionalSpec.scala` for Scala, `functional.test.ts` for TypeScript, `FunctionalTest.java` for Java. But quality_gate.sh (line 125) only checks `ls ${q}/test_functional.*` -- it would miss `FunctionalSpec.scala`, `FunctionalTest.java`, and `functional.test.ts`. A Java project where the agent correctly names the file `FunctionalTest.java` per SKILL.md's guidance would trigger a WARN ("test_functional.* missing") from the gate, despite having the test file SKILL.md asked for.

**The requirement:** quality_gate.sh's functional test detection must recognize all naming patterns that SKILL.md tells agents to use.

**How to verify:** Create test directories with each SKILL.md-specified test file name (FunctionalSpec.scala, FunctionalTest.java, functional.test.ts). Run quality_gate.sh on each. It must detect the file without warnings.

### Scenario 4: The Injection Vector

**Requirement tag:** [Req: inferred -- quality_gate.sh lines 439-448]

**What happened:** quality_gate.sh uses `eval "find '${repo_dir}' ..."` at line 439 to detect test file extensions. The `repo_dir` variable comes directly from command-line arguments (line 55). While single quotes inside the eval string provide some protection, a directory name containing a single quote breaks the quoting: `repo'$(id)'name` would execute the `id` command. This is a developer tool typically not exposed to untrusted input, but the pattern is unnecessary -- the find command could be executed directly without eval, eliminating the injection risk entirely.

**The requirement:** quality_gate.sh must not pass user-supplied arguments through `eval`. Use direct command execution with proper quoting.

**How to verify:** Create a directory with a shell-metacharacter name (containing `$()` or backticks). Run quality_gate.sh on it. Verify no command injection occurs.

### Scenario 5: The Missing AGENTS.md Check

**Requirement tag:** [Req: formal -- SKILL.md line 102]

**What happened:** SKILL.md's artifact contract table (line 102) lists AGENTS.md as "Required: Yes, Created In: Phase 2." But quality_gate.sh has no check for AGENTS.md at all -- it is not mentioned anywhere in the 676-line script. A playbook run that produces every other required artifact but skips AGENTS.md would pass the gate. Without AGENTS.md, the next AI session working on the project has no bootstrap context and wastes time figuring out what's going on -- exactly the problem AGENTS.md is designed to prevent.

**The requirement:** quality_gate.sh must check for AGENTS.md existence with fail() severity.

**How to verify:** Run quality_gate.sh on a directory that has all required artifacts except AGENTS.md. It must exit 1.

### Scenario 6: The Dangling Reference

**Requirement tag:** [Req: inferred -- quality_gate.sh line 116]

**What happened:** quality_gate.sh line 116 contains the comment "should not halt the skill if absent, per BUG-005." This references a bug ID that does not appear anywhere else in the repository. The comment explains WHY six required artifacts were downgraded from FAIL to WARN -- but without the referenced BUG-005 documentation, a maintainer cannot evaluate whether the downgrade is still justified. The reference is from a previous quality playbook run whose artifacts were not committed. This means a design decision affecting the gate's correctness is justified by evidence that no longer exists.

**The requirement:** Code comments referencing external artifacts (bug IDs, issue numbers) must either link to accessible documentation or include sufficient inline context to justify the decision without the external reference.

**How to verify:** Search the repository for "BUG-005". Verify it either resolves to documentation or the comment includes self-contained justification.

### Scenario 7: The Stale Artifact Passthrough

**Requirement tag:** [Req: inferred -- quality_gate.sh lines 256-276]

**What happened:** quality_gate.sh validates TDD sidecar dates by rejecting placeholders (YYYY-MM-DD, 0000-00-00) and future dates, but accepts any past date with no lower bound. A date of 1970-01-01 would pass. The cross-run contamination check (lines 605-624) compares version stamps but not date freshness. If a developer runs the playbook, generates artifacts, then months later re-runs only Phase 3 without regenerating the TDD sidecar, the old sidecar with a stale date would pass the gate -- even though the code review findings it references may no longer be valid against the current code.

**The requirement:** quality_gate.sh should either warn on dates older than a configurable threshold or include date freshness in cross-run contamination detection.

**How to verify:** Create a tdd-results.json with date "2020-01-01" and current skill_version. Run quality_gate.sh. Verify it at minimum warns about the stale date.

## AI Session Quality Discipline

1. Read QUALITY.md before starting any work on this repository.
2. Run quality_gate.sh after making any changes to verify artifact conformance is maintained.
3. When modifying SKILL.md's artifact contract table, verify quality_gate.sh and verification.md are updated to match.
4. When modifying quality_gate.sh check logic, verify SKILL.md's artifact contract table agrees on severity (FAIL vs WARN).
5. Never remove a fitness-to-purpose scenario. Only add new ones based on discovered failure modes.
6. Output a quality compliance checklist before ending a session that modifies the skill.

## The Human Gate

- **Specification design decisions** -- whether a given artifact should be required vs optional requires understanding the user's workflow and priorities. The WARN-vs-FAIL decision for each artifact is ultimately a product decision.
- **Phase structure changes** -- adding, removing, or reordering phases affects all downstream documentation and the gate. Human judgment is needed to assess the impact.
- **Reference file resolution strategy** -- the 4-path fallback chain depends on installation conventions that vary across AI coding tools. Changes require testing in multiple environments.
- **Security assessment of eval usage** -- determining whether the eval injection risk in quality_gate.sh is acceptable requires understanding the threat model (who runs this script, in what context).
- **Backward compatibility decisions** -- whether to maintain support for older SKILL.md versions in quality_gate.sh requires product-level judgment about the user base.
