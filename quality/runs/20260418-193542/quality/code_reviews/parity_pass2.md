### REQ-022: Gate summary sub-key checks must use `json_key_count` for consistency with per-bug field checks

**Status:** VIOLATED
**Evidence:** `quality_gate.sh:259-265` — `for skey in total verified confirmed_open red_failed green_failed; do if json_has_key "$json_file" "$skey"`
**Analysis:** The gate uses `json_has_key` for summary sub-key presence. `json_has_key` (line 75-78) uses `grep -q "\"${key}\""` which matches the key name anywhere in the file. The per-bug field check at lines 239-248 uses `json_key_count` which uses `grep -c "\"${key}\"[[:space:]]*:"` — a colon-anchored match that only finds actual JSON keys. Both check for required field presence, but REQ-022 requires consistency. The inconsistency means the summary check has a higher false-positive rate than the per-bug check.
**Severity:** LOW — in practice, summary key names like `total`, `verified` rarely appear in string values, but the requirement for consistency is violated.

### REQ-023: Patch existence check must iterate per-bug ID, not count aggregates

**Status:** VIOLATED
**Evidence:** `quality_gate.sh:562-588` — `reg_patch_count=$(ls ${q}/patches/BUG-*-regression*.patch 2>/dev/null | wc -l | tr -d ' ')`; compared to `quality_gate.sh:316-318` — `for bid in $bug_ids; do if [ -f "${q}/results/${bid}.red.log" ]`
**Analysis:** The TDD log section uses per-bug iteration with named bug IDs extracted from BUGS.md. The patch section uses an aggregate count with no per-bug verification. REQ-023 requires per-bug iteration for patches. The current approach passes when the count is right but the distribution is wrong (e.g., one bug has two patches and another has none). 
**Severity:** LOW — the structural inconsistency is confirmed, but generating a concrete false-pass scenario requires a specifically malformed patch directory.

### REQ-024: Phase 5 must include an entry gate verifying Phase 4 artifacts before proceeding

**Status:** VIOLATED
**Evidence:** `SKILL.md:1573-1590` (Phase 5 start) — reads: "Re-read quality/PROGRESS.md — specifically the cumulative BUG tracker." No check for Phase 4 artifact existence. Compare with `SKILL.md:897-907` (Phase 2 entry gate): "Before generating any artifacts, read quality/EXPLORATION.md from disk and verify ALL of the following exact section titles exist."
**Analysis:** Phase 2 entry gate mechanically verifies Phase 1 artifacts by checking for specific section titles in EXPLORATION.md. Phase 5 has no equivalent check. The only backstop is the terminal gate at the END of Phase 5 (SKILL.md:1609+), which fires after all Phase 5 work is done. The fail-early pattern from Phase 2 is not applied to Phase 5. REQ-024 requires a Phase 5 entry gate.
**Severity:** LOW — the terminal gate provides a backstop, but it fires late.

### REQ-025: SEED_CHECKS.md must be added to the artifact contract table when Phase 0b runs

**Status:** VIOLATED
**Evidence:** `SKILL.md:1641` — "If Phase 0 or 0b ran: quality/SEED_CHECKS.md exists as a standalone file (not inlined in PROGRESS.md)." `SKILL.md:88-119` — artifact contract table with 18 rows, none including SEED_CHECKS.md.
**Analysis:** The Phase 5 artifact file-existence gate explicitly requires SEED_CHECKS.md when Phase 0b ran. The canonical artifact contract table, described as "the canonical list — any artifact not listed here should not be gate-enforced," does not include this artifact. This creates a contradiction: the table says it's not gate-enforced, but Phase 5 says it's required. An agent reading only the table would not create SEED_CHECKS.md. Additionally, quality_gate.sh does not check for this file.
**Severity:** LOW — only applies when Phase 0b runs, which is a conditional mode.
