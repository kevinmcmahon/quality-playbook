## Findings

### quality_gate.sh

**Lines 259-265:** DIVERGENT [Req: inferred — REQ-022] From a reliability perspective: the gate makes a PASS/FAIL determination about required JSON keys using a function (`json_has_key`) that has a confirmed false-positive vulnerability (BUG-H1). This means the summary key check can PASS when summary keys are actually absent. The reliability claim "summary has 'total'" is not actually enforced — it's a probabilistic check. Using `json_key_count` would make it deterministic. The existing per-bug field check already uses the deterministic pattern. The summary check is a reliability regression relative to the per-bug check.

**Lines 316-345 vs 562-588:** DIVERGENT [Req: inferred — REQ-023] Reliability analysis: per-bug log iteration gives strong individual guarantees ("BUG-H1 has its log"). Aggregate patch counting gives a weak aggregate guarantee ("there are N patches for N bugs"). From a reliability standpoint, the gate can be gamed by the aggregate approach — submitting patches with wrong names (or duplicated names) that inflate the count. The per-bug approach cannot be gamed in this way. Both enforce the same contract but at different reliability levels.

### SKILL.md

**Lines 88-119 vs 1641:** DIVERGENT [Req: formal — SKILL.md:88] Reliability concern: the artifact contract table is described as authoritative ("canonical list"), but SEED_CHECKS.md is required by Phase 5 prose and not in the table. Any tool or agent relying on the table as the source of truth will not check for SEED_CHECKS.md. The table's reliability claim ("this is the canonical list") is false when Phase 0b runs. This creates a hidden gap in artifact completeness validation.

**Lines 1573-1590:** MISSING [Req: inferred — REQ-024] Reliability concern: Phase 5 reads PROGRESS.md as evidence that Phase 4 ran. PROGRESS.md is agent-maintained prose, not a mechanically generated artifact. An agent can write "Phase 4: complete" without producing the actual Phase 4 artifacts. Phase 2's entry gate (lines 897-907) checks mechanically generated content (EXPLORATION.md section titles). Phase 5's equivalent check (PROGRESS.md Phase 4 checkbox) checks prose. From a reliability standpoint, prose attestation is weaker than mechanical artifact verification. The asymmetry in reliability between Phase 2 and Phase 5 entry checks means the quality system is more trustworthy at Phase 2 boundaries than at Phase 5 boundaries.
