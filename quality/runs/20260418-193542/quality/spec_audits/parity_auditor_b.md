## Findings

### quality_gate.sh

**Lines 259-265:** UNDOCUMENTED [Req: inferred — internal consistency] The gate uses two different JSON validation patterns for what appears to be the same operation:
- `json_has_key` for summary key presence: `if json_has_key "$json_file" "$skey"`
- `json_key_count` for per-bug field presence: `fcount=$(json_key_count "$json_file" "$field")`
From a user perspective: if the gate fails with "summary missing 'total' count", the agent debugging this would reasonably check that `"total":` appears in the JSON. But if `total` appears only in a string value, the gate PASSES despite the agent's fix being incorrect. The inconsistency creates a scenario where the gate's diagnostic output ("summary missing 'X'") is unreliable — a PASS result from `json_has_key` doesn't actually mean the key is present as a JSON property.

**Lines 562-588:** UNDOCUMENTED [Req: inferred] An agent that generates all patches correctly but names one incorrectly (e.g., `BUG-H1-test-patch.patch` instead of `BUG-H1-regression-test.patch`) would cause `reg_patch_count` to be < `bug_count`. The gate would fail with "Only N regression-test patch(es) for M bug(s)" without identifying WHICH bug's patch is missing. The per-bug log check at lines 316-345 would identify the specific missing log. The inconsistency in iteration strategy means patch errors are harder to diagnose than log errors.

### SKILL.md

**Line 88 vs line 1641:** DIVERGENT [Req: formal] SKILL.md line 88 says the artifact contract table is "the canonical list." Line 1641 says SEED_CHECKS.md is required. These statements cannot both be true. From an agent's perspective: which one to follow? If the table is canonical, SEED_CHECKS.md is optional. If line 1641 is authoritative, the table is incomplete. The contradiction is user-hostile — an agent must choose which part of the spec to trust.

**Lines 1573 vs 897:** MISSING [Req: inferred — pattern consistency] From a user/agent experience perspective: Phase 2 has a clear "HARD STOP" entry gate with 6 explicit checks. An agent that reaches Phase 5 without completing Phase 4 would only discover this at the terminal gate after running the entire phase. Phase 2's design tells agents immediately if they're starting from a bad state. Phase 5's design allows agents to do substantial work before discovering they started from an incorrect state. This inconsistency in user experience creates wasted work.
