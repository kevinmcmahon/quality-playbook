## The Definitive Audit Prompt

Give this prompt identically to three independent AI tools (e.g., Claude, GPT, Gemini).

---

**Context files to read:**
1. `SKILL.md` — The master specification (2083 lines). This is the authoritative source for all behavioral rules, artifact contracts, and phase gate definitions.
2. `quality_gate.sh` — The only executable code (676 lines of Bash). Validates artifact conformance.
3. `quality/REQUIREMENTS.md` — 14 testable requirements derived from specification analysis.
4. `quality/QUALITY.md` — Quality constitution with 7 fitness-to-purpose scenarios.
5. `references/review_protocols.md` — Code review and integration test templates (611 lines).
6. `references/verification.md` — 45 self-check benchmarks for Phase 6 validation.
7. `references/requirements_pipeline.md` — Five-phase requirements generation pipeline.

**Task:** Act as the Tester. Read the actual code in `quality_gate.sh` and the specifications in `SKILL.md` and `references/*.md`. Compare the code's behavior against what the specifications say it should do.

**Requirement confidence tiers:**
- **formal** — stated in SKILL.md or reference files. Authoritative. Divergence is a real finding.
- **inferred** — deduced from code behavior or defensive patterns. Lower confidence. Report as NEEDS REVIEW.

**Rules:**
- ONLY list defects. Do not summarize what matches.
- For EVERY defect, cite specific file and line number(s).
  If you cannot cite a line number, do not include the finding.
- Before claiming missing, grep the codebase.
- Before claiming exists, read the actual function body.
- Classify each finding: MISSING / DIVERGENT / UNDOCUMENTED / PHANTOM

**Defect classifications:**
- **MISSING** — Spec requires it, code doesn't implement it
- **DIVERGENT** — Both spec and code address it, but they disagree
- **UNDOCUMENTED** — Code does it, spec doesn't mention it
- **PHANTOM** — Spec describes it, but it's actually implemented differently than described

**Project-specific scrutiny areas:**

1. **Artifact contract table vs gate checks.** SKILL.md lines 86-115 define required artifacts. quality_gate.sh lines 106-129 implement the checks. For EACH artifact in the table marked "Required: Yes," does quality_gate.sh check it? Does it use fail() or warn()? List every discrepancy.

2. **Sidecar JSON field validation.** SKILL.md lines 120-162 define the tdd-results.json and integration-results.json schemas. quality_gate.sh lines 208-291 and 368-388 validate them. For EACH required field in the schema, does the gate check it? What enum values does the gate accept vs what the spec allows?

3. **Phase gate enforcement.** SKILL.md defines entry and completion gates for each phase. Are these mechanically verifiable, or do they rely on agent self-attestation? Identify any gate that has no mechanical enforcement.

4. **Strictness mode consistency.** quality_gate.sh has --benchmark and --general modes. For EACH check that behaves differently between modes, is the difference justified by the specification? Specifically check: triage probe evidence (lines 147-163), integration sidecar (lines 382-388), and UC count (lines 416-419).

5. **Internal numbering.** Check all "Phase N" headings across all reference files. Are they sequential? Do any collide with the playbook's own phase numbers in a way that could confuse an agent?

6. **Version detection robustness.** quality_gate.sh lines 59-67 try 6 paths to find SKILL.md. What happens if none resolves? What happens if the script is run from a directory different from the repo root?

7. **Date validation completeness.** quality_gate.sh validates sidecar dates. Does it catch all invalid dates (placeholders, future dates, impossibly old dates)?

8. **Eval usage safety.** Search quality_gate.sh for `eval`. Is any user-controlled data passed through it? Can it be exploited?

9. **Bug count extraction robustness.** quality_gate.sh counts bugs by grepping for heading patterns. What happens if bugs are listed in a table instead of headings? What if the BUGS.md format changes?

10. **AGENTS.md coverage.** SKILL.md line 102 says AGENTS.md is required. Does quality_gate.sh check for it?

## Triage Process

After collecting reports from three auditors:

### Step 1: Merge findings

Create a table of all unique findings across all three auditors:

| Finding | Auditor 1 | Auditor 2 | Auditor 3 | Classification | Confidence |
|---------|-----------|-----------|-----------|----------------|------------|
| [description] | Found | Found | Not found | DIVERGENT | High (2/3) |

### Step 2: Investigate minority findings

For any finding where only 1 of 3 auditors flagged it: re-read the specific code location. Make an explicit CONFIRMED / FALSE POSITIVE determination. Do not discard minority findings by default.

### Step 3: Verification probes

For each finding being confirmed or rejected, write an executable assertion:

**For rejections (false positive):**
```bash
# Verify [claim] is false
grep -n "exact pattern" file.ext | head -1
# Expected: pattern found at line N, proving the finding is wrong
```

**For confirmations (real bug):**
```bash
# Verify [bug claim] is true
grep -c "pattern" file.ext
# Expected: count is 0, proving the code is missing the check
```

Write all probes to `quality/spec_audits/triage_probes.sh`.

### Step 4: Categorize confirmed findings

For each confirmed finding:
- **Real code bug** -> Add to BUGS.md, write regression test
- **Spec inconsistency** -> Update SKILL.md or reference file
- **Documentation gap** -> Update relevant file
- **Design choice** -> Document in triage report with justification
