# BUG-011: json_has_key() Grep Matches Keys Inside String Values (Exempt)

> Generated by [Quality Playbook](https://github.com/andrewstellman/quality-playbook) v1.4.0 — Andrew Stellman
> Date: 2026-04-15 · Project: quality-playbook (bootstrap self-audit)

## 1. Summary

quality_gate.sh's `json_has_key()` function (lines 76-78) uses flat `grep` to check for JSON key presence. This can produce false positives if a key name appears inside a string value. The bug is exempt from regression testing because fixing it requires a JSON parser, and grep-based parsing is a deliberate design choice for portability.

## 2. Spec reference

quality_gate.sh design: grep-based JSON parsing is chosen for portability (no dependency on jq or python3).

## 3. The code

```
quality_gate.sh:76-78
json_has_key() {
    grep -q "\"$2\"" "$1"
}
```

This grep matches `"total"` anywhere in the file, including inside string values like `"description": "see total count"`.

## 4. Observable consequence

A tdd-results.json file with a string value containing a key name (e.g., `"notes": "check total"`) would satisfy the key-presence check even if the actual `"total"` key is missing. In practice, this is unlikely given the structured nature of the generated JSON.

## 6. The fix

```diff
--- a/quality_gate.sh
+++ b/quality_gate.sh
@@ -76,7 +76,9 @@
 json_has_key() {
-    grep -q "\"$2\"" "$1"
+    # Use jq for accurate key checking (requires jq dependency)
+    jq -e "has(\"$2\")" "$1" > /dev/null 2>&1
 }
```

This fix adds a jq dependency. The current grep approach is by design for portability.

## 7. The test

Exempt: structural limitation of grep-based JSON parsing. Fixing requires a JSON parser dependency.
