# BUG-018: json_key_count false PASS masks missing per-bug fields in tdd-results.json

> Generated by [Quality Playbook](https://github.com/andrewstellman/quality-playbook) v1.4.0 — Andrew Stellman
> Date: 2026-04-15 · Project: quality-playbook (QFB-bootstrap)

## 1. Summary

quality_gate.sh's `json_key_count()` function at lines 88-91 uses `grep -c "\"${key}\""` to count JSON key occurrences. This counts lines containing the quoted key name anywhere, including inside string values. When a bug's `requirement` field contains a JSON field name (e.g., `"the \"verdict\" enum"`), the count is inflated. The per-bug field validation at lines 226-234 checks `fcount >= bug_count` -- an inflated count masks a missing field in a different bug entry, producing a false PASS.

## 2. Spec reference

SKILL.md line 1424: "every required per-bug field is present in each `bugs[]` entry." The gate's per-bug field check is supposed to enforce this. A false PASS defeats the enforcement.

## 3. The code

quality_gate.sh lines 88-91:
```bash
json_key_count() {
    local file="$1" key="$2"
    grep -c "\"${key}\"" "$file" 2>/dev/null || echo 0
}
```

Lines 226-234 use this function:
```bash
for field in id requirement red_phase green_phase verdict fix_patch_present writeup_path; do
    local fcount
    fcount=$(json_key_count "$json_file" "$field")
    if [ "$fcount" -ge "$bug_count" ]; then
        pass "per-bug field '${field}' present (${fcount}x)"
    ...
```

The grep pattern `"verdict"` matches both the actual JSON key `"verdict": "TDD verified"` and the word "verdict" inside any string value like `"requirement": "REQ-014: \"verdict\" field must use canonical values"`.

## 4. Observable consequence

A tdd-results.json with 2 bugs where BUG-001 has all fields and mentions "verdict" in its requirement text, and BUG-002 is missing the verdict field entirely, produces: json_key_count("verdict") = 2 (one real key, one string match). The gate reports PASS (2 >= 2) when BUG-002 is actually missing the field. The missing field goes undetected.

## 6. The fix

Change the grep pattern to match key-value pairs (key followed by colon) instead of bare key names:

```diff
--- a/quality_gate.sh
+++ b/quality_gate.sh
@@ -87,8 +87,10 @@
 # Helper: count occurrences of a key in JSON
+# Use a pattern that matches key-value pairs (key followed by colon)
+# to avoid false matches on key names inside string values
 json_key_count() {
     local file="$1" key="$2"
-    grep -c "\"${key}\"" "$file" 2>/dev/null || echo 0
+    grep -cE "\"${key}\"[[:space:]]*:" "$file" 2>/dev/null || echo 0
 }
```

## 7. The test

The regression test creates a tdd-results.json with 2 bugs where BUG-002 is missing the verdict field but BUG-001's requirement text contains "verdict". On unpatched code, json_key_count returns 2 (false PASS). After applying the fix, json_key_count returns 1 (correctly detects the missing field).

Run: `bash quality/test_regression.sh` and look for `test_bug_018_json_key_count_false_pass`.
- Unpatched: XFAIL (count is 2, masks missing field)
- Patched: PASS (count is 1, correctly identifies gap)

## 8. Related issues

BUG-018 is related to the broader class of grep-based JSON parsing limitations noted in REQ-014 and SA-09. The json_has_key() function at line 77 has the same pattern-matching issue but is used for presence checks (boolean), where a false positive is less harmful. json_key_count() is used for per-bug field counting where the false match directly masks a missing field.
