# BUG-019 — Local pytest shim does not honor documented CLI forms

## 1. Summary

The repository-local `pytest` shim always executes the unittest runner and routes unknown path
syntax directly into `unittest` discovery. As a result, `--collect-only` still runs tests and
pytest node IDs can crash with `ImportError`, even though `SKILL.md` documents those CLI forms.

## 2. Spec reference

REQ-020 — Local pytest shim honors documented CLI forms.

> `[Tier 1] [SKILL.md] §RUN_TDD_TESTS examples` — "python -m pytest -xvs quality/test_regression.py::test_bug_nnn"

## 3. The code

`pytest/__main__.py:16-34` parses paths with `parse_known_args()` but always ends by calling
`unittest.TextTestRunner(...).run(suite)`. Its catch-all path branch forwards any non-directory /
non-`.py` token into `loader.discover(str(path))`, which is not valid for pytest node IDs.

## 4. Observable consequence

Discovery-only commands can execute the full suite, and targeted TDD commands using `::` can die
before the regression test runs. Both failures are operator-visible and break documented
maintenance workflows.

## 6. The fix

```diff
--- a/pytest/__main__.py
+++ b/pytest/__main__.py
@@
-    parser.add_argument("paths", nargs="*")
+    parser.add_argument("--collect-only", action="store_true")
+    parser.add_argument("paths", nargs="*")
@@
-        path = Path(raw_path)
+        if "::" in raw_path:
+            raise SystemExit("pytest node IDs are not supported by the local unittest shim")
+        path = Path(raw_path)
@@
-    result = unittest.TextTestRunner(verbosity=2).run(suite)
+    if known.collect_only:
+        for test in suite:
+            print(test)
+        return 0
+    result = unittest.TextTestRunner(verbosity=2).run(suite)
```

## 7. The test

Two regression tests cover the bug:

1. `quality/test_regression.py::GapIterationRegressions::test_reg_gap19_collect_only_does_not_execute_tests`
   asserts that `TextTestRunner.run()` is not called for `--collect-only`.
2. `quality/test_regression.py::GapIterationRegressions::test_reg_gap19_nodeid_is_handled_without_importerror`
   asserts that a documented node-ID path is handled explicitly instead of surfacing raw
   `ImportError`.
