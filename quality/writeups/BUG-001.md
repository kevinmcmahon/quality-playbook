# BUG-001: Contradictory Summary Key in tdd-results.json Example 1

> Generated by [Quality Playbook](https://github.com/andrewstellman/quality-playbook) v1.4.0 — Andrew Stellman
> Date: 2026-04-15 · Project: quality-playbook (bootstrap self-audit)

## 1. Summary

SKILL.md line 127 uses `"tdd_verified": 2` as a summary key in the first canonical tdd-results.json example. The normative text at line 1394 defines the required key as `"verified"`. An AI agent copying Example 1 verbatim produces a summary with a non-standard key that quality_gate.sh does not validate, causing silent schema drift across runs.

## 2. Spec reference

SKILL.md lines 1392-1394: "The `summary` object must contain exactly these keys: `total`, `verified`, `confirmed_open`, `red_failed`, `green_failed`. All five are required."

The behavioral contract: the summary key for verified bugs is `verified`, not `tdd_verified`.

## 3. The code

```
SKILL.md:127
    "total": 3, "confirmed_open": 1, "red_failed": 0, "green_failed": 0, "tdd_verified": 2
```

This contradicts the normative field list at line 1394 which specifies `verified` as the canonical key name. Example 2 (line 1384) correctly uses `"verified": 4`.

## 4. Observable consequence

An agent following Example 1 produces `"tdd_verified": 2` in the summary. Downstream tooling reading the `verified` key gets `null`/undefined. The quality_gate.sh script (line 229) checks for neither `tdd_verified` nor `verified`, so the contradiction is invisible to the mechanical gate. Different runs using different examples produce incompatible JSON schemas.

## 6. The fix

```diff
--- a/SKILL.md
+++ b/SKILL.md
@@ -124,7 +124,7 @@
   ],
   "summary": {
-    "total": 3, "confirmed_open": 1, "red_failed": 0, "green_failed": 0, "tdd_verified": 2
+    "total": 3, "confirmed_open": 1, "red_failed": 0, "green_failed": 0, "verified": 2
   }
```

Change `"tdd_verified"` to `"verified"` to match the normative text and Example 2.
