# BUG-015: integration-results.json validation depth asymmetry vs tdd-results.json

> Generated by [Quality Playbook](https://github.com/andrewstellman/quality-playbook) v1.4.0 — Andrew Stellman
> Date: 2026-04-15 · Project: quality-playbook (QFB-bootstrap)

## 1. Summary

quality_gate.sh validates tdd-results.json with 7 distinct checks (schema_version value, date format/placeholder/future, per-bug field counts, bad field detection, verdict enum, summary sub-keys) but validates integration-results.json with only 2 checks (root key presence, recommendation enum). The spec requires both sidecars to have `schema_version: "1.1"`, but the gate never validates this for integration-results.json. A non-conformant integration-results.json with `schema_version: "0.9"`, a placeholder date, or a stale skill_version passes the gate silently.

## 2. Spec reference

verification.md benchmark 41 (line 232): "Both must have `schema_version: "1.1"`." SKILL.md line 1644: "For `integration-results.json`, the required root keys are: `schema_version`, `skill_version`, `date`, `project`, `recommendation`, `groups`, `summary`, `uc_coverage`."

## 3. The code

**tdd-results.json validation (quality_gate.sh:207-291):**
```bash
# Line 222: schema_version value check
[ "$sv" = "1.1" ] && pass "schema_version is '1.1'" || fail "schema_version is '${sv:-missing}', expected '1.1'"

# Lines 253-276: date validation (format, placeholder, future check)
# Lines 278-284: verdict enum validation
# Lines 244-251: summary sub-key check (5 keys)
# Lines 224-235: per-bug field count matching (7 fields)
# Lines 237-242: bad field name detection (5 patterns)
```

**integration-results.json validation (quality_gate.sh:367-388):**
```bash
# Line 371: key existence only — no value checks
for key in schema_version skill_version date project recommendation groups summary uc_coverage; do
    json_has_key "$ij" "$key" && pass "has '${key}'" || fail "missing key '${key}'"
done

# Lines 376-381: recommendation enum — the ONLY value check
```

No schema_version value check. No date validation. No skill_version cross-check. No per-group field validation.

## 4. Observable consequence

An integration-results.json with `schema_version: "0.9"`, `date: "YYYY-MM-DD"`, and `skill_version: "0.0.0"` passes the quality gate. A runner or CI tool consuming integration-results.json may encounter an incompatible schema version with no prior warning from the gate.

## 6. The fix

Add schema_version value check, date validation, and skill_version cross-check to the integration sidecar section, matching the checks already present for tdd-results.json:

```diff
--- a/quality_gate.sh
+++ b/quality_gate.sh
@@ -372,6 +372,23 @@
         for key in schema_version skill_version date project recommendation groups summary uc_coverage; do
             json_has_key "$ij" "$key" && pass "has '${key}'" || fail "missing key '${key}'"
         done
+
+        # schema_version value check (parity with tdd-results.json)
+        local isv
+        isv=$(json_str_val "$ij" "schema_version")
+        [ "$isv" = "1.1" ] && pass "integration schema_version is '1.1'" || fail "integration schema_version is '${isv:-missing}', expected '1.1'"
+
+        # Date validation (parity with tdd-results.json)
+        local int_date
+        int_date=$(json_str_val "$ij" "date")
+        if [ -n "$int_date" ]; then
+            if echo "$int_date" | grep -qE '^[0-9]{4}-[0-9]{2}-[0-9]{2}$'; then
+                if [ "$int_date" = "YYYY-MM-DD" ] || [ "$int_date" = "0000-00-00" ]; then
+                    fail "integration-results.json date is placeholder '${int_date}'"
+                else
+                    local today_int; today_int=$(date +%Y-%m-%d)
+                    [[ "$int_date" > "$today_int" ]] && fail "integration-results.json date '${int_date}' is in the future" || pass "integration-results.json date '${int_date}' is valid"
+                fi
+            else
+                fail "integration-results.json date '${int_date}' is not ISO 8601 (YYYY-MM-DD)"
+            fi
+        fi
 
         # Recommendation enum
```

## 7. The test

The regression test creates an integration-results.json with schema_version "0.9" and date "YYYY-MM-DD". On unpatched code, the gate exits 0 (these violations are undetected). After applying the fix, the gate reports FAIL for the non-conformant schema_version and placeholder date.
