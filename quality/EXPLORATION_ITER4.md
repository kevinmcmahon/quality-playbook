### Comparison 1: tdd-results.json deep validation vs integration-results.json shallow validation [PG-1, PG-8]

**tdd-results.json validation (lines 207-291):**
- Root key checking: 6 keys verified (schema_version, skill_version, date, project, bugs, summary)
- schema_version value check: verifies value is "1.1"
- Per-bug required field count matching: 7 fields (id, requirement, red_phase, green_phase, verdict, fix_patch_present, writeup_path) — each checked against bug_count
- Bad field name detection: 5 anti-patterns checked (bug_id, bug_name, status, phase, result)
- Summary sub-key check: 5 keys (total, verified, confirmed_open, red_failed, green_failed)
- Date validation: format check, placeholder rejection, future-date rejection
- Verdict enum validation: rejects non-canonical verdict values

**integration-results.json validation (lines 367-388):**
- Root key checking: 8 keys verified (schema_version, skill_version, date, project, recommendation, groups, summary, uc_coverage)
- Recommendation enum check: validates against 3 allowed values
- Nothing else. No schema_version value check, no date validation, no per-group field check, no bad field detection, no skill_version cross-check

**Discrepancies found:**
1. **schema_version value not checked:** tdd-results.json verifies schema_version is "1.1" (line 222). integration-results.json only checks the key exists (line 371). A non-conformant schema_version like "0.9" passes the integration sidecar check.
2. **Date not validated:** tdd-results.json has 20 lines of date validation (lines 253-276) checking format, placeholder, and future-date. integration-results.json has zero date validation. A placeholder "YYYY-MM-DD" passes.
3. **skill_version not cross-checked:** tdd-results.json's skill_version is cross-checked against SKILL.md at lines 594-598 and 617-624. integration-results.json's skill_version is never cross-checked — only key presence is verified.
4. **Per-group field validation absent:** SKILL.md line 106 specifies per-group required fields (group, name, use_cases, result). quality_gate.sh never validates per-group fields.
5. **No bad field detection:** tdd-results.json checks for 5 non-canonical field names. integration-results.json has no equivalent check.

**Bug determination:** This is a confirmed parity bug. verification.md benchmark 41 (line 232) says: "Both must have `schema_version: "1.1"`." SKILL.md line 162 specifies the integration-results.json template with date, schema_version "1.1", per-group fields, etc. The gate validates tdd-results.json deeply but integration-results.json shallowly. The most concrete violation: schema_version "1.1" is required by benchmark 41 but not checked for integration-results.json.

**Phases with explicit entry prerequisites (entry gates):**

- Phase 3: Lines 1496-1502. "Required references" section names prerequisites but no formal gate checking prerequisites exist on disk.
- Phase 4: Line 1525-1530. "Required references" section — no formal pre-check.
- Phase 5: Lines 1569-1576. "Required references" section — no formal pre-check.

**Asymmetry:** Phase 2 has a backstop entry gate that checks EXPLORATION.md before proceeding (lines 886-902). Phases 3-6 have "Required references" headers that list prerequisites but do not gate on their existence. Phase 3 could proceed even if Phase 2 artifacts (REQUIREMENTS.md, CONTRACTS.md) don't exist — SKILL.md just says "Re-read quality/PROGRESS.md and quality/REQUIREMENTS.md before starting Phase 3" without a hard stop if REQUIREMENTS.md is missing.

**Assessment:** This is an asymmetry in phase entry gate strictness. Phase 2 has an explicit backstop ("STOP and go back to Phase 1" if EXPLORATION.md is malformed), but Phases 3-6 use soft prerequisites ("Re-read" instructions without "STOP if missing" language). This is arguably by design — Phases 3-6 are typically run in sequence after Phase 2, so the backstop is less critical. Classifying as design choice, not bug.

### Comparison 6: quality_gate.sh writeup diff check — complete vs partial writeups [lines 546-575]

The writeup section has two checks:
1. **Count check (lines 558-564):** writeup_count >= bug_count = pass; writeup_count > 0 but < bug_count = warn (BUG-010); writeup_count == 0 = fail.
2. **Diff check (lines 567-575):** writeup_diff_count >= writeup_count = pass; writeup_diff_count > 0 but < writeup_count = fail; writeup_diff_count == 0 = fail.

**Parity observation:** The count check uses warn() for partial writeups (BUG-010), but the diff check uses fail() for partial diffs. So a run with 3 bugs, 2 writeups, and 2 diffs would: WARN (2 writeups for 3 bugs, line 561) but PASS (2/2 writeups have diffs, lines 568-569). The incomplete set is warned, but the diff check passes because all *existing* writeups have diffs.

**Assessment:** The diff check's denominator is writeup_count, not bug_count. This means: if a run is missing writeups (only warned, not failed per BUG-010), and the existing writeups all have diffs, the diff check passes. This is consistent with BUG-010 (the WARN-not-FAIL issue) — the diff check inherits the permissiveness. Not a separate bug; it's a consequence of BUG-010.
