# Behavioral Requirements — quality-playbook
> Generated by [Quality Playbook](https://github.com/andrewstellman/quality-playbook) v1.5.3 — Andrew Stellman
> Date: 2026-04-28 · Project: quality-playbook

Total requirements: 11

quality-playbook is a hybrid repository: part executable Python benchmark/orchestration tool, part product-spec bundle made of `SKILL.md`, `references/`, install surfaces, and bootstrap documentation. Its users are maintainers evolving the playbook, benchmark operators installing it into target repos, and downstream AI agents that inherit whatever persistent instructions and artifacts the repository leaves on disk. That means the repository's "runtime" is broader than the Python process. A Markdown heading in `SKILL.md`, a copied install file, a gate invariant, and a live `quality/INDEX.md` payload all participate in whether later phases see the right truth.

The dominant data flow in this run starts at documentation intake and ends at later-phase evidence. Phase 1 should read `reference_docs/` when present, fall back cleanly to Tier 3 source evidence when not, and record what it learned in `quality/EXPLORATION.md`. Phase 2 and later phases then consume those artifacts by name: the exploration headings, the requirement manifest, the use cases, and the live run index all become machine- and human-readable sources of truth. If the repo publishes mismatched surfaces—for example, a skill that names one gate contract while `bin/run_playbook.py` enforces a weaker one, or an install path that copies `quality_gate.py` without its verifier dependency—later phases inherit a false but plausible view of the system.

The main design philosophy visible in exploration is "executed evidence outranks prose," but the repo currently contains several places where prose/code/install paths drift apart. The requirements below therefore focus on preserving closed sets, parity surfaces, and compensating paths. The review lens for this project is not just "does the command run?" but "does every documented path teach and enforce the same quality bar?" A playbook that silently weakens itself during install or self-audit still produces artifacts, but those artifacts are no longer trustworthy.

For this Phase 2 baseline, all requirements are Tier 3. `reference_docs/` was effectively empty in this checkout, so no Tier 1 or Tier 2 formal-doc citations were available. The authoritative source of truth for these REQ records is the explored code, the shipped skill bundle, and the repository's published bootstrap artifacts.

### UC-10: Citation verifier reads only files inside the repo root

- **Actor:** Quality gate
- **Preconditions:** A citation record (or manifest entry) names a `source_path` to be verified.

  1. Resolve `source_path` against `root`.

- **Postconditions:** A tampered manifest or hostile citation record cannot read files outside the repo root.
- **Alternative paths:** Manifest-supplied and citation-supplied paths must both pass containment validation; symlinks that point back inside `root` are acceptable.
- **Requirements:** REQ-011

## Cross-cutting concerns

Three concerns span the whole requirements set. First, **closed-set preservation**: REQ-003 and REQ-004 protect enumerated contracts where omission is the bug. Required headings and supported project types are both small, explicit sets, so they need mechanical preservation instead of best-effort prose alignment. Second, **parity across surfaces**: REQ-001, REQ-002, REQ-006, and REQ-008 all defend places where two legitimate entry points should describe or enforce the same behavior. When these paths drift, the repo still appears usable while different operators get different answers.

Third, **safe degradation**: REQ-005 and REQ-007 cover fallback paths that should stay truthful under weaker conditions. Missing PID files must not widen cleanup scope, and curated docs must not become stale enough to mislead self-audits. Fourth, **resolution-fidelity across install layouts**: REQ-009 and REQ-010 extend the same discipline to the path-resolution surface itself. Reviewers should treat these "still works" paths as first-class quality surfaces, because this repository's biggest risks are not hard crashes but success-shaped drift.

## Safety, Cleanup, and Documentation Fidelity

This section covers the repository's fallback and support surfaces. These behaviors often execute only when something is missing or unusual, which makes them easy to under-test and easy to rationalize away. That is precisely why they need explicit requirements here.

### REQ-011: Citation verifier must enforce repository-root containment for all dereferenced paths

- **Tier:** 3
- **Functional section:** Safety, Cleanup, and Documentation Fidelity
- **Source type:** code-derived
- **Summary:** `verify_citation()` must reject `source_path` values that resolve outside the supplied `root` (absolute paths, `..` traversal, symlink escapes) before reading bytes from disk, so a tampered manifest or auditor-supplied citation cannot be turned into an out-of-tree file-read primitive.
- **User story:** As an operator running the gate against a target repo, I expect the citation verifier to refuse to read files outside the repo root **so that** an attacker who tampers with `formal_docs_manifest.json` or supplies a hostile citation record cannot escalate the gate into a workstation-wide file-read.
- **Implementation note:** `bin/citation_verifier.py:244-266` currently computes `doc_path = Path(root) / source_path` and immediately calls `doc_path.read_bytes()` with no `is_relative_to(root.resolve())` check. `Path(root) / "/etc/passwd"` evaluates to `/etc/passwd` (Python `Path.__truediv__` discards the left side when the right is absolute), and `Path(root) / "../../etc/passwd"` resolves outside the root after `read_bytes()`.
- **Conditions of satisfaction:**
  1. `verify_citation()` rejects absolute `source_path` values with a non-OK `VerificationResult` and a structured error code (`ERROR_DOCUMENT_OUT_OF_ROOT` or equivalent), without reading file bytes.
  2. `verify_citation()` rejects relative `source_path` values whose resolved location is outside `Path(root).resolve()`, including `..` segments and symlink targets that escape `root`.
  3. The rejection is observable in `VerificationResult.error_code` so callers can distinguish boundary violations from missing-document errors.
- **Alternative paths:**
  - Manifest-supplied paths and citation-supplied paths must both pass containment validation; the verifier may use a single helper to check either input shape.
  - Symlink resolution must compare the resolved target against the resolved root; a symlink that points back inside the root is acceptable.
- **Use cases:** UC-10
- **References:** `bin/citation_verifier.py:244-266`; `SKILL.md:94`; `quality/EXPLORATION_ITER5.md` finding A-1
