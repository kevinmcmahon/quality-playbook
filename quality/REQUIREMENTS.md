# Requirements — Quality Playbook

## Overview

Quality Playbook (QPB) is an AI-driven quality-engineering skill: given any codebase, a compliant AI agent follows its six-phase procedure to produce a complete quality system — requirements, functional tests, a three-pass code review with regression tests, a Council-of-Three spec audit, TDD-verified patches, and a post-run validation gate. The skill is installed into a target repo under `.github/skills/` or `.claude/skills/` and is invoked by asking an AI agent to "run the quality playbook." The repository also ships a Python benchmark harness (`bin/run_playbook.py`) that runs the skill non-interactively across multiple target repos for cross-model comparison, and a post-run gate (`quality_gate.py`) that mechanically validates artifact conformance.

**Actors.** (1) *QPB maintainer* — develops the skill, runs benchmarks, writes releases. (2) *Benchmark operator* — researchers replicating published benchmark numbers across models. (3) *Non-maintainer end-user developer* — a developer on an unrelated project who installs the skill locally and asks their AI agent to run it. (4) *AI agent* — a compliant large-language-model assistant (Claude Code, GitHub Copilot CLI, Cursor, Windsurf, Cowork) that executes the six phases interactively. (5) *CI / operator* — runs the validation gate after any of the above produced artifacts.

**Primary deliverables.** The `SKILL.md` specification, the Python harness (`bin/run_playbook.py`, `bin/benchmark_lib.py`, and supporting `.github/skills/quality_gate/quality_gate.py`), the reference-file cluster in `.github/skills/references/`, and the two agent-instruction files in `agents/`.

**Non-goals.** QPB does not host an HTTP server, a database, or a background worker. It has no runtime authentication layer. Third-party Python dependencies are forbidden — the entire surface must run on the stdlib. Any requirement that would introduce a dependency is by construction out of scope.

## Use Cases

### UC-01 — Maintainer runs the playbook against a single target repository

**Actor.** QPB maintainer.

**Goal.** Generate a full quality system for a third-party repo so bug yield can be measured and compared across models.

**Trigger.** The maintainer runs `python3 bin/run_playbook.py <target>` from the QPB directory.

**Preconditions.** The target repo exists under `repos/<target>/` or at an absolute path. SKILL.md is installed in the target at one of the four documented paths. `docs_gathered/` in the target is populated.

**Main flow.** (1) `resolve_target_dirs` maps the argument to a directory. (2) `display_run_header` prints config. (3) `execute_run` spawns a worker subprocess. (4) Worker invokes `claude -p` or `gh copilot -p` with per-phase prompts. (5) Artifacts land under `<target>/quality/`. (6) `cleanup_repo` reverts tracked non-protected edits in the target. (7) Summary printed; exit code reflects worker status.

**Acceptance.** Exit 0 when all phases complete. `<target>/quality/` contains every artifact required by the Phase 2/3/4/5 contract. Log file written to `<target>.log`.

**Related requirements.** REQ-001, REQ-002, REQ-005, REQ-009, REQ-010, REQ-011.

### UC-02 — Maintainer runs a bootstrap self-audit (QPB audited against itself)

**Actor.** QPB maintainer.

**Goal.** Catch quality issues in the skill itself using the skill's own methodology. This run is an instance of this use case.

**Trigger.** `bin/run_playbook.py .` from the QPB root (target = `.`).

**Preconditions.** QPB repo has docs_gathered/ populated, `.github/skills/SKILL.md` installed, and no stale `quality/` directory (or the user accepts that Phase 1 will archive it).

**Main flow.** Same as UC-01 except the target is the QPB repo itself. The agent writing `quality/EXPLORATION.md` is examining SKILL.md, bin/, references/, quality_gate.py — the very files defining the process the agent is following.

**Acceptance.** Findings land in `quality/EXPLORATION.md` with citations into QPB's own files. `cleanup_repo` does not silently revert agent-authored fixes to source files (REQ-006). The agent uses patches (`quality/patches/BUG-NNN-fix.patch`), not direct edits, for proposed fixes.

**Related requirements.** REQ-006, REQ-017.

### UC-03 — Benchmark operator compares multiple AI agents against the same target

**Actor.** Benchmark operator (researcher replicating v1.4.5 numbers).

**Goal.** Run the same target against Claude Opus, Claude Sonnet, GPT-5.4, Copilot, etc., and compare bug yield.

**Trigger.** Multiple invocations: `bin/run_playbook.py --claude --model opus <target>`, `--claude --model sonnet <target>`, `--copilot --model gpt-5.4 <target>`.

**Preconditions.** Target repo exists. Every run's `model`, `model_provider`, `runner`, `start_time`, `end_time` is captured in run-metadata JSON for cross-comparison.

**Main flow.** Each run archives prior `quality/` into `previous_runs/TIMESTAMP/quality/`. Each run writes its own run-metadata JSON. Operator later reads the JSONs for analysis.

**Acceptance.** Run-metadata JSON exists for every run, has the correct filename format, and the five cross-comparison fields are populated. Archive atomicity is preserved across rapid-fire runs.

**Related requirements.** REQ-007, REQ-008, REQ-009.

### UC-04 — Benchmark operator runs a full-cycle iteration chain

**Actor.** Benchmark operator (same as UC-03).

**Goal.** Capture the 40–60% iteration uplift described in SKILL.md:227 by running every strategy in the iteration cycle.

**Trigger.** `bin/run_playbook.py --full-run <target>` or chained `--next-iteration --strategy gap|unfiltered|parity|adversarial`.

**Preconditions.** A baseline run exists. Baseline EXPLORATION.md is the input to the first iteration.

**Main flow.** Runner iterates the strategy chain. Each strategy's findings merge into `EXPLORATION_MERGED.md`. Early stop on zero-gain. Final Demoted Candidates Manifest populated for the adversarial pass.

**Acceptance.** Each completed strategy's artifacts survive the next strategy's run. Suggested-next-command output at the end of the chain says "all strategies run" and does not loop.

**Related requirements.** REQ-014.

### UC-05 — Maintainer kills a runaway parallel run

**Actor.** Any operator who started a parallel benchmark.

**Goal.** Stop all worker processes on demand.

**Trigger.** `python3 bin/run_playbook.py --kill`, or Ctrl-C in the controlling terminal.

**Preconditions.** One or more parents have written `.run_pids.<pid>` files, or have spawned workers whose PIDs can be recovered by pattern matching.

**Main flow.** (1) `kill_recorded_processes` reads every PID file and SIGTERMs each PID. (2) If no PID file is found, `_pkill_fallback` pkills every dispatched command pattern. (3) Orphaned PID files are removed.

**Acceptance.** No worker process (Claude or Copilot) survives the kill. No unrelated CLI sessions on the same machine are killed.

**Related requirements.** REQ-012.

## Requirements

Requirements are tiered per `requirements_pipeline.md`:
- **Tier 1** — Documented contract (SKILL.md or installed reference file explicit).
- **Tier 2** — Source-of-truth contract (derivable from the code; not spelled out in prose but enforced by implementation).
- **Tier 3** — Proactive risk mitigation (code-review-informed; the implementation works today but is brittle).

Every requirement below carries seven fields: **Summary**, **User story**, **Implementation note**, **Conditions of satisfaction**, **Alternative paths**, **References**, **Doc source**, **Specificity**. Field order is fixed across all entries.

### REQ-001: Single canonical version-parser helper

**Summary.** The repository must expose exactly one authoritative `skill_version()` helper. All other call sites must delegate to it.

**User story.** As the QPB maintainer, when I bump the version in SKILL.md's frontmatter using either bare (`version: 1.4.6`) or bold (`**Version:** 1.4.6`) form, I want every tool in the repo — the runner, the library, and the gate — to read the same string so that benchmark runs and validation are consistent.

**Implementation note.** `benchmark_lib._read_version()` uses a regex that handles both forms. `benchmark_lib.skill_version()` uses case-sensitive `startswith("version:")` and misses the bold form. `quality_gate.detect_skill_version()` uses substring match with no numeric validation. Consolidate to a single helper (the regex path) and have all call sites import from it.

**Conditions of satisfaction.** (1) Only one function in the codebase reads `version:` from SKILL.md. (2) Given SKILL.md lines `  version: 1.4.5`, `  VERSION: 1.4.5`, `  **Version:** 1.4.5`, the helper returns `1.4.5` for all three. (3) Every other module that used to parse version text now calls the canonical helper. (4) A functional test pins each of the three form variations and asserts equality.

**Alternative paths.** If consolidation is too invasive, the three existing parsers must be synchronized function-by-function and guarded by a shared set of unit tests that all three must pass.

**References.** `benchmark_lib.py:37` (`VERSION_PATTERN`); `benchmark_lib.py:74-81` (`_read_version`); `benchmark_lib.py:90-116` (`skill_version`); `run_playbook.py:217` (call site); `quality_gate.py:176-191` (`detect_skill_version`).

**Doc source.** `[source]` — EXPLORATION.md §F-1, §QR-1, Cross-Implementation Consistency deep dive.

**Specificity.** specific.

### REQ-003: Version parsers reject substring-only matches

**Summary.** Version-parsing helpers must reject lines that only *contain* the substring `version:` (e.g., inside a quoted description field). The match must be anchored at the start of a stripped line.

**User story.** As the QPB maintainer writing a YAML frontmatter where a description field happens to include the word "version:", I want the version parser to not confuse that description with the real version line.

**Implementation note.** Current `quality_gate.detect_skill_version` uses `if "version:" in line` (substring anywhere on the line). A YAML key `description: "The version: 1.4.5 release"` matches. The regex in `_read_version` already anchors with `^\s*`. All parsers must anchor identically.

**Conditions of satisfaction.** (1) Given `description: "The version: 1.4.5 release of the skill"`, the parser does not return `1.4.5`. (2) Given `# version: 1.3.5 (prior)` (commented line), the parser does not match. (3) A functional test pins both negative cases.

**Alternative paths.** A YAML-aware parser (stdlib-only: custom minimal YAML reader limited to the frontmatter block) may be used instead of anchored regex. The chosen approach must be unified across parsers.

**References.** `benchmark_lib.py:74-81`; `quality_gate.py:176-191`.

**Doc source.** `[source]` — EXPLORATION.md §F-1, Spec-Structured Parsing deep dive.

### REQ-007: ISO 8601 validation matches the field being checked

**Summary.** `validate_iso_date` in `quality_gate.py` must align with the field grammar: date-only fields (sidecar `date`) use `YYYY-MM-DD`; datetime fields (run-metadata `start_time` / `end_time`) accept the full ISO 8601 datetime grammar (`T`, optional timezone).

**User story.** As a benchmark operator whose run-metadata has `start_time: 2026-04-18T23:43:14Z` (valid ISO 8601), I want the quality gate to accept that value without failing validation.

**Implementation note.** The current regex `\d{4}-\d{2}-\d{2}` only matches date-only. Split into `validate_iso_date` (date-only) and `validate_iso_datetime` (datetime) helpers, or extend the existing function with a flag. Use stdlib-only parsing (`datetime.fromisoformat` in Python 3.7+ accepts most ISO forms; for the trailing `Z`, strip before parsing).

**Conditions of satisfaction.** (1) Given `2026-04-18`, date-only validator returns True. (2) Given `2026-04-18T23:43:14Z`, datetime validator returns True. (3) Given `2026-04-18T23:43:14+00:00`, datetime validator returns True. (4) Given `not-a-date`, both return False. (5) The run-metadata check (REQ-008) uses the datetime validator; sidecar `date` fields continue to use date-only.

**Alternative paths.** A regex covering both forms with optional groups is acceptable as long as it's fully anchored and clearly documented.

**References.** `SKILL.md:154,187`; `quality_gate.py:156-173`.

**Doc source.** `[SKILL.md §Sidecar JSON Canonical Examples, line 154]`; `[SKILL.md §Run Metadata, line 187]`.

**Specificity.** specific.

(4) File missing any required field → FAIL per field. (5) `start_time` not a valid ISO 8601 datetime → FAIL. (6) `check_repo` calls `check_run_metadata` unconditionally.

### REQ-009: archive_previous_run is atomic and preserves control_prompts

**Summary.** `archive_previous_run` must leave the repository in a consistent state under crash recovery: either both `quality/` and `previous_runs/TIMESTAMP/quality/` exist simultaneously (no destructive move until copy is verified), or the operation is atomic at the OS level. `control_prompts/` must be archived alongside `quality/`, not discarded.

**User story.** As an operator whose machine was force-rebooted mid-run, I want to come back to either the old `quality/` intact (run not yet archived) or a clean archive in `previous_runs/` with no leftover `quality/` (run archived and ready for next invocation), never both.

**Implementation note.** Current code does `copytree(quality_dir, archive_dir)` then `rmtree(quality_dir)` then `rmtree(control_prompts)`. Crash between copytree and rmtree leaves both. Fix: write to a temp path (e.g., `previous_runs/TIMESTAMP.partial/`), then rename to final name (atomic on POSIX); only then remove source. Include `control_prompts/` in the temp archive.

**Conditions of satisfaction.** (1) After function returns, `quality/` does not exist. (2) `previous_runs/TIMESTAMP/quality/` exists and is a complete copy. (3) `previous_runs/TIMESTAMP/control_prompts/` exists and is a complete copy (not deleted). (4) If SIGKILL hits between copy and rename, `previous_runs/TIMESTAMP.partial/` exists and a startup check either completes it or cleans it up. (5) Two parent processes starting within the same second use PID-qualified timestamps so their archive directories don't collide.

**Alternative paths.** A lockfile-based serialization is acceptable instead of PID qualification. The invariant is "no two parents can race on the same archive directory."

**References.** `run_playbook.py:565-576`.

**Doc source.** `[source]` — EXPLORATION.md §F-4, §QR-4.

### REQ-010: Phase 3/4/5 entry gates require all Phase 2 / prior-phase artifacts
