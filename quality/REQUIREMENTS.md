## Project Overview

The Quality Playbook is a specification-primary repository: the primary product is SKILL.md (~2069 lines), a long-form instruction document that AI coding agents read and follow to perform quality audits on arbitrary codebases. It is accompanied by 12 reference files (~3442 lines total), a shell validation script (quality_gate.sh, 632 lines), two orchestrator agent definitions, and documentation files.

The system works by reading SKILL.md into an AI agent's context window. The agent follows the instructions sequentially across six phases — explore, generate, review, audit, reconcile, verify — producing nine quality artifacts in a target project's quality/ directory. After execution, quality_gate.sh mechanically validates the artifacts for conformance.

The architecture has five subsystems: (1) SKILL.md (the specification), (2) references/ (phase-specific reference documents), (3) quality_gate.sh (mechanical validation), (4) agents/ (orchestrator definitions), and (5) ai_context/ (documentation). The most critical design decision is the separation of specification from validation: SKILL.md tells the agent what to produce, and quality_gate.sh verifies the output conforms. When these two drift, the specification becomes unreliable — the gate either under-validates (passing non-conformant output) or over-validates (failing conformant output).

The primary users are developers who install the skill into their projects and invoke it through AI coding tools (Claude Code, GitHub Copilot, Cursor). The secondary users are skill maintainers who modify SKILL.md and reference files. A key design philosophy is that every quality standard must have a concrete "why" grounded in specific failure scenarios — abstract targets get argued down by future AI sessions.

## Cross-Cutting Concerns

### Internal Consistency Model
The playbook's reliability depends on internal consistency across multiple documents that describe the same concepts. SKILL.md, both orchestrator agents, TOOLKIT.md, DEVELOPMENT_CONTEXT.md, the README, and AGENTS.md all describe phases, installation paths, and artifacts. When any one document is updated, all others must be checked for drift. The most dangerous inconsistencies are not contradictions (which are visible) but omissions (one document describes something the others do not mention, creating confusion about whether the feature exists).

### Specification-Validation Alignment
SKILL.md is the specification; quality_gate.sh is the validation. These must remain synchronized: every claim SKILL.md makes about what the gate checks must be true in quality_gate.sh, and every artifact SKILL.md marks as required must be validated by the gate. The artifact contract table (SKILL.md lines 72–101) is the canonical bridge between the two.

### Path Resolution
The skill can be installed at four different locations. All internal references to reference files, all installation instructions, and all orchestrator path-finding logic must work at every supported location. Hardcoded paths create installation-specific failures.

**Spec basis:** [Tier 2] SKILL.md lines 107–132 (Example 1), lines 1357–1398 (Example 2 + normative text), quality_gate.sh lines 192–268 (validation)
**Narrative:** The tdd-results.json schema is defined in three places within SKILL.md and validated by quality_gate.sh. An agent producing this file follows one of the canonical examples. If the examples disagree, the agent cannot produce conformant output.

### REQ-009: quality_gate.sh glob patterns must handle empty matches safely
**Spec basis:** [Tier 3] [inferred] quality_gate.sh lines 507, 116, 125–126
**Narrative:** Unquoted glob patterns in for loops iterate over the unexpanded literal string when no files match. The `for wf in ${q}/writeups/BUG-*.md` pattern (line 507) will test `[ -f "${q}/writeups/BUG-*.md" ]` against a non-existent literal path.
**Conditions of satisfaction:**
1. All glob patterns used in for loops must be guarded against empty expansion.
2. Specifically: the writeup loop (line 507) must handle the case where no writeup files exist without testing against the literal glob string.
