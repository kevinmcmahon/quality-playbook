## Project Overview

The Quality Playbook is a skill for AI coding agents that performs complete quality engineering audits on any codebase. It is not a test generator or a linter -- it is a structured methodology encoded as a 2,083-line specification (SKILL.md) that AI agents read and follow to produce nine interconnected quality artifacts: requirements, a quality constitution, functional tests, review protocols, spec audit protocols, TDD verification protocols, and a consolidated bug report with regression test patches.

The skill's primary users are developers who invoke it through AI coding agents (Claude Code, GitHub Copilot, Cursor). The developer says "run the quality playbook" and the agent reads SKILL.md, explores the target codebase, and produces the quality artifacts across seven phases. A secondary user is the skill maintainer who modifies SKILL.md and the reference files, benchmarks changes against test repositories, and validates output with quality_gate.sh.

The architecture has five subsystems: (1) the phase instruction system (SKILL.md), which encodes all behavioral rules, artifact contracts, and gate definitions; (2) the reference document system (references/*.md, 12 files), which provides detailed instructions for specific tasks like contract extraction and spec auditing; (3) the mechanical validation system (quality_gate.sh, 676 lines of Bash), which programmatically validates artifact conformance; (4) the orchestration layer (agents/*.md), which defines agent configurations for automated multi-phase execution; and (5) the context/documentation layer (ai_context/*.md, AGENTS.md), which provides bootstrap context.

The highest-risk areas are: (a) consistency between the three artifact-requirement sources (SKILL.md table, quality_gate.sh checks, verification.md benchmarks) -- inconsistency means the gate lies about what's required; (b) quality_gate.sh's false-pass risk when required artifacts are classified as WARN instead of FAIL; (c) internal numbering and cross-reference accuracy in the 12 reference documents that agents read alongside SKILL.md.

## Cross-Cutting Concerns

### Specification-implementation consistency
The playbook's correctness depends on three documents agreeing: SKILL.md (the specification), quality_gate.sh (the enforcement), and verification.md (the benchmarks). When any of these diverges, the system has a gap. This concern spans REQ-001, REQ-002, REQ-006, REQ-007, and REQ-010. Every change to one document must be verified against the other two.

### Version stamp traceability
Every generated artifact carries a version stamp that traces back to the skill version that created it. This enables cross-run contamination detection and artifact provenance tracking. The concern spans REQ-008 and REQ-011. Version stamps must use the exact metadata.version from SKILL.md's frontmatter.

### Defensive input handling
quality_gate.sh processes user-supplied directory paths and parses JSON with grep. Both paths need defensive handling: directory paths must not be passed through eval, and JSON parsing must handle edge cases. This spans REQ-009 and REQ-014.

### REQ-001: Artifact Contract Enforcement Consistency

**Summary:** The three artifact-requirement sources (SKILL.md artifact contract table, quality_gate.sh file existence checks, verification.md benchmarks) must agree on which artifacts are required and enforce the same severity for missing artifacts.

**User story:** As a developer running the quality playbook, I need the gate to catch all missing required artifacts, so that a passing gate means the run is actually complete.

**Conditions of satisfaction:**
1. Every artifact marked "Required: Yes" in SKILL.md lines 86-115 must be checked by quality_gate.sh with fail() severity -- SKILL.md:86-115 vs quality_gate.sh:106-129.
2. Every artifact checked by quality_gate.sh must appear in SKILL.md's artifact contract table -- quality_gate.sh:106-129 vs SKILL.md:86-115.
3. verification.md benchmark 40 (lines 226-229) must list the same required artifacts as SKILL.md's table -- references/verification.md:226-229.
4. No artifact may be classified differently (FAIL vs WARN) between SKILL.md and quality_gate.sh without documented justification that references an accessible bug report or design decision.

**Spec basis:** [Tier 1] SKILL.md lines 83-85: "If the gate checks for it, this skill must instruct its creation. This is the canonical list -- any artifact not listed here should not be gate-enforced, and any gate check should trace to an artifact listed here."

**Alternative paths:**
- Conditional artifacts (e.g., integration-results.json "When integration tests run") must have gate logic that detects the condition, not unconditionally WARN.
- Artifacts with "Yes (benchmark)" in the Required column may use different enforcement in --general mode.

**Verification method:** Compare the three sources mechanically: extract "Required: Yes" artifacts from SKILL.md, extract fail() and warn() targets from quality_gate.sh, extract benchmark 40 file list from verification.md. Diff the three lists.
