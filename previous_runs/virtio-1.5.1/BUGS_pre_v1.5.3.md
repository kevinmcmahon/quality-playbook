### BUG-001: Shared transport filtering drops `VIRTIO_F_RING_RESET` for MMIO and vDPA
- Primary requirement: REQ-004
- Covers: [REQ-004/cell-RING_RESET-MMIO, REQ-004/cell-RING_RESET-VDPA]
- Consolidation rationale: Both absent cells are caused by the same missing `case VIRTIO_F_RING_RESET:` in `vring_transport_features()`, and both finalize paths commit the shared filtered bitset without their own compensation.
- Severity: MEDIUM
- File: `drivers/virtio/virtio_ring.c:3505-3532`; `drivers/virtio/virtio_mmio.c:109-131`; `drivers/virtio/virtio_vdpa.c:389-397`
- Spec basis: `include/uapi/linux/virtio_config.h:114-116`; `quality/mechanical/vring_transport_features_cases.txt:1-8`; `quality/REQUIREMENTS.md:148-166`
- Expected: The shared transport whitelist preserves `VIRTIO_F_RING_RESET`, so MMIO and vDPA finalize the same queue-reset capability the device advertised.
- Actual: `vring_transport_features()` falls through to `default:` for bit 40, so MMIO and vDPA commit a feature set with `VIRTIO_F_RING_RESET` cleared while PCI modern has to re-add it later in `vp_transport_features()`.
- Regression test: `quality.test_regression.TestPhase3RegressionBugs.test_bug_001_ring_reset_survives_shared_whitelist`
- Patch files: `quality/patches/BUG-001-regression-test.patch`, `quality/patches/BUG-001-fix.patch`

### BUG-008: PCI modern finalize silently drops `VIRTIO_F_NOTIF_CONFIG_DATA`
- Primary requirement: REQ-001
- Severity: MEDIUM
- File: `drivers/virtio/virtio_pci_modern.c:367-443`; `drivers/virtio/virtio_ring.c:3509-3529`
- Spec basis: `include/uapi/linux/virtio_config.h:108-111`; `drivers/virtio/virtio_pci_modern.c:405-413`; `quality/spec_audits/2026-04-25-triage.md`
- Expected: If a PCI modern device advertises `VIRTIO_F_NOTIF_CONFIG_DATA`, `vp_finalize_features()` preserves the bit through `vp_transport_features()`, lets `vp_check_common_size()` validate `queue_notify_data`, and then commits the validated feature set.
- Actual: `vring_transport_features()` clears bit 39 in its default branch, and `vp_transport_features()` only restores `VIRTIO_F_SR_IOV`, `VIRTIO_F_RING_RESET`, and `VIRTIO_F_ADMIN_VQ`. `vp_check_common_size()` still checks `VIRTIO_F_NOTIF_CONFIG_DATA`, but the bit has already been dropped, so the validation is skipped and PCI modern commits a negotiated feature subset with notification-config data silently disabled.
- Regression test: `quality.test_regression.TestPhase3RegressionBugs.test_bug_008_pci_preserves_notification_config_data`
- Patch files: `quality/patches/BUG-008-regression-test.patch`, `quality/patches/BUG-008-fix.patch`

### BUG-009: PCI sparse-slot queue bookkeeping can tear down the wrong queue metadata

- Severity: HIGH
- File: `drivers/virtio/virtio_pci_common.c:271-285`; `drivers/virtio/virtio_pci_common.c:420-429`; `drivers/virtio/virtio_pci_common.c:481-490`
- Spec basis: `include/linux/virtio_config.h:22-27`; `drivers/virtio/virtio_balloon.c:583-607`; `quality/REQUIREMENTS.md`
- Expected: when `virtqueue_info.name == NULL` leaves sparse caller slots, PCI queue creation and teardown keep `vp_dev->vqs[]` keyed to the actual queue number that was created, so cleanup resolves the right `struct virtio_pci_vq_info`.
- Actual: PCI create paths store side metadata in `vp_dev->vqs[i]` by sparse caller-slot index, but `vp_del_vqs()` later resolves it with `vp_dev->vqs[vq->index]`, so optional queues can free the wrong metadata entry or dereference NULL during remove/error cleanup.
- Regression test: `quality.test_regression.TestPhase3RegressionBugs.test_bug_009_pci_sparse_queue_slots_store_metadata_by_actual_queue_index`

### BUG-010: vDPA initial affinity setup uses sparse slot indices instead of actual queue numbers

### BUG-012: PCI IRQ-affinity reporting answers vector queries in queue-slot space
- Primary requirement: REQ-012
- Severity: MEDIUM
- File: `drivers/virtio/virtio.c:387-401`; `drivers/virtio/virtio_pci_common.c:163-191`; `drivers/virtio/virtio_pci_common.c:327-370`; `drivers/virtio/virtio_pci_common.c:581-591`
- Spec basis: `drivers/virtio/virtio.c:387-401`; `quality/REQUIREMENTS.md`
- Expected: `virtio_irq_get_affinity(..., irq_vec)` asks the transport about an interrupt vector, so PCI should answer the config vector and shared/per-queue vectors by their MSI-X/vector identity.

### BUG-014: Prepare-only driver reset hooks can strand `config_core_enabled` after a successful prepare phase
- Primary requirement: REQ-014
- Severity: MEDIUM
- File: `include/linux/virtio.h:240-243`; `include/linux/virtio.h:261-262`; `drivers/virtio/virtio.c:180-193`; `drivers/virtio/virtio.c:676-706`; `drivers/virtio/virtio_pci_common.c:794-827`
- Spec basis: `include/linux/virtio.h:240-243`; `quality/REQUIREMENTS.md`
- Expected: if a driver is allowed to wire `reset_prepare`, the reset flow must also guarantee the corresponding restoration of core state before the transport wrapper treats reset completion as benign; a successful prepare phase must not leave `config_core_enabled` false forever.
- Actual: `struct virtio_driver` documents `reset_prepare` and `reset_done` as independently optional hooks, but `virtio_device_reset_prepare()` disables core config delivery and returns success with only `.reset_prepare` present. `virtio_device_reset_done()` is the only helper that re-enables `config_core_enabled`, yet it returns `-EOPNOTSUPP` immediately when `.reset_done` is absent, and the PCI transport wrapper explicitly tolerates that return.
- Regression test: `quality.test_regression.TestPhase3RegressionBugs.test_bug_014_reset_prepare_requires_matching_reset_done_or_repair`
- Patch files: `quality/patches/BUG-014-regression-test.patch`, `quality/patches/BUG-014-fix.patch`
