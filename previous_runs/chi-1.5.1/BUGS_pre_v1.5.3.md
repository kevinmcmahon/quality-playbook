### BUG-001: Compression matches disabled and substring codings

- Source: Phase 3 code review (Pass 1, Pass 2 REQ-007)
- File:line: `middleware/compress.go:214-246`
- Severity: MEDIUM
- Primary requirement: REQ-007
- Disposition: `code-fix`
- Fix type: `code`
- Spec basis: `quality/REQUIREMENTS.md:225-240` requires compression to honor exact `Accept-Encoding` tokens and quality values; `docs_gathered/14_middleware_reference.md` documents request-driven compression negotiation.
- Divergence: The requirement says compression must activate only for an exact supported coding that remains enabled after parsing token lists and `q` parameters. `selectEncoder()` lowercases and comma-splits the header, but `matchAcceptEncoding()` then uses `strings.Contains`, so a disabled token like `gzip;q=0` and an unrelated token like `brgzip` both satisfy the gzip branch.
- Documented intent: Compression should only activate when the client explicitly advertises a supported coding that is still enabled after quality-value parsing.
- Code behavior: `matchAcceptEncoding()` returns true for any comma token that merely contains the encoder name instead of matching the coding token and respecting `q=0` (`middleware/compress.go:240-246`).
- Minimal reproduction: Send `Accept-Encoding: gzip;q=0` or `Accept-Encoding: brgzip` to a compressible route that uses `middleware.Compress`.
- Expected behavior: Both requests remain uncompressed because gzip is either explicitly disabled or never advertised as a token.
- Actual behavior: The middleware selects the gzip encoder and emits `Content-Encoding: gzip`.
- Regression test: `quality/test_regression_test.go:TestRegressionBUG001CompressionRejectsDisabledAndSubstringTokens`
- Regression patch: `quality/patches/BUG-001-regression-test.patch`
- Fix patch: `quality/patches/BUG-001-fix.patch`
- Disposition rationale: This is a code-side parser defect, not a spec gap: the generated requirement and middleware docs already define exact-token negotiation, and the implementation simply checks the wrong condition.

- Spec basis: `REQ-016` requires route-probing middleware to leave final request route metadata truthful; `docs_gathered/13_api_reference.md:221-279` and `docs_gathered/05_context_and_values.md:153-173` present `Match()`, `Find()`, and `RoutePattern()` as one coherent route-metadata surface.
- Divergence: The requirement says middleware-owned route probes must not corrupt the live request metadata seen by handlers and instrumentation. `SupressNotFound` runs `Match()` against the live request context before real dispatch, so the eventual successful route pattern is appended once during the probe and again during actual routing.
- Documented intent: A middleware that short-circuits obvious 404s should not change the route metadata for requests that do match and continue.
- Code behavior: `middleware/supress_notfound.go:18-20` calls `rctx.Routes.Match(rctx, ...)` with the live request context; `Match()` delegates to `Find()` (`mux.go:353-360`), and `FindRoute()` rebuilds route state without clearing `RoutePatterns` (`tree.go:375-394`, `context.go:123-133`), so the final request reports `RoutePattern() == "/users/{id}/users/{id}"`.
- Minimal reproduction: Add `middleware.SupressNotFound(r)` above a route plus a post-handler instrumentation middleware that records `chi.RouteContext(r.Context()).RoutePattern()` after `next.ServeHTTP`, then send `GET /users/123`.
- Expected behavior: The final route pattern is reported once as `/users/{id}`.
- Actual behavior: The final route pattern is reported as `/users/{id}/users/{id}`.
