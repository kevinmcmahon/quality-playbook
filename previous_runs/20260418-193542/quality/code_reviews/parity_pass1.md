### quality_gate.sh — JSON Helper Parity

**Lines 259-265:** BUG [CAND-P1] — `json_has_key` used for summary sub-key checks while `json_key_count` used for per-bug field checks:
```bash
for skey in total verified confirmed_open red_failed green_failed; do
    if json_has_key "$json_file" "$skey"; then
```
vs lines 239-248:
```bash
fcount=$(json_key_count "$json_file" "$field")
if [ "$fcount" -ge "$bug_count" ]; then
```
Two parallel "does JSON contain required field?" checks use different validators. `json_has_key` at line 77 uses `grep -q "\"${key}\""` (matches key name anywhere). `json_key_count` at line 90 uses `grep -c "\"${key}\"[[:space:]]*:"` (colon-anchored, only matches actual JSON keys). The per-bug check correctly uses the stronger pattern; the summary check uses the weaker one. A `tdd-results.json` file with string content like `"notes": "total failures and verified fixes noted"` would pass the summary check for `total` and `verified` even though those keys don't exist as actual JSON properties. Expected: both checks should use consistent pattern. Actual: different patterns with different false-positive rates.

**Lines 562-588:** QUESTION [CAND-P2] — Patch counting uses aggregate ls-glob rather than per-bug iteration:
```bash
reg_patch_count=$(ls ${q}/patches/BUG-*-regression*.patch 2>/dev/null | wc -l | tr -d ' ')
```
vs TDD log section lines 316-345 which iterates `$bug_ids`. The structural inconsistency means the patch section cannot detect that a specific bug's patch is missing (only that the total count is wrong). For a run with 3 bugs but patches for bug IDs that don't match the actual bugs, the count would match but coverage would not. This is a design gap but requires a concrete failure scenario for full confirmation as a bug.

**Line 331:** QUESTION — Existing BUG-M8 scope: `if ls ${q}/patches/${bid}-fix*.patch &>/dev/null` inside the per-bug loop at line 316. This IS per-bug iteration, but uses ls-glob (nullglob vulnerable). The patch COUNT section at lines 567-568 also uses ls-glob. Two ls-glob vulnerabilities in the patch-related code, one within a per-bug loop (nullglob can cause false positive on fix.patch existence) and one in aggregate count (nullglob can inflate count). Both are within BUG-M8 scope.

### SKILL.md — Phase Gate Parity

**Lines 897-907 vs 1573-1590:** QUESTION [CAND-P3] — Phase 2 has mandatory entry gate for Phase 1 artifacts; Phase 5 has no equivalent entry gate for Phase 4 artifacts. Phase 2 entry gate checks for specific section titles in EXPLORATION.md. Phase 5 merely says "Re-read quality/PROGRESS.md" — which is a text file the agent writes itself. An agent that writes "Phase 4 complete" in PROGRESS.md without actually running the spec audit could proceed through Phase 5 undetected until the terminal gate. The fail-early pattern at Phase 2 is not mirrored at Phase 5.

**Lines 88-119 vs line 1641:** QUESTION [CAND-P4] — SEED_CHECKS.md required by Phase 5 prose gate but absent from artifact contract table. The table says: "This is the canonical list — any artifact not listed here should not be gate-enforced." Yet SEED_CHECKS.md is required by the Phase 5 gate (line 1641). This creates a contradiction: the table says it's not gate-enforced, but Phase 5 says it's required. An agent reading only the table would not know to create SEED_CHECKS.md. An agent following the Phase 5 gate would fail if it doesn't exist after a Phase 0b run.
