### BUG-002: Trusted uppercase proto values silently disable `req.secure`

- **Source:** Code Review (Pass 1, Pass 2, Pass 3)
- **Primary requirement:** `REQ-002`
- **File:** `lib/request.js:307-315`, `lib/request.js:326-328`
- **Severity:** HIGH
- **Spec basis:** `REQ-002` requires the trusted-header path to canonicalize `X-Forwarded-Proto` before exposing `req.protocol` and deriving `req.secure`.
- **Expected behavior:** When trust proxy is enabled, `X-Forwarded-Proto: HTTPS` should behave the same as `https`, leaving `req.protocol === 'https'` and `req.secure === true`.
- **Actual behavior:** The trusted-header branch trims whitespace but never lowercases the selected token, so `req.protocol` becomes `"HTTPS"` and `req.secure` stays `false`.
- **Evidence:** A runtime probe with trust proxy enabled and `X-Forwarded-Proto: HTTPS` returned `"no"` from a `req.secure ? 'yes' : 'no'` handler.
- **Regression test:** `quality/regression_tests/BUG-002.js`
- **Regression patch:** `quality/patches/BUG-002-regression-test.patch`
- **Fix patch:** `quality/patches/BUG-002-fix.patch`
- **Disposition:** `code-fix`

### BUG-003: `res.sendFile()` ignores explicit per-call `etag: false` overrides

- **Source:** Code Review (Pass 2, Pass 3)
- **Primary requirement:** `REQ-004`
- **File:** `lib/response.js:376-401`
- **Severity:** HIGH
- **Spec basis:** `REQ-004` requires `res.sendFile()` to honor caller-supplied `options.etag` values instead of replacing them with the app default.
- **Expected behavior:** `res.sendFile(path, { etag: false })` should omit the `ETag` header even when the app-wide ETag setting is enabled.
- **Actual behavior:** `res.sendFile()` always runs `opts.etag = this.app.enabled('etag')`, which overwrites the caller's `false` value before the send module sees it.
- **Evidence:** A runtime probe serving `test/fixtures/name.txt` through `res.sendFile(..., { etag: false })` still returned `ETag: W/"4-19d823ba1db"`.
- **Regression test:** `quality/regression_tests/BUG-003.js`
- **Regression patch:** `quality/patches/BUG-003-regression-test.patch`
- **Fix patch:** `quality/patches/BUG-003-fix.patch`
- **Disposition:** `code-fix`

### BUG-004: File-backed helpers flatten non-boolean ETag policies to the weak default

- **Source:** Code Review (Pass 2, Pass 3)
- **Primary requirement:** `REQ-005`
- **File:** `lib/response.js:399-401`, `lib/response.js:433-482`
- **Severity:** HIGH
- **Spec basis:** `REQ-005` requires `res.sendFile()` to preserve the app's strong/weak/custom ETag policy, and `REQ-006` requires `res.download()` to stay aligned with `res.sendFile()`.
- **Expected behavior:** `app.set('etag', 'strong')` and `app.set('etag', customFn)` should keep their distinct validator behavior on both `res.sendFile()` and `res.download()`.
- **Actual behavior:** Both helpers reduce the policy to `this.app.enabled('etag')`, so the send module falls back to its weak stat validator for every truthy policy.
- **Evidence:** Runtime probes with `app.set('etag', 'strong')` and `app.set('etag', function () { return '"custom"'; })` still returned the send module's weak stat-based validator instead of a strong or custom value.
- **Affected surfaces:** `res.sendFile()` and `res.download()`
- **Regression test:** `quality/regression_tests/BUG-004.js`
- **Regression patch:** `quality/patches/BUG-004-regression-test.patch`
- **Fix patch:** `quality/patches/BUG-004-fix.patch`

### BUG-006: `res.jsonp()` returns invalid JavaScript for digit-prefixed callback names

- **Source:** Code Review (Pass 1, Pass 2, Pass 3)
- **Primary requirement:** `REQ-008`
- **File:** `lib/response.js:281-300`
- **Severity:** MEDIUM
- **Spec basis:** `REQ-008` requires JSONP callback targets to be validated after sanitization and to fall back safely instead of returning `200 text/javascript` with invalid JavaScript.
- **Expected behavior:** A callback like `1foo` should be rejected or fall back to plain JSON instead of producing a broken script wrapper.
- **Actual behavior:** The sanitization regex keeps digits, so `callback=1foo` becomes `/**/ typeof 1foo === 'function' && 1foo({...});`, which is invalid JavaScript even though the response still claims success.
- **Evidence:** A runtime probe against `/?callback=1foo` returned `200 text/javascript` with `/**/ typeof 1foo === 'function' && 1foo({"ok":true});`.
- **Regression test:** `quality/regression_tests/BUG-006.js`
- **Regression patch:** `quality/patches/BUG-006-regression-test.patch`
- **Fix patch:** `quality/patches/BUG-006-fix.patch`

### BUG-007: `res.clearCookie()` treats signed delete options as signed-cookie creation

- **Source:** Code Review (gap iteration Pass 1, Pass 2, Pass 3)
- **Primary requirement:** `REQ-009`
- **File:** `lib/response.js:709-715`, `lib/response.js:742-757`
- **Severity:** MEDIUM
- **Spec basis:** `REQ-009` requires the clear-cookie path to ignore signing metadata and emit an expired empty tombstone cookie without requiring `cookieParser("secret")`.
- **Expected behavior:** `res.clearCookie('sid', { signed: true, path: '/admin' })` should emit `sid=; Path=/admin; Expires=Thu, 01 Jan 1970 00:00:00 GMT` even when no signing secret is installed.
- **Actual behavior:** `res.clearCookie()` forwards `signed: true` into `res.cookie()`, which throws `cookieParser("secret") required for signed cookies` when no secret is configured. With a secret present, the helper emits a signed `s:` tombstone instead of a plain empty cookie.
- **Evidence:** A runtime probe without `cookie-parser` returned `500 "cookieParser(\"secret\") required for signed cookies"` and no `Set-Cookie` header; the same probe with a secret emitted `sid=s%3A...; Path=/admin; Expires=Thu, 01 Jan 1970 00:00:00 GMT` instead of an unsigned empty tombstone.
- **Regression test:** `quality/regression_tests/BUG-007.js`
- **Regression patch:** `quality/patches/BUG-007-regression-test.patch`
- **Fix patch:** `quality/patches/BUG-007-fix.patch`
- **Disposition:** `code-fix`

### BUG-008: `app.render()` keeps using the old cached `View` after `views` changes

- **Source:** Code Review (gap iteration Pass 1, Pass 2, Pass 3)
- **Primary requirement:** `REQ-010`
- **File:** `lib/application.js:543-570`, `lib/view.js:52-95`, `lib/view.js:104-123`
- **Severity:** MEDIUM
- **Spec basis:** `REQ-010` requires view-cache reuse to respect the current `views` setting rather than pinning the first resolved path for a logical view name.

### BUG-011: `res.redirect(204|205|304, ...)` drifts from Express's bodyless-response cleanup rules

- **Source:** Code Review (unfiltered iteration Pass 1, Pass 2, Pass 3) and Spec Audit triage
- **Primary requirement:** `REQ-013`
- **File:** `lib/response.js:839-863`, compared with `lib/response.js:191-207`
- **Severity:** LOW
- **Spec basis:** `REQ-013` requires bodyless redirect statuses to keep the `Location` header while scrubbing entity headers such as `Content-Type` and `Content-Length`.
- **Expected behavior:** `res.redirect(204, '/target')` and `res.redirect(304, '/target')` should return an empty response without entity headers, while `res.redirect(205, '/target')` should mirror `res.sendStatus(205)` by returning an empty body with `Content-Length: 0`.
- **Actual behavior:** `res.redirect()` still computes a redirect body for all three statuses. `204` / `304` responses retain redirect-body metadata, and `205` goes further by returning a body and a nonzero `Content-Length`.
- **Evidence:** Runtime probes returned `Location: /target` plus `Content-Type: text/plain; charset=utf-8` for `204`, `205`, and `304`; the `205` probe returned body text `"Reset Content. Redirecting to /target"` with `Content-Length: 32` instead of the empty-body semantics already encoded in `res.send()`.
- **Regression test:** `quality/regression_tests/BUG-011.js`
- **Regression patch:** `quality/patches/BUG-011-regression-test.patch`
- **Fix patch:** `quality/patches/BUG-011-fix.patch`
- **Disposition:** `code-fix`

### BUG-012: `res.redirect(status, url)` validates the status only after writing redirect metadata

- **Source:** Code Review (parity iteration Pass 1, Pass 2, Pass 3) and Spec Audit triage
- **Primary requirement:** `REQ-014`
- **File:** `lib/response.js:835-857`, compared with `lib/response.js:64-75` and `lib/response.js:321-327`
- **Severity:** MEDIUM
- **Spec basis:** `REQ-014` requires redirect status validation to fail before `Location` or redirect-body metadata is written, matching the validation order already used by `res.status()` and `res.sendStatus()`.
- **Expected behavior:** `res.redirect('301', '/target')`, `res.redirect(true, '/target')`, and `res.redirect(99, '/target')` should fail before the response advertises any redirect target or redirect-body `Content-Type`.
- **Actual behavior:** the helper calls `this.location(address)` and `this.format(...)` first, then finally calls `this.status(status)`. Invalid statuses therefore bubble out as `500` responses that still carry `Location: /target` and negotiated redirect metadata.
- **Evidence:** Runtime probes for `'301'`, `true`, `99`, and `1000` all produced `500` responses that still included `Location: /x`; the same bad statuses fail cleanly before mutation when routed through `res.status()` / `res.sendStatus()`.
- **Regression test:** `quality/regression_tests/BUG-012.js`
- **Regression patch:** `quality/patches/BUG-012-regression-test.patch`
- **Fix patch:** `quality/patches/BUG-012-fix.patch`
- **Disposition:** `code-fix`

### BUG-013: `res.download()` bypasses `res.sendFile()`'s invalid-path validation surface

- **Source:** Code Review (adversarial iteration Pass 1, Pass 2, Pass 3) and Spec Audit triage
- **Primary requirement:** `REQ-006`
- **File:** `lib/response.js:378-394`, `lib/response.js:433-482`
- **Severity:** LOW
- **Spec basis:** `REQ-006` requires `res.download()` to preserve `res.sendFile()`'s helper-level path validation while adding attachment metadata.
- **Expected behavior:** invalid download paths should fail through the same Express-level `TypeError` contract as `res.sendFile()`, so `res.download()` remains a thin attachment wrapper over the base file helper.
- **Actual behavior:** the wrapper reaches `content-disposition` and `path.resolve` before it delegates to `res.sendFile()`, so `res.download()` raises lower-level errors such as `filename must be a string` or `The "paths[0]" argument must be of type string` instead of the base helper's own path-validation errors.
- **Evidence:** runtime probes showed `res.download()` -> `TypeError: The "paths[0]" argument must be of type string. Received undefined`, `res.download(true)` -> `TypeError: filename must be a string`, and `res.download(42, 'x.txt')` -> `TypeError: The "paths[0]" argument must be of type string. Received type number (42)`; the corresponding `res.sendFile()` invalid-path tests are already pinned to `path argument is required to res.sendFile` / `path must be a string to res.sendFile` in `test/res.sendFile.js:16-39`.
- **Regression test:** `quality/regression_tests/BUG-013.js`
- **Regression patch:** `quality/patches/BUG-013-regression-test.patch`
- **Fix patch:** `quality/patches/BUG-013-fix.patch`
- **Disposition:** `code-fix`
